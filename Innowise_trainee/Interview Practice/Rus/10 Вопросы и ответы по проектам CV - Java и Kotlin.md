## Как пользоваться заметкой

В заметке ровно 60 вопросов: 48 по финансовой экосистеме на Java и 12 по интернет-магазину на Kotlin. Сначала прочитай объяснение, затем закрой его и ответь вслух. Большинство ответов должны занимать 60–90 секунд.

Используются три статуса:

- **Факт из CV** — информация прямо указана в CV. Её можно объяснять от первого лица, если она соответствует реальному опыту.
- **Техническая реконструкция** — правдоподобная модель, но она не подтверждена CV. На интервью используй «я бы проверил» или «вероятная схема».
- **Требует подтверждения** — без дополнительной информации нельзя безопасно сформулировать личный опыт. Заполни такой ответ только реальным примером.

> [!danger] Основное правило
> Не превращай техническую реконструкцию в выдуманный production-опыт. Если точная архитектура, состав команды или реальный дефект неизвестны, скажи об этом прямо и объясни свой подход к проверке.

Вероятная схема финансовой экосистемы:

```text
Внешние источники → Kafka → Processing services → PostgreSQL / MongoDB
                                            ↓
                                          Redis
                                            ↓
                                      REST API → Web UI
```

Вероятная схема интернет-магазина:

```text
Web / Mobile → REST API → Order service → PostgreSQL
                                │
                                ├→ Payment / Delivery HTTP services
                                ├→ Redis
                                └→ RabbitMQ → Inventory / Notification consumers
```

Обе схемы являются технической реконструкцией. CV не содержит названий services, topics, queues, tables, collections и точного направления каждого вызова.

### Назначение проекта и рабочий процесс

#### 1. Расскажите о финансовой экосистеме

**Статус:** Факт из CV.

**Готовый ответ:** На последнем проекте я работал AQA инженером в аналитической финансовой экосистеме. Платформа собирала данные из разных источников, обрабатывала их и показывала финансовую отчётность в едином интерфейсе. В мои задачи входили ручные и автоматизированные проверки Web UI и REST API, проверка данных в PostgreSQL и MongoDB, анализ логов в OpenSearch и работа с асинхронными взаимодействиями через Kafka. Автотесты были написаны на Java с JUnit 5, Selenium и REST Assured и запускались через GitLab CI/CD.

**Объяснение:** Этот ответ использует только сведения из CV. Не добавляй названия клиентов, services или финансовых инструментов, если они не подтверждены.

#### 2. Какую бизнес-задачу решал проект?

**Статус:** Факт из CV.

**Готовый ответ:** Проект создавал единую точку для мониторинга финансовых активов и управления капиталом. Данные поступали из нескольких источников, поэтому главными рисками были потеря информации, дублирование операций, расхождение между хранилищами и неправильные расчёты в отчётах. Для QA было важно проверить не только интерфейс, но и весь путь данных: вход, обработку, сохранение, API и итоговую визуализацию.

**Объяснение:** Основной акцент — целостность финансовых данных. Конкретные виды активов и формулы в CV не указаны.

#### 3. Какова была ваша роль на проекте?

**Статус:** Факт из CV.

**Готовый ответ:** Я выполнял роль AQA инженера, но работа включала и ручное тестирование. Я анализировал требования, проектировал regression scenarios, проверял Web UI, Mobile и REST API, работал с PostgreSQL, MongoDB и Redis, локализовал дефекты по логам OpenSearch и разрабатывал автотесты на Java. Также я настраивал локальное окружение в Docker и подключал UI- и API-тесты к GitLab CI/CD.

**Объяснение:** Роль можно назвать Fullstack QA Automation, потому что она покрывает UI, API, data и infrastructure levels. Официальное название в CV — AQA инженер.

#### 4. Как была устроена команда?

**Статус:** Требует подтверждения.

**Готовый ответ:** В CV состав и размер команды не указаны, поэтому здесь нужно назвать только реальные данные. Безопасно можно сказать, что для таких задач AQA обычно взаимодействует с разработчиками, аналитиками, Manual QA и DevOps: уточняет acceptance criteria, согласует test data, разбирает дефекты и добавляет tests в pipeline. Точное количество людей и распределение обязанностей необходимо подтвердить перед интервью.

**Объяснение:** Не придумывай Scrum-команду, роли и число участников. Интервьюер часто задаёт уточняющие вопросы о взаимодействии с каждым участником.

**Статус:** Техническая реконструкция.

**Готовый ответ:** Я бы начал с анализа задачи в Jira и требований в Confluence, выделил business rules, integrations и риски. Оценку разделил бы на подготовку test data и environment, ручную проверку, automation, code review и подключение к CI/CD, отдельно обозначив неизвестные зависимости. Затем определил test levels, проверил feature через API, базы данных, логи и UI и автоматизировал стабильные regression scenarios. По результатам сохранил бы test report, logs и evidence для дефектов.

**Объяснение:** Jira, Confluence и GitLab подтверждены CV. Реальный способ оценки, процесс review, Definition of Done и система отчётов не указаны и требуют подтверждения.

#### 6. Как вы работали с требованиями и test design?

**Статус:** Факт из CV + требует конкретного примера.

**Готовый ответ:** В CV указано, что я проектировал ручные и автоматизированные regression scenarios с применением test design techniques. Для финансовой формы я мог разделить значения на equivalence classes, проверить boundaries суммы и периода, использовать Decision Table для ролей и типов операций и State Transition для статусов обработки. На интервью я должен дополнить этот ответ одним реальным полем или workflow из проекта.

**Объяснение:** Названия техник в CV не перечислены. Не утверждай, что применял конкретную технику, пока не подготовлен реальный пример.

#### 7. Какие test levels вы использовали?

**Статус:** Факт из CV.

**Готовый ответ:** На проекте были ручные и автоматизированные проверки нескольких уровней. REST API и business rules проверялись на API и integration levels, Web UI — через Selenium, а полный путь финансовых данных — в E2E scenarios. Данные дополнительно проверялись в PostgreSQL и MongoDB, а асинхронные результаты — через Kafka и логи. Unit tests с Mockito нужно отделять от integration tests: mock не проверяет настоящий broker или network.

**Объяснение:** CV прямо упоминает UI, API, integration, regression и E2E testing. Точный объём unit tests требует уточнения.

#### 8. Как вы выбирали, что автоматизировать?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Я бы автоматизировал стабильные, повторяемые и критичные scenarios, которые регулярно входят в regression. Business rules и data transformations выгоднее проверять на API или component level, потому что такие tests быстрее и точнее локализуют дефект. В UI оставил бы ключевые пользовательские flows и проверки отображения, которые нельзя надёжно покрыть ниже. Одноразовые, нестабильные или часто меняющиеся функции сначала проверял бы вручную.

**Объяснение:** Это применение Test Pyramid. CV не сообщает реальное распределение tests по уровням.

### Архитектура и поток данных

#### 9. Как выглядела архитектура системы?

**Статус:** Техническая реконструкция.

**Готовый ответ:** По CV можно предположить микросервисную архитектуру: внешние источники передают финансовые данные, Kafka доставляет events сервисам обработки, PostgreSQL хранит relational и transactional data, MongoDB — documents или analytical results, Redis ускоряет чтение, а REST API передаёт результат Web UI. Это вероятная схема, а не подтверждённая архитектура. На интервью я бы сначала рассказал только компоненты из CV, а затем обозначил, что точная декомпозиция требует уточнения.

**Объяснение:** Не называй вымышленные service names, topic names и database schemas.

#### 10. Как одна финансовая операция могла проходить через систему?

**Готовый ответ:** Вероятный flow начинается с получения операции от внешнего источника. После validation event публикуется в Kafka, consumer обрабатывает его и сохраняет основные данные в PostgreSQL, а analytical document — в MongoDB. #### 11. Как проверить целостность данных на всём пути?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Я бы создал данные с уникальным business ID и заранее рассчитал expected result. Затем проверил бы исходный request или event, запись в PostgreSQL, соответствующий document в MongoDB, состояние cache, API response и отображение в UI. Для денежных значений отдельно сравнил бы currency, precision, scale и rounding. Если обработка асинхронная, использовал бы polling с ограниченным timeout.

**Объяснение:** Уникальный ID позволяет связать logs, broker message, database records и UI.

#### 12. Какие главные риски есть у финансовых данных?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Главные риски — потеря или duplicate operation, неправильный знак суммы, неверная currency, ошибка округления, нарушение ordering и повторная обработка одного event. Также опасны расхождения между PostgreSQL и MongoDB, stale cache, неправильная timezone и доступ пользователя к чужим данным. Приоритет я бы отдавал consistency, idempotency, audit trail и корректности расчётов.

**Объяснение:** Это общие риски финансовых систем. Конкретные правила проекта должны быть подтверждены requirements.

#### 13. Как проверить денежные расчёты?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Я бы уточнил currency, precision, scale и rounding rule в требованиях. Затем проверил positive, negative и zero values, минимальные и максимальные boundaries, дробные суммы и операции с разными валютами. Expected result лучше рассчитывать независимо от application logic, например через заранее подготовленные test data. Дополнительно нужно сравнить результат в database, API и UI.

**Объяснение:** Нельзя использовать ту же production formula в test как единственный oracle: одинаковая ошибка может появиться в обоих местах.

#### 14. Как проверить согласованность PostgreSQL и MongoDB?

**Статус:** Факт из CV + техническая реконструкция.

**Готовый ответ:** В CV указано, что я проверял consistency данных в PostgreSQL и MongoDB. Технически я бы находил связанные записи по business ID, сравнивал общие поля, статусы, timestamps и version, учитывая различие моделей хранения. Если MongoDB обновляется асинхронно, test должен ждать expected state, а не сравнивать данные сразу после request. Точный набор полей и направление синхронизации нужно подтвердить.

**Объяснение:** Равенство всех полей не всегда требуется: разные хранилища могут содержать разные представления одного объекта.

#### 15. Как тестировать eventual consistency?

**Статус:** Техническая реконструкция.

**Готовый ответ:** После запуска операции я бы периодически проверял ожидаемое состояние до успеха или timeout. Polling interval должен быть достаточно редким, чтобы не создавать нагрузку, а общий timeout — соответствовать SLA системы. При failure test должен сохранить последнее состояние, correlation ID и связанные logs. Fixed sleep хуже, потому что либо замедляет test, либо не учитывает реальные задержки.

**Объяснение:** Для Java можно использовать Awaitility или собственный polling helper, но конкретная библиотека в CV не указана.

#### 16. Как локализовать расхождение между UI, API и базой данных?

**Статус:** Факт из CV + техническая реконструкция.

**Готовый ответ:** Я бы начал с UI и проверил network request в DevTools: корректны ли параметры и какой response пришёл от backend. Затем сравнил бы response с данными в PostgreSQL или MongoDB и нашёл связанные logs по correlation ID в OpenSearch. Если database содержит правильное значение, а API возвращает неправильное, проблема вероятна в backend или cache. Если API правильный, а UI показывает другое, дефект вероятнее на frontend.

**Объяснение:** Это базовый алгоритм локализации client-server defect. Для окончательного вывода нужны evidence, а не только предположение.

### API и backend testing

#### 17. Что именно вы автоматизировали через REST Assured?

**Статус:** Факт из CV + требует подтверждения деталей.

**Готовый ответ:** В CV указана автоматизация integration testing REST API микросервисов через REST Assured и Maven. Я проверял бы request parameters, status code, headers, response body, schema, business rules и side effects в database или broker. Для interview-ready ответа нужно добавить один реальный endpoint, его method, основной positive scenario и один negative scenario. Название endpoint нельзя придумывать.

**Объяснение:** REST Assured — библиотека HTTP API testing, а Maven управляет dependencies и test execution.

#### 18. Что проверять в API кроме status code?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Кроме status code я бы проверил headers, content type, response time в рамках requirement, структуру body и значения business fields. Также важны error contract, authorization, pagination, filtering, sorting и idempotency. Для изменяющих operations нужно проверить side effects в database, cache или Kafka. Один `200 OK` не доказывает, что операция выполнена правильно.

**Объяснение:** Набор проверок зависит от endpoint. Не нужно механически проверять все свойства для каждого request.

#### 19. Как тестировать positive и negative API scenarios?

**Статус:** Техническая реконструкция.

**Готовый ответ:** В positive scenario я бы отправил валидный request и проверил полный business result. В negative scenarios изменял бы один фактор: обязательное поле, формат, boundary, authorization, несуществующий ID или недопустимый state transition. Проверил бы правильный `4xx` status, стабильный error code, понятное message и отсутствие нежелательного side effect. Ошибка клиента не должна приводить к `500`.

**Объяснение:** Изменение одного фактора упрощает локализацию и показывает, какое правило проверяется.

#### 20. Как проверять API contract?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Я бы сравнил endpoint, HTTP method, parameters, headers и schema request/response с OpenAPI specification. В automation можно проверить required fields, types, enum values и backward compatibility. Contract test не заменяет business validation: корректный JSON может содержать неправильные финансовые значения. При изменении schema важно проверить consumers и versioning.

**Объяснение:** Swagger или OpenAPI прямо не указан в окружении Java-проекта, поэтому его использование там требует подтверждения.

#### 21. Как проверить idempotency REST operation?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Я бы повторил одинаковый request с тем же idempotency key или business ID и проверил ожидаемое поведение по контракту. Система не должна создавать duplicate transaction или повторно применять финансовый эффект. Нужно сравнить responses, количество database records и число downstream events. Для `GET`, `PUT` и `DELETE` ожидаемая idempotency отличается от `POST`, поэтому решение зависит от API contract.

**Объяснение:** Idempotency означает одинаковый итоговый effect при повторении, а не обязательно полностью одинаковый response.

#### 22. Как тестировать authorization и роли?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Я бы подготовил пользователей с разными roles и проверил разрешённые и запрещённые operations. Negative checks включают отсутствие token, expired token, неправильный scope и попытку получить данные другого клиента. Важно проверять не только HTTP status, но и отсутствие утечки данных в body и logs. Для финансового UI также нужна согласованность ограничений между UI и API.

**Объяснение:** Конкретный protocol авторизации в CV не указан.

#### 23. Как организовать API client в automation framework?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Я бы вынес HTTP configuration, authentication, serialization и logging в общий client layer. Service-specific clients содержали бы методы business API, а tests — только scenario и assertions. Request и response models можно описать отдельными DTO, test data создавать через Builder. Такой подход уменьшает duplication и позволяет менять endpoint details в одном месте.

**Объяснение:** REST Assured может находиться внутри client layer. Tests не должны повторять полный low-level request setup.

#### 24. Как диагностировать падение REST Assured test?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Сначала я бы определил, упал test на подготовке данных, request, response validation или post-condition. Проверил бы URL и environment, request, status, response body и authentication без вывода secrets. Затем использовал correlation ID для поиска logs и сравнил database state. Если ручной request проходит, проверил бы test data, parallel execution и различия configuration.

**Объяснение:** Полезный report должен сохранять достаточно evidence, но маскировать tokens, passwords и персональные данные.

### PostgreSQL, MongoDB и Redis

#### 25. Для чего в проекте могли использовать PostgreSQL?

**Статус:** Техническая реконструкция.

**Готовый ответ:** PostgreSQL вероятно использовался для relational data, где важны constraints, relations и transactions. Это могли быть финансовые операции, users, reference data или статусы обработки, но точные tables в CV не указаны. При тестировании я бы проверял primary и foreign keys, обязательные поля, uniqueness, точные значения и rollback при ошибке. Конкретное назначение базы нужно подтвердить.

**Объяснение:** Не называй PostgreSQL единственным source of truth без знания архитектуры.

### Kafka, асинхронность, WireMock и Mockito

#### 32. Зачем проекту могла быть нужна Kafka?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Kafka могла передавать financial events между независимыми services и позволять обрабатывать большой поток данных асинхронно. Producer публикует record в topic, а consumers читают его в составе consumer groups. Kafka хранит ordering только внутри partition, поэтому для связанных операций важен правильный message key. Точная роль Kafka и список topics в CV не указаны.

**Объяснение:** Корректнее говорить «topics Kafka» или «взаимодействия через Kafka», а не называть Kafka обычной очередью.

#### 33. Как проверить настоящий Kafka flow?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Для integration test нужен доступ к тестовому broker или container с Kafka. Test публикует message с уникальным key, ждёт обработки, затем проверяет output topic, database или API state. Нужно проверить serialization, headers, schema, business payload и side effects. Mockito alone этого не делает, потому что mock не проверяет network, broker, partitioning и delivery.

**Объяснение:** Варианты окружения — Testcontainers, Embedded Kafka или отдельный test cluster. Конкретный вариант проекта требует подтверждения.

#### 34. Как тестировать асинхронную обработку без fixed sleep?

**Статус:** Техническая реконструкция.

**Готовый ответ:** После отправки event я бы использовал polling: через заданный interval проверял expected state до timeout. Condition может читать output topic, database, API или logs. После timeout test должен показать последнее состояние и correlation ID. Такой подход быстрее стабильного `Thread.sleep()` при быстрой обработке и надёжнее при временной задержке.

**Объяснение:** Timeout и interval выбираются по SLA и характеристикам environment, а не случайно.

#### 35. Как проверить duplicate event и idempotency consumer?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Я бы дважды отправил event с одинаковым business ID и проверил итоговый effect. Consumer не должен дважды начислить сумму, создать duplicate record или повторно изменить status, если business rule требует idempotency. Затем сравнил бы database, output events и logs. Важно отличать техническую повторную доставку от двух разных business operations.

**Объяснение:** At-least-once delivery делает duplicate handling важной частью consumer logic.

#### 36. Как проверить ordering в Kafka?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Я бы отправил несколько связанных events с одним key и проверил их обработку в ожидаемом порядке. Один key обычно направляет records в одну partition, где Kafka сохраняет ordering. Затем проверил бы итоговый state и поведение при event с более старой version или timestamp. Глобальный порядок между разными partitions Kafka не гарантирует.

**Объяснение:** Test должен проверять business ordering, а не только последовательность получения records.

#### 37. Как тестировать retry и ошибочное message?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Я бы создал временную ошибку dependency и проверил количество retries, интервалы и успешную обработку после восстановления. Для постоянной ошибки проверил бы ограничение retries и попадание message в DLQ или другой механизм обработки ошибок. В logs должны быть correlation ID и понятная причина. Poison message не должно бесконечно блокировать обработку остальных records.

**Объяснение:** DLQ не является обязательной встроенной моделью Kafka; это архитектурное решение приложения.

#### 38. Как соотносятся Kafka, WireMock и Mockito?

**Статус:** Требует подтверждения.

**Готовый ответ:** Эти инструменты решают разные задачи. Kafka обеспечивает асинхронную передачу events, WireMock эмулирует HTTP dependencies, а Mockito подменяет Java objects в unit tests. Они могут использоваться в одном business scenario, например consumer получает Kafka event и вызывает внешний HTTP service, который заменён WireMock. Но WireMock и Mockito сами по себе не проверяют настоящий Kafka broker, поэтому исходную формулировку CV нужно уточнить.

**Объяснение:** На интервью нужно назвать, чем именно создавали и читали Kafka messages. Без этого нельзя утверждать полноценный Kafka integration test.

#### 39. Что можно проверить через Mockito в Kafka-related code?

**Статус:** Техническая реконструкция.

**Готовый ответ:** Через Mockito можно подменить producer client, repository или внешний service и проверить логику отдельного Java class. Например, убедиться, что после валидной операции class создаёт event с нужными fields и один раз вызывает producer abstraction. Такой unit test быстрый, но не проверяет serialization, broker, topic configuration, delivery и consumer processing. Для этого нужен отдельный integration test.

**Объяснение:** TestNG или JUnit не определяет test level. Уровень зависит от реальных dependencies и границы проверяемого компонента.

Точные tables, keys и cache policy нельзя придумывать.

**Объяснение:** Формулировка «валидация кэширования» должна раскрываться конкретной проверкой.

#### 59. Что проверяли unit tests расчёта доставки с Mockito?

**Статус:** Факт из CV + требует подтверждения границы test.

**Готовый ответ:** Формулировка CV описывает unit tests business logic расчёта доставки на TestNG с Mockito. Это корректно, если test напрямую вызывал application class, а внешние логистические dependencies были заменены mocks. Такой test проверяет input, calculated result, ошибки и вызовы dependency, но не network и реальный логистический service. Если проверка выполнялась через REST API, её нужно назвать component или integration test.

**Объяснение:** TestNG является runner и не определяет test level; границу задаёт проверяемый code и dependencies.

#### 60. Как Gradle, Jenkins и Docker Compose участвовали в запуске tests?

Вероятный pipeline сначала собирал project, затем запускал быстрые tests, после них API и UI suites и публиковал reports. Точные stages, triggers и artifacts нужно подтвердить. Важно разделять личную настройку pipeline и использование уже готовой job.

**Объяснение:** CV подтверждает интеграцию Selenium tests в Jenkins и работу с Docker Compose, но не полный pipeline design.

## Рискованные формулировки CV

> [!caution] Что необходимо уточнить до интервью
> | Формулировка | Риск | Безопасная версия |
> |---|---|---|
> | «28 лет» при дате рождения 29.07.1998 | До 29 июля 2026 года возраст — 27 лет | Указать 27 лет или убрать возраст |
> | «Selenium для мобильных представлений» | Selenium не автоматизирует native Mobile application | Уточнить: mobile web через Selenium, native/hybrid через Appium или ручное Mobile testing |
> | «Kafka с WireMock и Mockito» | WireMock эмулирует HTTP, Mockito изолирует Java objects; они не заменяют Kafka broker | Разделить Kafka integration, HTTP stubs и unit tests |
> | «Модульные tests с Mockito» | Unit level корректен только при прямой проверке class с mocked dependencies | Если вход через API или real dependencies, назвать component/integration test |
> | «Форма заказа с Postman» | Можно понять так, будто Postman тестирует UI | Разделить ручную проверку формы и воспроизведение HTTP requests в Postman |
> | Два проекта в 10.2025 | Возможный вопрос о пересечении периодов | Уточнить месяц перехода или объяснить реальный handover |
> | «Framework с нуля» | Требует глубокого объяснения architecture и личного вклада | Назвать созданные layers, components и принятые решения либо смягчить формулировку |
> | SoapUI и SOAP в общем стеке | В обязанностях проектов применение не показано | Добавить реальный пример или убрать из основного стека |
