## Как пользоваться заметкой

Заметка собирает правдоподобные достижения и челленджи AQA-инженера на двух проектах из CV. Это **техническая реконструкция**: модели того, что обычно делает AQA на таких системах, а не подтверждённый production-опыт.

> [!warning] Как это подавать
> Не пересказывай эти кейсы как факты, которых не было. Используй их как каркас: выбери ситуации, которые сможешь защитить вживую, и рассказывай через «я бы сделал» / «на таком проекте типично». Если интервьюер копнёт глубже, а ты не сможешь продолжить — кейс провалится. Сильная сторона не в заученной истории, а в умении рассуждать о сценарии на лету.

Каждый кейс построен по схеме: **ситуация → действие → почему это ценно**.

## Финансовая экосистема — Java

### Достижения — финансовая экосистема

- **Сквозная проверка данных по business ID.** Одна финансовая операция проходит путь Kafka → обработка → PostgreSQL → MongoDB → Redis → API → UI. Типичная сильная практика: тест создаёт операцию с уникальным ID и проходит с этим ID по каждому слою, сверяя значения. Это превращает «проверил API и готово» в полноценную E2E-верификацию и ловит расхождения между хранилищами, которые иначе не найти.
- **Структура UI-фреймворка, а не просто тесты.** Разделение слоёв: driver management, Page/Component Objects, business flows, конфигурация окружений, test data через Builder, reporting. Такое построение позволяет добавлять проверки без дублирования и менять локаторы в одном месте.
- **Изоляция backend-тестов в Docker.** PostgreSQL и MongoDB поднимаются локально через Docker Compose, чтобы тесты не зависели от общего окружения и не конфликтовали с чужими данными. Тесты становятся воспроизводимыми и пригодными для параллельного запуска.
- **UI- и API-тесты в GitLab CI/CD с артефактами.** При падении job сохраняет screenshots, page source и логи как artifacts. Это сокращает разбор ситуации «в pipeline упало, а локально зелёное».

### Челленджи — финансовая экосистема

- **Расхождение PostgreSQL и MongoDB из-за асинхронности.** MongoDB обновляется с задержкой, поэтому assert сразу после запроса падает нестабильно. Решение — polling ожидаемого состояния с timeout (Awaitility) вместо fixed sleep и мгновенных проверок.
- **Stale cache в Redis.** Данные обновились в БД, а API продолжает отдавать старое из кеша, потому что invalidation не срабатывает при определённом сценарии. Дефект находится сравнением значения в Redis, в БД и в API response.
- **Локализация дефекта в распределённой системе через OpenSearch.** Тест упал, но причина не в UI и не в API, а в одном из сервисов обработки. По correlation ID находятся связанные логи в OpenSearch, и проблема сужается до конкретного сервиса. Без этого дефект ушёл бы как «не воспроизводится».

## Интернет-магазин — Kotlin

### Достижения — интернет-магазин

- **Checkout покрыт на нескольких уровнях.** Один и тот же сценарий оформления заказа проверяется через REST Assured (API-контракт и бизнес-правила), через Selenium (web UI), через Appium (mobile) и через проверку итоговой записи в PostgreSQL. Дефект на любом уровне ловится своим тестом.
- **Эмуляция платёжного шлюза через WireMock.** Реальный payment-сервис нестабилен и медленный, поэтому для backend-тестов он заменяется WireMock-заглушкой. Это даёт быстрые и детерминированные тесты, включая сценарии timeout и ошибки оплаты, которые на реальном сервисе не воспроизвести.
- **Расчёт доставки изолирован в unit-тестах с Mockito.** Логика расчёта стоимости доставки зависит от внешнего логистического сервиса. Он заменяется mock, и граничные случаи (вес, расстояние, зоны) покрываются быстро и без сети.
- **Локальное окружение через Docker Compose.** PostgreSQL, Redis и RabbitMQ поднимаются локально, тесты запускаются в изоляции и не трогают shared-среду.

### Челленджи — интернет-магазин

- **Расхождение цены между корзиной и checkout.** Товар лежит в корзине по одной цене, а к моменту оформления цена меняется, и итоговая сумма в заказе не совпадает с ожидаемой. Тест ловит это, потому что сверяет финальную сумму в API и в БД, а не только статус.
- **Разные локаторы одного элемента на iOS и Android в Appium.** Один и тот же элемент находится по-разному на платформах. Platform-specific локаторы выносятся в отдельный слой, чтобы тесты не дублировались под каждую ОС.
- **Сообщение ушло в DLQ без видимой ошибки.** Consumer упал, ретраи закончились, сообщение попало в dead-letter queue, а статус заказа остался «в обработке». Тест проверяет итоговый бизнес-статус и поэтому выявляет проблему, которую не показал бы просто HTTP-ответ.

## Структура самопрезентации

Речь на 3–4 минуты собирается из этих блоков:

1. **Вступление (20–30 секунд).** Кто ты, общий опыт, основной стек (Java/Kotlin, Selenium, REST Assured, JUnit 5/TestNG).
2. **Проекты (60–90 секунд).** Коротко о каждом: финансовая экосистема — консолидация и визуализация финансовых данных; интернет-магазин — путь заказа от корзины до доставки.
3. **Пара сильных кейсов (60–90 секунд).** Один-два кейса из заметки, которые ты можешь защитить вживую. Лучше один, но рассказанный глубоко, чем три поверхностных.
4. **Куда растёшь (20–30 секунд).** Какие области интересны дальше: углубление в архитектуру тестирования, performance, инфраструктура.

> [!tip] Выбор кейса
> Бери кейс, в котором понимаешь каждый шаг. На интервью ценят не количество историй, а то, что ты можешь объяснить, почему сделано именно так, и что бы изменил.
