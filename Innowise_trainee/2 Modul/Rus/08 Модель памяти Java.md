## Зачем знать про память

Когда ты пишешь `int x = 5;` или `new User("Alice")`, Java хранит эти данные где-то в памяти. Понимание **где** и **как** помогает:

- Предсказывать, когда объекты уничтожаются (важно для управления ресурсами в тестах)
- Понимать, почему `==` ведёт себя по-разному для примитивов и объектов
- Отлаживать `NullPointerException` и `OutOfMemoryError`
- Отвечать на частые вопросы на собеседованиях

В Java не нужно управлять памятью вручную (в отличие от C/C++). Но нужно понимать модель, чтобы писать корректный код и объяснять свои решения.

## Stack и Heap

Память Java runtime делится на две основные области:

### Stack (стек)

**Стек** — быстрая, упорядоченная область памяти. У каждого потока свой стек. Работает как стопка тарелок — последний пришёл, первый ушёл (LIFO).

Что живёт на стеке:
- **Локальные переменные** (примитивы и ссылки)
- **Фреймы вызовов методов** (какой метод выполняется, его параметры, локальные переменные)
- **Адреса возврата** (куда вернуться после завершения метода)

Стек **автоматически очищается** при возврате из метода — его фрейм удаляется, и все локальные переменные в нём исчезают.

### Heap (куча)

**Куча** — общая область памяти для всех потоков. Она больше и медленнее стека.

Что живёт в куче:
- **Все объекты** (созданные через `new`)
- **Массивы** (даже `int[]` — массивы являются объектами)
- **String Pool** (специальная область внутри кучи)
- **Статические поля** (хранятся в области метаданных класса)

Объекты в куче живут, пока **сборщик мусора** не удалит их.

### Визуальная модель

```
STACK (у каждого потока свой)      HEAP (общая)
+------------------+               +------------------------+
| main()           |               |                        |
|   int age = 25   |  ----------  |  Объект User           |
|   User u = ref --|------------->|  { name: "Alice",      |
|                  |               |    age: 25 }           |
+------------------+               |                        |
| login()          |               |  String "Alice"        |
|   String s = ref-|------------->|  (в String Pool)       |
|   int count = 0  |               |                        |
+------------------+               +------------------------+
```

> [!danger] Ключевое правило
> Примитивы хранятся **там, где они объявлены**. Если объявлены внутри метода -> стек. Если объявлены как поле объекта -> куча (внутри объекта). Ссылки всегда на стеке (или в объекте), но объект, на который они указывают, всегда в куче.

## Что где хранится

Это самый частый вопрос о памяти в Java. Ответ зависит от того, **где объявлена переменная**, а не от её типа.

### Локальные переменные (внутри метода)

```java
public void calculate() {
    int x = 10;              // примитив -> стек
    double pi = 3.14;        // примитив -> стек
    String name = "Alice";   // ссылка -> стек, объект -> куча (пул)
    User user = new User();  // ссылка -> стек, объект -> куча
    int[] arr = {1, 2, 3};   // ссылка -> стек, массив -> куча
}
// когда calculate() завершается -> все ссылки удаляются со стека
// объекты в куче становятся доступны для GC, если других ссылок нет
```

### Поля экземпляра (внутри объекта)

```java
public class User {
    private int age;           // живёт внутри объекта User в куче
    private String name;       // ссылка внутри объекта User, String в куче
    private int[] scores;      // ссылка внутри объекта User, массив в куче
}
```

Когда ты создаёшь `new User()`, JVM выделяет один блок памяти в куче для всего объекта — включая место для `age`, `name` и `scores`.

### Статические поля

```java
public class Config {
    static String url = "https://example.com";   // хранится в метаданных класса
    static int timeout = 30;                      // хранится в метаданных класса
}
```

Статические поля хранятся в специальной области, связанной с самим классом (metaspace в современной JVM). Они живут всё время работы программы.

> [!tip] Краткая справка
> | Объявление | Значение примитива | Ссылка | Объект |
> |---|---|---|---|
> | Локальная переменная | Стек | Стек | Куча |
> | Поле экземпляра | Куча (в объекте) | Куча (в объекте) | Куча |
> | Статическое поле | Metaspace | Metaspace | Куча |

## Стек вызовов методов

Когда метод вызывается, новый **фрейм** помещается на стек. Когда метод завершается, фрейм снимается.

```java
public class Demo {
    public static void main(String[] args) {        // фрейм 1 (дно)
        int result = multiply(3, 4);                 // фрейм 2 помещается
        System.out.println(result);
    }

    static int multiply(int a, int b) {              // фрейм 2
        int product = a * b;
        return product;                              // фрейм 2 снимается
    }
}
```

**Стек во время выполнения `multiply()`:**

```
| multiply()          |  <- вершина (текущий)
|   a = 3, b = 4     |
|   product = 12      |
|---------------------|
| main()              |  <- ожидает
|   args = ref        |
|   result = ???      |
+---------------------+
```

### StackOverflowError

Если методы вызывают сами себя бесконечно (плохая рекурсия), стек переполняется:

```java
public void oops() {
    oops();   // не останавливается -> StackOverflowError
}
```

> [!info] Размер стека
> Размер стека по умолчанию — ~512КБ–1МБ на поток. Этого достаточно для большинства программ. Глубокая рекурсия или очень большие локальные массивы могут исчерпать его. Увеличить можно флагом JVM `-Xss`, но обычно правильнее избегать глубокой рекурсии.

## Сборка мусора

В Java не нужно освобождать память вручную. **Сборщик мусора (Garbage Collector, GC)** автоматически находит и удаляет объекты, до которых больше нельзя добраться.

### Когда объект становится доступен для GC?

Объект становится доступным для сборки, когда на него **не указывает ни одна живая ссылка**:

```java
User user = new User("Alice");  // объект User доступен через 'user'
user = new User("Bob");         // объект "Alice" теперь недоступен -> доступен для GC
user = null;                    // объект "Bob" теперь недоступен -> доступен для GC
```

### Как работает GC (упрощённо)

1. **Фаза разметки (Mark)** — GC начинает с "корней" (локальные переменные, статические поля, стеки потоков) и помечает каждый достижимый объект
2. **Фаза очистки (Sweep)** — недостижимые (непомеченные) объекты удаляются, их память освобождается
3. JVM запускает GC автоматически, когда в куче заканчивается место

### Поколенческая сборка

JVM делит кучу на области по возрасту объектов:

| Область | Содержит | Частота GC |
|---|---|---|
| **Young Generation** | Недавно созданные объекты | Часто (minor GC) |
| **Old Generation** | Долгоживущие объекты (пережили много GC) | Редко (major GC) |

Большинство объектов умирают молодыми (временные переменные, итерации цикла). Young generation собирается часто и быстро. Объекты, пережившие несколько сборок, перемещаются в Old Generation.

### Чего делать нельзя

- Нельзя принудительно запустить GC. `System.gc()` — лишь **рекомендация**, JVM может её проигнорировать.
- Нельзя предсказать, **когда** запустится GC.
- Нельзя контролировать, **какие** объекты будут собраны.

> [!caution] finalize() устарел
> Метод `finalize()` (вызываемый перед сборкой объекта) устарел с Java 9 и удалён в Java 18. Не используй его. Для очистки ресурсов используй try-with-resources.

## null и NullPointerException

`null` означает "эта ссылка никуда не указывает". Это не объект — это отсутствие объекта.

```java
String name = null;    // ссылка существует, но указывает в никуда
name.length();          // NullPointerException — нет объекта для вызова length()
```

### Частые причины NullPointerException

```java
// 1. Неинициализированное поле объекта
User user = new User();
user.getAddress().getCity();     // если getAddress() вернёт null -> NPE

// 2. Метод возвращает null
String result = map.get("key"); // вернёт null, если ключ не найден
result.toUpperCase();            // NPE

// 3. Элемент массива равен null
String[] names = new String[3]; // [null, null, null]
names[0].length();               // NPE
```

### Как предотвратить NPE

```java
// 1. Проверка на null
if (name != null) {
    System.out.println(name.length());
}

// 2. Безопасный порядок вызова методов
"expected".equals(name);   // безопасно — "expected" никогда не null
// name.equals("expected"); // небезопасно — name может быть null

// 3. Optional (Java 8+) — рассмотрен в отдельной теме
Optional<String> maybeName = Optional.ofNullable(name);
maybeName.ifPresent(n -> System.out.println(n.length()));
```

> [!tip] NPE в AQA
> `NullPointerException` — самая частая ошибка в автотестировании. Типичные причины: WebDriver возвращает `null`, когда элемент не найден, поле page object не инициализировано, или метод подготовки тестовых данных вернул `null`. Всегда проверяй на `null` при работе с внешними данными.

## Утечки памяти в Java

У Java автоматическое управление памятью, но утечки всё равно возможны. **Утечка памяти** происходит, когда объекты больше не нужны, но на них всё ещё есть ссылки — GC не может их собрать.

### Типичные паттерны утечек

```java
// 1. Забытые ссылки в коллекциях
List<User> cache = new ArrayList<>();
while (true) {
    cache.add(new User(...));   // список растёт бесконечно — старые пользователи не удаляются
}

// 2. Статические коллекции
static Map<String, Object> globalCache = new HashMap<>();
// объекты в статических коллекциях живут всю жизнь программы

// 3. Незакрытые ресурсы
Connection conn = DriverManager.getConnection(url);
// если забыть conn.close(), объект соединения и его буферы остаются в памяти
```

### Предотвращение

- Закрывай ресурсы через try-with-resources
- Используй слабые ссылки (`WeakHashMap`) для кешей
- Не храни всё подряд в статических коллекциях
- В тестах: всегда вызывай `driver.quit()` в `@AfterEach` — WebDriver занимает много памяти

> [!warning] OutOfMemoryError
> Когда куча заполнена и GC не может освободить достаточно места, JVM выбрасывает `OutOfMemoryError`. Частые причины: утечка памяти, слишком много тестовых данных загружено одновременно, или размер кучи слишком мал. Увеличить можно флагом `-Xmx`: `java -Xmx2g MyApp`.

## Вопросы для собеседования

### Вопросы начального уровня

**1. В чём разница между стеком и кучей?**
Стек — у каждого потока свой, хранит локальные переменные и фреймы методов, быстрый, LIFO, автоматически очищается при возврате из метода. Куча — общая, хранит все объекты, медленнее, управляется сборщиком мусора.

**2. Где хранятся локальные переменные? Где хранятся объекты?**
Локальные примитивные переменные — на стеке. Локальные ссылочные переменные тоже на стеке, но объекты, на которые они указывают — в куче. Все объекты живут в куче.

**3. Что такое сборка мусора (Garbage Collection)?**
Автоматическое управление памятью. GC находит объекты без живых ссылок (недостижимые) и освобождает их память. Программист не освобождает память вручную.

**4. Когда объект становится доступен для сборки мусора?**
Когда на него не указывает ни одна живая ссылка. Это происходит когда: ссылке присвоили `null`, ссылка вышла из области видимости (метод завершился), или ссылке присвоили другой объект.

**5. Что такое `NullPointerException`?**
Исключение, выбрасываемое при попытке использовать ссылку, указывающую на `null` — вызвать метод, обратиться к полю или обратиться к элементу массива через null-ссылку.

**6. Что такое String Pool?**
Специальная область в куче, где Java хранит строковые литералы. При создании `"hello"` JVM сначала проверяет пул и переиспользует существующий объект, если найден. Это экономит память, потому что строковые литералы часто повторяются.

**7. Можно ли принудительно запустить сборку мусора?**
Нет. `System.gc()` — это запрос, а не команда. JVM сама решает, когда и как запускать GC. На практике нельзя полагаться на `System.gc()`.

**8. Что такое `StackOverflowError`?**
Ошибка, выбрасываемая при переполнении стека, обычно из-за бесконечной или слишком глубокой рекурсии. Каждый вызов метода добавляет фрейм на стек — если он никогда не возвращается, стек заполняется.

**9. Что такое поколения в Java GC?**
Young Generation (новые объекты, собирается часто) и Old Generation (долгоживущие объекты, собирается редко). Большинство объектов умирают молодыми, поэтому частая сборка Young Generation эффективна.

**10. Что такое metaspace?**
Metaspace (Java 8+) хранит метаданные классов — определения классов, данные методов и **статические поля**. Он не является частью кучи Java; использует нативную память и растёт автоматически. Здесь живут статические поля вроде `static int timeout`.

### Вопросы среднего уровня

**1. В чём разница между `StackOverflowError` и `OutOfMemoryError`?**
`StackOverflowError` — стек потока заполнен (слишком много вложенных вызовов методов). `OutOfMemoryError` — куча заполнена (слишком много объектов, утечка памяти, или куча слишком мала). Оба — `Error`, а не `Exception`.

**2. Где живёт примитивное поле объекта — на стеке или в куче?**
В куче. Поле является частью объекта, а объект находится в куче. Только локальные примитивные переменные внутри методов живут на стеке.

**3. Возможна ли утечка памяти в Java?**
Да. Если на объект всё ещё есть ссылка, но он больше не нужен (например, в растущей статической коллекции), GC не может его собрать. Это логическая утечка памяти — ссылка существует, поэтому GC считает объект живым.

**4. Что произойдёт, если `finalize()` выбросит исключение?**
Исключение будет молча проигнорировано, и объект будет финализирован в любом случае. Это одна из многих причин, почему `finalize()` ненадёжен и устарел (удалён в Java 18).

**5. В чём разница между `==` и `.equals()` для объектов и обёрток?**
`==` сравнивает ссылки (две переменные указывают на один объект). `.equals()` сравнивает содержимое. Для обёрток `==` может удивить из-за кэширования — всегда используй `.equals()`.

### Вопросы повышенного уровня

**1. Может ли объект быть "воскрешён" во время финализации?**
Да. Если `finalize()` сохранит `this` в статическом поле, объект снова станет достижимым и GC его не соберёт. Это называется object resurrection — ужасная практика, делающая код непредсказуемым. Важно: `finalize()` вызывается максимум один раз для объекта.

**2. Java передаёт параметры по значению или по ссылке?**
**По значению.** Всегда. Для примитивов копируется само значение. Для объектов копируется **значение ссылки** (адрес объекта), а не сам объект. Внутри метода можно менять поля переданного объекта, но нельзя заменить сам объект — внешняя ссылка остаётся прежней.

```java
public void change(String s, StringBuilder sb) {
    s = "new";              // внешняя ссылка не изменится
    sb.append(" world");    // объект изменится — ссылка та же
}
// Вызов:
String a = "hello";
StringBuilder b = new StringBuilder("hello");
change(a, b);
System.out.println(a); // "hello" — не изменилось
System.out.println(b); // "hello world" — изменилось
```

### Вопросы с кодом

**1. Где живёт каждая переменная?**
```java
public class MemoryTest {
    static int staticCounter = 0;              // ?
    int instanceValue;                         // ?

    public void test() {
        int localPrimitive = 5;                // ?
        User user = new User("Alice");         // user (ссылка) — ?, User (объект) — ?
        int[] arr = {1, 2, 3};                 // arr (ссылка) — ?, массив — ?
    }
}
```
**Ответ:** `staticCounter` — metaspace. `instanceValue` — куча (внутри объекта MemoryTest). `localPrimitive` — стек. `user` (ссылка) — стек. `User` (объект) — куча. `arr` (ссылка) — стек. массив `{1,2,3}` — куча.
**Разбор:** Примитивы живут там, где объявлены (локальные — стек, поля — куча). Ссылки живут там, где объявлены. Объекты (new) — всегда в куче. Статика — metaspace.

**2. Что выведет этот код? (String Pool — `==` vs `equals`)**
```java
String s1 = "hello";
String s2 = "hello";
String s3 = new String("hello");
System.out.println(s1 == s2);
System.out.println(s1 == s3);
System.out.println(s1.equals(s3));
```
**Ответ:** `true`, `false`, `true`. `s1` и `s2` — один объект из String Pool (== true). `s3` — новый объект в куче (== false). `equals()` сравнивает содержимое — true.

**3. Что выведет этот код? (кэш Integer)**
```java
Integer a = 127;
Integer b = 127;
Integer c = 128;
Integer d = 128;
System.out.println(a == b);
System.out.println(c == d);
```
**Ответ:** `true`, `false`. Integer кэширует значения от -128 до 127. Для 127 возвращается один кэшированный объект. Для 128 создаются новые объекты. Это касается всех обёрток: `Byte`, `Short`, `Integer`, `Long` (до 127), `Character` (до 127), `Boolean`. Всегда сравнивай обёртки через `.equals()`, не через `==`.

**4. На каком шаге каждый объект становится доступен для GC?**
```java
public class GcTest {
    public static void main(String[] args) {
        User u1 = new User("Alice");  // шаг 1
        User u2 = new User("Bob");    // шаг 2
        u1 = u2;                      // шаг 3
        u2 = null;                    // шаг 4
        System.gc();                  // шаг 5
    }
}
```
**Ответ:** После шага 3 объект "Alice" теряет последнюю ссылку (u1 теперь указывает на "Bob") — доступен для GC. После шага 4 и u1, и u2 фактически указывают на null для "Bob" — "Bob" тоже доступен для GC. `System.gc()` — лишь рекомендация, а не команда.

## Практические задачи

> [!tip] Как тренироваться
> Сначала попробуй решить сам, потом сверься с решением.

### 1. Где хранится переменная?

**Задание:** Определи, где в памяти хранится каждая переменная: стек, куча или metaspace.

```java
public class MemoryTest {
    static int staticCounter = 0;              // ?
    int instanceValue;                         // ?

    public void test() {
        int localPrimitive = 5;                // ?
        User user = new User("Alice");         // user (ссылка) — ?, User (объект) — ?
        int[] arr = {1, 2, 3};                 // arr (ссылка) — ?, массив — ?
    }
}
```

**Ответ:** `staticCounter` — metaspace. `instanceValue` — куча (внутри объекта MemoryTest). `localPrimitive` — стек. `user` (ссылка) — стек. `User` (объект) — куча. `arr` (ссылка) — стек. массив `{1,2,3}` — куча.

**Разбор:** Примитивы живут там, где объявлены (локальные — стек, поля — куча). Ссылки — там же, где объявлены. Объекты (new) — всегда куча. Статика — metaspace.

### 2. String Pool — == vs equals

**Задание:** Напиши, что выведет каждая строка и почему.

```java
String a = "hello";
String b = "hello";
String c = new String("hello");
String d = c.intern();

System.out.println(a == b);      // ?
System.out.println(a == c);      // ?
System.out.println(a == d);      // ?
System.out.println(c == d);      // ?
```

**Ответ:** `true` (оба из пула), `false` (c — новый объект в куче), `true` (intern вернул ссылку из пула), `false` (d из пула, c из кучи).

**Разбор:** Литералы автоматически интернируются в String Pool (a и b — один объект). `new String()` создаёт новый объект в куче. `intern()` возвращает каноническую ссылку из пула.

### 3. Когда объект готов к GC?

**Задание:** На каком шаге объект User становится доступен для сборки мусора?

```java
public class GcTest {
    public static void main(String[] args) {
        User u1 = new User("Alice");  // шаг 1
        User u2 = new User("Bob");    // шаг 2
        u1 = u2;                      // шаг 3
        u2 = null;                    // шаг 4
        System.gc();                  // шаг 5
    }
}
```

**Ответ:** После шага 3 объект "Alice" теряет последнюю ссылку (u1 теперь указывает на "Bob") — доступен для GC. После шага 4 оба объекта u1 и u2 указывают на null — "Bob" тоже доступен для GC. `System.gc()` — лишь рекомендация.

**Разбор:** Объект доступен для GC, когда на него не указывает ни одна живая ссылка. Переприсваивание ссылки или присвоение null — самые частые причины.

### 4. Integer кэш — 127 vs 128

**Задание:** Что выведет этот код?

```java
Integer a = 100;
Integer b = 100;
Integer c = 200;
Integer d = 200;
System.out.println(a == b);
System.out.println(c == d);
```

**Ответ:** `true`, `false`. Integer кэширует значения от -128 до 127. Для 100 возвращается один объект из кэша. Для 200 — разные объекты. Это же касается Byte, Short, Long (до 127), Character (до 127), Boolean.

**Разбор:** Автоупаковка (autoboxing) использует `Integer.valueOf()`, который возвращает кэшированный объект для малых значений. Для сравнения обёрток всегда используй `.equals()`, не `==`.

### 5. OutOfMemoryError

**Задание:** Напиши код, который гарантированно вызовет OutOfMemoryError. Сколько объектов ты сможешь создать до падения?

```java
public class OomDemo {
    public static void main(String[] args) {
        List<byte[]> list = new ArrayList<>();
        int count = 0;
        try {
            while (true) {
                list.add(new byte[10 * 1024 * 1024]); // 10 MB each
                count++;
            }
        } catch (OutOfMemoryError e) {
            System.out.println("Created " + count + " blocks before OOM");
        }
    }
}
```

**Разбор:** Каждый блок — 10 MB. Количество зависит от размера кучи (`-Xmx`). Если куча 256 MB — создастся примерно 20-25 блоков до OOM. В catch можно поймать Error, но программа после этого в нестабильном состоянии.
