## Что такое SOAP

**SOAP** — это протокол обмена сообщениями между системами. Обычно он использует HTTP, но может работать и через другие транспортные протоколы. Данные передаются в XML.

**Цель:** дать системам строгий и заранее описанный контракт взаимодействия.

SOAP часто встречается в банковских, государственных и корпоративных системах, где важны строгие правила, безопасность и совместимость.

---

## Структура SOAP-сообщения

SOAP-запрос состоит из `Envelope`, внутри которого находятся:

- `Header` — необязательные служебные данные: аутентификация, идентификатор запроса, настройки безопасности;
- `Body` — обязательные данные запроса или ответа;
- `Fault` — описание ошибки, если сервер не смог обработать запрос.

```xml
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <GetUserRequest>
      <id>42</id>
    </GetUserRequest>
  </soap:Body>
</soap:Envelope>
```

> [!caution] XML чувствителен к структуре
> Неправильное имя элемента, пространство имён или вложенность могут сделать запрос невалидным, даже если данные выглядят правильно.

## WSDL и контракт

**WSDL** — XML-файл, который описывает SOAP-сервис: доступные операции, входные и выходные сообщения, адрес сервиса и правила обмена.

По WSDL можно понять:

- какие операции поддерживает сервис;
- какие поля обязательны;
- какие типы данных ожидаются;
- как выглядит ответ и ошибка.

Для тестировщика WSDL — основной источник ожидаемого контракта. Он не заменяет бизнес-требования, но помогает проверить техническую корректность.

## SOAP и REST

> [!tip] Краткое сравнение
>
> | Критерий | SOAP | REST |
> |---|---|---|
> | Основа | Протокол | Архитектурный стиль |
> | Формат данных | Обычно XML | Часто JSON, возможен XML |
> | Контракт | Обычно WSDL | Может быть OpenAPI |
> | Ошибки | SOAP Fault | HTTP status code и тело ответа |
> | Типичный случай | Строгие корпоративные интеграции | Веб- и мобильные API |

## Работа в SoapUI

**SoapUI** — инструмент для ручной проверки SOAP API. Он умеет импортировать WSDL и создавать готовые шаблоны запросов.

Обычный порядок работы:

1. Создать SOAP-проект и указать URL WSDL.
2. Выбрать нужную операцию.
3. Заполнить значения в сгенерированном XML.
4. Отправить запрос и изучить ответ.
5. Добавить проверки: статус, XPath-выражение, время ответа или содержимое `Fault`.

Для повторяющихся сценариев можно использовать TestSuite и TestCase. Тестовые данные лучше хранить отдельно от самого запроса.

---

## Что проверять в SOAP API

- успешный ответ и ожидаемые данные в `Body`;
- обязательные и необязательные поля;
- неверные типы, пустые и граничные значения;
- корректный `SOAP Fault` для ошибки;
- аутентификацию и права доступа;
- соответствие ответа WSDL и бизнес-требованиям;
- побочный эффект: создана ли запись, изменился ли статус, отправилось ли событие.

## Типичные ошибки

| Ошибка | Как сделать лучше |
|---|---|
| Проверять только HTTP 200 | Проверять содержимое `Body` и результат операции |
| Игнорировать `Fault` | Проверять код и понятность ошибки |
| Менять XML вручную без проверки структуры | Использовать шаблон из WSDL и валидировать XML |
| Считать WSDL полным описанием поведения | Сверять контракт с бизнес-требованиями |
