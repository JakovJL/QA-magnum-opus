## Логирование

**Logging** записывает, что происходило во время test run.

**Цель:** помочь разобраться с failure после завершения запуска, особенно в CI.

Полезные logs включают:

- start и finish теста
- environment и browser
- важные ids тестовых данных
- API request и response status
- error messages

Распространённые Java logging tools:

- SLF4J
- Logback
- Log4j2

> [!caution] Не логируй secrets
> Никогда не записывай passwords, tokens, cookies или personal data в logs.

## Тестовая отчётность

**Test report** показывает результаты тестов в читаемом виде.

Типичные данные report:

- passed, failed, skipped tests
- причина failure
- screenshots
- logs
- execution time
- ссылки на requirements или issues

Reports помогают QA-инженерам, разработчикам и менеджерам понимать состояние качества.

---

## Allure reports

**Allure** — популярный reporting tool для Java AQA проектов.

Он может показывать:

- test suites
- steps
- attachments
- screenshots
- history и trends
- severity и tags

Пример идеи:

```java
Allure.step("Open login page");
```

**Когда использовать:** UI и API automation, где важен удобный анализ failures.

## Метрики

**Metrics** — это числа, которые помогают понимать тестовую работу, качество продукта и реальную пользу от автоматизации.

**Цель:** Понять, делает ли автоматизация обратную связь быстрее, регрессию стабильнее и поддержку дешевле.

Полезно смотреть не на одну метрику, а на несколько групп.

### Stability metrics

Эти метрики показывают, насколько набор тестов надёжен.

| Метрика | Значение |
|---|---|
| pass rate | passed tests / executed tests |
| failure rate | failed tests / executed tests |
| flaky test count | tests, которые падают без реального product bug |
| rerun rate | как часто тесты приходится перезапускать |

### Speed metrics

Эти метрики показывают, насколько быстро команда получает feedback.

| Метрика | Значение |
|---|---|
| execution time | сколько времени работает suite |
| feedback time in CI | сколько проходит времени от коммита до результата |
| time to detect failure | как быстро команда замечает проблему |

### Coverage and value metrics

Эти метрики помогают понять, покрывает ли автоматизация действительно важные проверки.

| Метрика | Значение |
|---|---|
| critical flow coverage | покрыты ли автотестами ключевые пользовательские сценарии |
| regression coverage | какая часть важной регрессии автоматизирована |
| API/UI/unit balance | нет ли перекоса, например слишком большого числа дорогих UI-тестов |

### Quality outcome metrics

Эти метрики показывают итоговый эффект автоматизации на продукт.

| Метрика | Значение |
|---|---|
| defect leakage | bugs, найденные после release |
| escaped critical defects | сколько критичных дефектов ушло в production |
| bug detection before release | сколько дефектов найдено автотестами до релиза |

### Maintenance metrics

Эти метрики показывают цену поддержки automation suite.

| Метрика | Значение |
|---|---|
| false positive rate | сколько падений не связано с реальным дефектом продукта |
| time to fix broken tests | сколько времени уходит на починку тестов после изменений |
| maintenance effort | сколько ресурсов команда тратит на поддержку suite |

**Какие метрики особенно полезны для оценки эффективности автоматизации:**
- flaky test count или false positive rate
- execution time и feedback time in CI
- regression coverage критичных сценариев
- defect leakage после релиза
- time to fix broken tests

> [!tip] Метрики, которые легко понять неправильно
> Большой `pass rate` сам по себе ещё не означает хорошую автоматизацию. Можно иметь 98% passed tests, но при этом слабое покрытие, много flaky tests или слишком долгий feedback. Метрики полезны только в контексте.

## Хорошие практики

- логируй достаточно, чтобы воспроизвести и разобрать failures
- прикладывай screenshots для UI failures
- прикладывай детали API request/response без secrets
- делай имена reports и tests читаемыми
- отслеживай flaky tests отдельно
- не используй metrics, чтобы обвинять людей

> [!tip] Правило хорошего отчёта
> Хороший report отвечает: что упало, где упало, почему могло упасть и какие есть evidence.

## Вопросы для собеседования

### Вопросы начального уровня

**1. Зачем автотестам logs?**
Logs помогают понять, что произошло во время failed run.

**2. Что такое test report?**
Test report — это читаемая сводка результатов тестирования.

### Вопросы среднего уровня

**3. Что нужно приложить к UI failure?**
Screenshot, logs и иногда page source или browser console logs.

**4. Для чего используется Allure?**
Allure создаёт подробные test reports со steps, attachments и history.

### Вопросы повышенного уровня

**5. Почему pass rate недостаточно как метрики качества?**
Pass rate может выглядеть хорошо, даже если важные области не покрыты или tests слабые.

**6. Какие метрики помогают оценить эффективность автоматизации?**
Обычно смотрят на стабильность (`flaky test count`, `false positive rate`), скорость (`execution time`, `feedback time in CI`), покрытие важной регрессии и итоговое качество (`defect leakage`). Хорошая автоматизация не только много запускается, но и даёт быстрый, стабильный и полезный feedback.

### Вопросы с кодом

**7. Что здесь опасно?**

```java
log.info("Token: " + token);
```

Код логирует secret token. Sensitive data нужно маскировать или не логировать.
