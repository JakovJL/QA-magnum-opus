## Почему эта тема важна

В Java часто нужно хранить не одно значение, а много: список логинов, набор уникальных email, карту тестовых данных или очередь задач. Для этого используются массивы и коллекции.

Эта тема важна, потому что ты будешь использовать её везде:

- В прикладном коде для хранения и обработки данных
- В AQA для локаторов, тестовых данных, API payload и результатов
- На собеседованиях, чтобы объяснить, какая структура подходит для какой задачи

Главная идея простая:

- **Массивы** — контейнеры фиксированного размера
- **Коллекции** — гибкие готовые структуры данных из Java Collections Framework

## Массивы

**Массив** хранит несколько значений одного типа в одном объекте. Его размер фиксируется при создании.

```java
int[] numbers = new int[3];
numbers[0] = 10;
numbers[1] = 20;
numbers[2] = 30;

String[] names = {"Ann", "Bob", "Kate"};
System.out.println(names[1]); // Bob
```

### Ключевые правила

- Индексация начинается с `0`
- Размер нельзя изменить после создания
- Массивы могут хранить примитивы и объекты
- У всех элементов есть значение по умолчанию

```java
int[] nums = new int[3];        // [0, 0, 0]
String[] words = new String[2]; // [null, null]
```

### Преимущества

- Очень быстрый доступ по индексу: `arr[i]`
- Простой синтаксис
- Хороший выбор, когда размер известен заранее

### Недостатки

- Фиксированный размер
- Нет готовых методов вроде `add()` или `remove()`
- Вставка в середину неудобна, потому что элементы нужно сдвигать

> [!tip] Array vs Collection
> Используй массив, когда количество элементов фиксировано и нужен простой доступ по индексу. Используй коллекцию, когда размер часто меняется или нужны готовые операции вроде `add`, `remove`, `contains` или сортировки.

## Обзор Collection Framework

**Java Collections Framework** — это набор интерфейсов и классов для хранения групп объектов.

Основные интерфейсы:

- `List` - упорядоченная коллекция, дубликаты разрешены
- `Set` - только уникальные элементы, без дубликатов
- `Queue` - элементы обрабатываются в определённом порядке
- `Map` - пары ключ-значение

> [!info] Map и интерфейс Collection
> `Map` относится к Java Collections Framework, хотя и не наследует `Collection`. Он принадлежит к тому же семейству структур данных (тот же пакет `java.util`, общая философия) и используется рядом с остальными коллекциями — но его API работает с парами ключ-значение, а не с одиночными элементами, поэтому он не вписывается в контракт `Collection`.

Популярные реализации:

- `ArrayList`
- `LinkedList`
- `HashSet`
- `HashMap`
- `PriorityQueue`

```java
List<String> list = new ArrayList<>();
Set<String> set = new HashSet<>();
Map<String, Integer> map = new HashMap<>();
Queue<String> queue = new LinkedList<>();
```

> [!warning] Коллекции хранят объекты, а не примитивы
> Коллекция может хранить `Integer`, `Double` или `Boolean`, но не `int`, `double` или `boolean`. При добавлении примитива Java автоматически упаковывает (autoboxing) его в класс-обёртку, а при чтении — распаковывает (unboxing).

### Интерфейсы, абстрактные классы и реализации

Во framework три слоя:

- **Интерфейс** — задаёт контракт (методы), без поведения. Примеры: `List`, `Set`, `Queue`, `Map`.
- **Абстрактный класс** — частичная реализация, на которой строятся конкретные классы. Примеры: `AbstractList`, `AbstractSet`, `AbstractQueue`. Напрямую используются редко.
- **Конкретная реализация** — готовый к использованию класс. Примеры: `ArrayList`, `HashSet`, `PriorityQueue`.

Программируй к интерфейсу, а не к реализации: объявляй тип как `List<String> list = new ArrayList<>();`, чтобы можно было поменять реализацию, не переписывая остальной код.

Корневой интерфейс `Collection` определяет методы, которые наследует любая коллекция (кроме `Map`):

| Метод | Что делает |
|---|---|
| `add(e)` / `addAll(c)` | Добавить один / добавить все из другой коллекции |
| `remove(o)` / `removeAll(c)` | Удалить один / удалить все найденные в `c` |
| `retainAll(c)` | Оставить только элементы, которые есть и в `c` |
| `contains(o)` / `containsAll(c)` | Проверки наличия |
| `size()` / `isEmpty()` | Количество / пустота |
| `clear()` | Удалить всё |
| `iterator()` | Возвращает `Iterator` по элементам |
| `toArray()` / `toArray(T[])` | Преобразовать в массив |
| `stream()` | Вернуть `Stream` по элементам |

## Контракты равенства и порядка

Коллекции постоянно сравнивают элементы: `List.contains()` проверяет равенство, `Set` отбрасывает дубликаты, `Map` находит значения по ключу, а отсортированные структуры упорядочивают элементы. Всё это держится на двух контрактах: `equals()`/`hashCode()` для равенства и `Comparable`/`Comparator` для порядка.

Если разобрать эти контракты заранее, каждая следующая секция будет понятнее.

---

### Контракт equals и hashCode

**Определение:** `equals()` решает, одинаковы ли два объекта логически. `hashCode()` возвращает целое число, которое хеш-коллекции (`HashMap`, `HashSet`) используют для быстрого размещения элементов по бакетам.

**Цель:** Позволить хеш-коллекциям находить объекты за O(1) в среднем.

**Контракт (обязателен):**

- Если `a.equals(b)` вернул `true`, то `a.hashCode() == b.hashCode()` тоже обязан быть `true`.
- Если `equals()` вернул `false`, хеш-коды **могут** совпадать (это коллизия) или нет.
- Если переопределил `equals()` — **обязательно** переопредели и `hashCode()`.

```java
// Сломано: equals() проверяет name, но hashCode() не переопределён.
// Два user с одинаковым name попадают в разные бакеты -> в Set появляются дубликаты.
public class User {
    String name;

    public boolean equals(Object o) {
        return o instanceof User u && this.name.equals(u.name);
    }
    // hashCode() отсутствует!
}
```

> [!danger] Главное правило
> Равные объекты **обязаны** иметь равные хеш-коды. Если нарушить это, `HashSet` будет хранить «дубликаты», а `HashMap` «терять» ключи, потому что будет искать не в том бакете.

### Comparable vs Comparator

Оба определяют порядок элементов, но из разных мест.

**`Comparable`** — естественный порядок, заданный **внутри** самого класса.

- Один порядок по умолчанию для класса
- Метод: `int compareTo(T o)`
- Пример: `String` сортируется лексикографически

**`Comparator`** — внешняя стратегия. Можно построить много разных порядков для одного класса.

- Метод: `int compare(T a, T b)`
- Используется через `Collections.sort(list, comparator)` или `list.sort(comparator)`

```java
// Comparable — внутри класса:
public class User implements Comparable<User> {
    String name;
    public int compareTo(User other) { return this.name.compareTo(other.name); }
}

// Comparator — внешний, переиспользуемый:
Comparator<User> byNameLength = Comparator.comparingInt(u -> u.name.length());
users.sort(byNameLength);
```

> [!info] Comparable vs Comparator
> `Comparable` отвечает на вопрос «как этот объект сравнивает себя с другим?» (один порядок по умолчанию). `Comparator` отвечает «как сравнить два объекта снаружи?» (возможны много разных порядков).

### Ordered vs Sorted

Эти два слова часто путают, но в JDK они значат разное:

- **Ordered** — коллекция отдаёт элементы в **предсказуемой последовательности**, но не по значению. `ArrayList` и `ArrayDeque` упорядочены по позиции добавления. `LinkedHashSet` и `LinkedHashMap` сохраняют **порядок вставки** (а `LinkedHashMap` опционально — порядок доступа).
- **Sorted** — элементы упорядочены **по значению**, через `Comparable` или `Comparator`. `TreeSet` и `TreeMap` — sorted.
- **Unordered** — без гарантированной последовательности. `HashSet` и `HashMap` — unordered.

**Natural ordering** — порядок по умолчанию, заданный `Comparable` класса (алфавитный для `String`, по возрастанию для чисел). Для любого другого порядка передают `Comparator`.

> [!info] Порядок вставки vs отсортированный порядок
> Порядок вставки = «в том порядке, в каком я добавлял». Отсортированный порядок = «упорядочены по значениям элементов». `LinkedHashMap` даёт первое, `TreeMap` — второе.

## List

`List` сохраняет порядок добавления и допускает дубликаты. Это самая повседневная коллекция: упорядоченная, индексируемая и гибкая.

---

### Интерфейс List

**Определение:** Упорядоченная коллекция с доступом по индексу, допускающая дубликаты (включая `null`).

**Цель:** Хранить последовательность элементов, где важна позиция.

Основные методы `List` (наследуются `ArrayList` и `LinkedList`):

| Метод              | Что делает                                                  |
| ------------------ | ----------------------------------------------------------- |
| `add(e)`           | Добавить в конец                                            |
| `add(i, e)`        | Вставить по индексу `i`, остальное сдвинуть вправо          |
| `get(i)`           | Прочитать элемент по индексу `i`                            |
| `set(i, e)`        | Заменить элемент по индексу `i`, возвращает старое значение |
| `remove(int i)`    | Удалить **по индексу**                                      |
| `remove(Object o)` | Удалить **по значению** (первое совпадение)                 |
| `removeIf(pred)`   | Удалить все элементы, подходящие под условие                |
| `indexOf(o)`       | Первый индекс значения, или `-1`                            |
| `contains(o)`      | `true`, если значение есть                                  |
| `size()`           | Количество элементов                                        |
| `isEmpty()`        | `true`, если размер `0`                                     |
| `clear()`          | Удалить всё                                                 |

```java
List<String> users = new ArrayList<>(List.of("ann", "bob", "kate"));
users.set(0, "alice");              // [alice, bob, kate]
users.add(1, "dave");               // [alice, dave, bob, kate]
users.removeIf(u -> u.length() > 3); // удалит "alice", "dave", "kate"
System.out.println(users.indexOf("bob")); // 0
```

> [!caution] remove(int) vs remove(Object) для List of Integers
> Для `List<Integer>` вызов `list.remove(1)` удалит элемент **по индексу 1**, а не значение `1`. Чтобы удалить по значению, используй `list.remove(Integer.valueOf(1))`.

---

### ArrayList

**Определение:** `List` на основе динамического массива, который растёт при заполнении.

**Цель:** Быстрая коллекция общего назначения.

**Как растёт внутри:** при создании через конструктор по умолчанию внутренний массив пуст. При первом `add()` он выделяется с `DEFAULT_CAPACITY = 10`. При заполнении `grow()` создаёт новый массив размером `oldCapacity + (oldCapacity >> 1)` (примерно в **1.5 раза**) и копирует элементы через `Arrays.copyOf()`. Разовый resize стоит O(n), но в пересчёте на множество вызовов каждый `add()` — это O(1).

**Преимущества:**

- Быстрый случайный доступ: `get(i)` за O(1)
- Экономит память — один непрерывный массив ссылок
- Дружественен к процессорному кешу, поэтому быстр на практике

**Недостатки:**

- Медленные вставки/удаления в середине: O(n) из-за сдвига элементов
- Resize иногда дорогой (но амортизируется)

**Когда использовать:** Выбор по умолчанию для упорядоченной коллекции почти всегда.

---

### LinkedList

**Определение:** `List` из двусвязных узлов. Каждый узел хранит значение плюс ссылку на предыдущий и следующий узел.

**Цель:** Поддерживать быстрые добавления/удаления по краям.

**Преимущества:**

- `add()`/`remove()` в начале или в конце — O(1)
- Можно использовать и как `Queue`, и как `Deque`

**Недостатки:**

- `get(i)` — O(n), нужно идти от одного конца до индекса
- Больше памяти на элемент: каждый узел — отдельный объект (value + prev + next)
- Плохая локальность для кеша

**Когда использовать:** Редко. Для очереди/стека лучше `ArrayDeque`, для списка общего назначения — `ArrayList`.

---

### ArrayList vs LinkedList

> [!tip] Быстрое сравнение
> | Аспект | ArrayList | LinkedList |
> |---|---|---|
> | Внутренняя структура | Динамический массив | Двусвязные узлы |
> | `get(i)` | O(1) | O(n) |
> | Вставка/удаление по краям | O(1)* амортизированно | O(1) |
> | Вставка/удаление в середине | O(n) | O(n) |
> | Память на элемент | Одна ссылка | Объект Node (prev + next + value) |
> | Лучше всего для | Чтения по индексу, общих задач | Частых add/remove по краям |

> [!caution] Вставка в LinkedList редко быстрее на практике
> `LinkedList` даёт O(1) вставку, только если ты уже держишь узел. Вставка по индексу всё равно стоит O(n) — нужно дойти до позиции. На реальных размерах `ArrayList` обычно выигрывает даже для вставок в середину благодаря кешу и меньшим накладным расходам.

В реальных проектах и тестах `ArrayList` — выбор по умолчанию почти всегда.

## Set

`Set` хранит только уникальные элементы. Используй его, когда дубликаты недопустимы.

---

### Интерфейс Set

**Определение:** Коллекция, которая отвергает дубликаты.

**Цель:** Гарантировать уникальность и быструю проверку наличия.

Основные методы `Set`:

| Метод | Что делает |
|---|---|
| `add(e)` | Добавить; возвращает `false`, если уже есть |
| `remove(o)` | Удалить значение |
| `contains(o)` | Проверка наличия (быстрая) |
| `addAll(c)` | Объединение с другой коллекцией |
| `retainAll(c)` | Оставить только элементы, которые есть и в `c` (пересечение) |
| `removeAll(c)` | Удалить все элементы, найденные в `c` (разность) |

```java
Set<String> a = new HashSet<>(List.of("chrome", "firefox", "edge"));
Set<String> b = new HashSet<>(List.of("firefox", "safari"));
a.retainAll(b);                       // пересечение -> [firefox]
System.out.println(a.add("firefox")); // false, уже есть
```

> [!danger] Уникальность зависит от equals() и hashCode()
> `Set` определяет дубликаты через `equals()`/`hashCode()`. Если они сломаны в кастомном классе, Set будет хранить «дубликаты». → см. [[#Контракт equals и hashCode]]

---

### HashSet

**Определение:** `Set` на основе хеш-таблицы. Внутри использует `HashMap`, где каждый элемент — ключ, а значение — один общий объект-заглушка.

**Цель:** Самый быстрый Set общего назначения.

**Преимущества:**

- `add`, `remove`, `contains` — O(1) в среднем
- Низкие накладные расходы

**Недостатки:**

- Нет гарантии порядка — порядок обхода может выглядеть случайным
- Производительность зависит от корректного `hashCode()`

**Когда использовать:** Set по умолчанию, когда нужна только уникальность.

---

### LinkedHashSet

**Определение:** `HashSet`, который дополнительно хранит связный список записей, сохраняя **порядок добавления**.

**Цель:** Уникальность **плюс** предсказуемый порядок обхода.

**Преимущества:**

- Обходит элементы в порядке добавления
- Операции всё ещё O(1) в среднем

**Недостатки:**

- Чуть больше памяти, чем у `HashSet` (на ссылки)

**Когда использовать:** Когда нужна уникальная коллекция с предсказуемым порядком (например, убрать дубликаты, сохранив порядок первого появления).

---

### TreeSet

**Определение:** `Set` на основе красно-чёрного дерева. Элементы хранятся **отсортированными**.

**Цель:** Хранить элементы в порядке и поддерживать запросы по диапазону.

**Преимущества:**

- Всегда обходит элементы в отсортированном порядке
- Поддерживает диапазонные представления: `subSet`, `headSet`, `tailSet`
- `add`/`remove`/`contains` — O(log n)

**Недостатки:**

- Медленнее `HashSet` (O(log n) против O(1))
- **Не** допускает `null`
- Уникальность определяет через `Comparable`/`Comparator`, а не через `equals()`/`hashCode()`

**Когда использовать:** Когда нужен отсортированный уникальный набор: автокомплит, диапазоны, таблица лидеров.

**Интерфейсы:** `TreeSet` реализует `SortedSet`, который расширяется до `NavigableSet`. Они добавляют навигационные методы: `first()`, `last()`, `lower(e)`, `higher(e)`, `floor(e)`, `ceiling(e)` и диапазонные представления `subSet(...)`, `headSet(...)`, `tailSet(...)`.

---

### HashSet vs LinkedHashSet vs TreeSet

> [!tip] Быстрое сравнение
> | Свойство | HashSet | LinkedHashSet | TreeSet |
> |---|---|---|---|
> | Порядок | Нет | Порядок добавления | Отсортированный |
> | `add`/`remove`/`contains` | O(1)* | O(1)* | O(log n) |
> | Допускает `null` | Да (один) | Да (один) | Нет |
> | Уникальность через | `hashCode`/`equals` | `hashCode`/`equals` | `compareTo`/`Comparator` |
> | Лучше всего для | Чистой уникальности | Уникальность + порядок | Отсортированной уникальности |

## Map

`Map` хранит данные в формате `ключ -> значение`. Это структура за конфигами, разобранным JSON, HTTP-заголовками и многими задачами с собеседований.

---

### Интерфейс Map

**Определение:** Коллекция пар ключ-значение, где каждый ключ уникален.

**Цель:** Быстро находить значение по ключу.

Основные методы `Map`:

| Метод | Что делает |
|---|---|
| `put(k, v)` | Добавить или перезаписать; возвращает старое значение |
| `get(k)` | Значение по ключу, или `null` |
| `getOrDefault(k, def)` | Значение, или `def`, если ключа нет |
| `putIfAbsent(k, v)` | Записать, только если ключа ещё нет |
| `containsKey(k)` / `containsValue(v)` | Проверки наличия |
| `remove(k)` | Удалить запись по ключу |
| `keySet()` / `values()` / `entrySet()` | Представления для итерации |
| `computeIfAbsent(k, fn)` | Лениво создать и сохранить значение, если ключа нет |
| `merge(k, v, fn)` | Объединить новое и существующее значение (удобно для подсчёта) |

```java
Map<String, Integer> count = new HashMap<>();
for (String word : List.of("a", "b", "a")) {
    count.merge(word, 1, Integer::sum); // {a=2, b=1}
}
System.out.println(count.getOrDefault("x", 0)); // 0

Map<String, List<String>> byType = new HashMap<>();
byType.computeIfAbsent("ui", k -> new ArrayList<>()).add("loginTest");
```

> [!caution] get() может вернуть null по двум причинам
> `map.get(k)` возвращает `null` и когда ключа нет, и когда ключ есть со значением `null`. Чтобы различить, используй `containsKey(k)` или `getOrDefault(k, def)`.

---

### HashMap

**Определение:** `Map` на основе хеш-таблицы — массива **бакетов**.

**Цель:** Самый быстрый Map общего назначения.

**Как работает внутри:**

1. Записи хранятся в массиве бакетов, каждый содержит `Node<K,V>` (key, value, hash, next).
2. При `put(k, v)`: вычисляется `k.hashCode()`, биты перемешиваются и мапятся на индекс бакета.
3. Если бакет пуст — кладём node. Если занят — сравниваем ключи через `equals()`: совпали → перезаписываем value; не совпали → цепляем новый node в связный список бакета (**коллизия**).
4. Начиная с Java 8, если в бакете больше **8** записей (и таблица достаточно большая), цепочка превращается в сбалансированное **красно-чёрное дерево**, улучшая поиск в худшем случае с O(n) до O(log n).
5. `get(k)` работает так же: hash → бакет → поиск по списку/дереву через `equals()`.

**Преимущества:**

- `put`, `get`, `remove`, `containsKey` — O(1) в среднем
- Широко поддерживается и хорошо изучена

**Недостатки:**

- Нет гарантии порядка
- Не потокобезопасна
- Зависит от корректного `hashCode()`

**Когда использовать:** Map по умолчанию почти во всех однопоточных сценариях.

> [!caution] Не используй изменяемые ключи
> Если поля ключа изменятся после вставки, его hash code тоже изменится — но запись останется в исходном бакете. При поиске вычислится новый hash и будет искаться в другом бакете, поэтому запись фактически потеряется. Используй неизменяемые ключи: `String`, `Integer`, `Long`.

---

### Load Factor и Capacity

**Определение:** **Capacity** — количество бакетов (по умолчанию `16`). **Load factor** (по умолчанию `0.75`) — порог заполнения: когда `size > capacity × loadFactor`, карта **увеличивается** — удваивает capacity и перехеширует все записи в новые бакеты.

**Почему 0.75:** компромисс между памятью и скоростью. Низкое значение (0.5) — меньше коллизий, но больше пустых бакетов (больше памяти). Высокое (0.9) — экономит память, но растёт число коллизий и замедляется поиск.

**Практический совет:** если финальный размер известен заранее, передай его в конструктор (`new HashMap<>(expectedSize)`), чтобы избежать повторных resize.

---

### LinkedHashMap

**Определение:** `HashMap`, который дополнительно хранит связный список записей, сохраняя **порядок добавления** (опционально — порядок доступа).

**Цель:** Быстрые поиск **плюс** предсказуемый порядок обхода.

**Преимущества:**

- Обходит записи в порядке добавления (или доступа)
- Операции всё ещё O(1) в среднем

**Недостатки:**

- Чуть больше памяти, чем у `HashMap`

**Когда использовать:** Когда важен порядок: упорядоченный конфиг или LRU-кэш (в режиме access-order).

---

### TreeMap

**Определение:** `Map` на основе красно-чёрного дерева. Ключи хранятся **отсортированными**.

**Цель:** Упорядоченные ключи и запросы по диапазону.

**Преимущества:**

- Обходит ключи в отсортированном порядке
- Диапазонные представления: `subMap`, `headMap`, `tailMap`
- O(log n) для `put`/`get`/`remove`

**Недостатки:**

- Медленнее `HashMap` (O(log n) против O(1))
- Ключи должны быть `Comparable`, либо нужно передать `Comparator`
- Не допускает `null` в качестве ключа

**Когда использовать:** Когда ключи должны оставаться отсортированными: временные ряды, упорядоченные конфиги, диапазонные запросы.

**Интерфейсы:** `TreeMap` реализует `SortedMap`, который расширяется до `NavigableMap`. Они добавляют навигационные методы по ключам: `firstKey()`, `lastKey()`, `lowerKey(k)`, `higherKey(k)`, `floorKey(k)`, `ceilingKey(k)` и диапазонные представления `subMap(...)`, `headMap(...)`, `tailMap(...)`.

---

### HashMap vs LinkedHashMap vs TreeMap

> [!tip] Быстрое сравнение
> | Свойство | HashMap | LinkedHashMap | TreeMap |
> |---|---|---|---|
> | Порядок | Нет | Добавления / доступа | Сортировка по ключу |
> | `put`/`get`/`remove` | O(1)* | O(1)* | O(log n) |
> | `null` ключ | Да (один) | Да (один) | Нет |
> | Диапазонные запросы | Нет | Нет | Да |
> | Лучше всего для | Поиска по умолчанию | Упорядоченного поиска, LRU | Отсортированных ключей |

---

### ConcurrentHashMap

**Определение:** Потокобезопасная `Map` для конкурентного доступа.

**Цель:** Безопасный и быстрый Map для многопоточного кода.

**Преимущества:**

- Потокобезопасна без блокировки всей карты — блокировка происходит на уровне бакета, поэтому чтение обычно идёт без блокировок, а запись не блокирует всю структуру
- Заметно выше пропускная способность, чем у полностью синхронизированной map
- **Не** выбрасывает `ConcurrentModificationException` при итерации

**Недостатки:**

- Не допускает `null` ни для ключей, ни для значений
- Больше накладных расходов, чем у обычной `HashMap`

**Когда использовать:** Везде, где карта разделяется между потоками.

> [!info] ConcurrentHashMap vs Collections.synchronizedMap()
> `synchronizedMap()` оборачивает **каждый** метод в одну блокировку — одновременно работает только один поток. `ConcurrentHashMap` блокирует по бакетам, допуская параллельных читателей и писателей, и не допускает `null`.

### Потокобезопасные коллекции

Стандартные коллекции (`ArrayList`, `HashMap`, `HashSet`) **не** потокобезопасны. Для конкурентного доступа используй специальные классы из `java.util.concurrent`:

- `ConcurrentHashMap` — потокобезопасная map, блокировка по бакету
- `CopyOnWriteArrayList` / `CopyOnWriteArraySet` — каждая запись копирует весь массив; хороши для часто читаемых и редко меняемых списков
- `ConcurrentLinkedQueue` — неблокирующая, lock-free FIFO-очередь
- `BlockingQueue` (например, `LinkedBlockingQueue`, `ArrayBlockingQueue`) — блокирует вызывающий поток, когда пуста или заполнена; используется в паттерне producer-consumer

**Legacy-классы с синхронизацией:** `Vector`, `Hashtable` и `Stack` (наследует `Vector`) синхронизируют каждый метод на одной блокировке. Они появились до `java.util.concurrent` и считаются устаревшими — грубая (coarse-grained) блокировка делает их медленными при contention. Предпочитай `ArrayList`/`HashMap` плюс concurrent-класс.

**Обёртка как альтернатива:** `Collections.synchronizedList/set/map()` оборачивает любую коллекцию в потокобезопасное представление, но блокирует всю коллекцию на каждой операции, а составные действия всё равно требуют внешней синхронизации. Dedicated concurrent-коллекции обычно лучше.

---

### WeakHashMap

**Определение:** `Map`, ключи в которой удерживаются через **слабые ссылки** (weak references).

**Цель:** Позволить записям исчезать автоматически, когда на ключ больше нет сильных ссылок снаружи.

**Как работает:** если на ключ нет сильных ссылок вне карты, GC может его собрать, и запись удалится из карты при следующей чистке.

**Когда использовать:** Кэши, которые должны истекать, когда ключ перестаёт использоваться, или реестры слушателей с самоочисткой.

---

### EnumMap и EnumSet

**Определение:** Специализированные очень быстрые коллекции, ключи/элементы которых — один `enum`.

**Как работают:** `EnumMap` хранит значения в массиве, проиндексированном `ordinal()` enum — без хеширования. `EnumSet` использует битовый вектор (`long[]`).

**Преимущества:**

- O(1) операции и минимальная память
- Нет хеш-коллизий

**Когда использовать:** Везде, где ключ/элемент — это `enum`.

```java
enum Status { ACTIVE, INACTIVE, BANNED }
Map<Status, List<User>> usersByStatus = new EnumMap<>(Status.class);
Set<Status> activeStatuses = EnumSet.of(Status.ACTIVE, Status.INACTIVE);
```

## Queue и Deque

`Queue` упорядочивает элементы для обработки (обычно FIFO). `Deque` (двусторонняя очередь) поддерживает добавление/удаление с обоих концов и может работать и как очередь, и как стек.

---

### Интерфейс Queue

**Определение:** Коллекция, обрабатывающая элементы в определённом порядке, обычно first-in-first-out (FIFO).

**Цель:** Обрабатывать элементы по одному в порядке очереди.

Основные методы `Queue`:

| Метод | Что делает |
|---|---|
| `offer(e)` / `add(e)` | Добавить в хвост |
| `poll()` | Удалить и вернуть голову, или `null`, если пусто |
| `peek()` | Посмотреть голову без удаления, или `null` |

```java
Queue<String> tasks = new ArrayDeque<>();
tasks.offer("login");
tasks.offer("checkout");
System.out.println(tasks.peek()); // login (не удалён)
System.out.println(tasks.poll()); // login (удалён)
```

> [!info] offer/poll vs add/remove
> `offer`/`poll`/`peek` при неудаче возвращают специальное значение (`false`, `null`). `add`/`remove`/`element` в этом случае бросают исключение. Предпочитай возвращающие версии.

---

### PriorityQueue

**Определение:** Очередь, в которой голова всегда **наименьший** элемент согласно порядку (по умолчанию min-heap).

**Цель:** Всегда извлекать элемент с наивысшим приоритетом.

**Как работает:** на основе бинарной кучи — дерева, где каждый родитель меньше своих потомков. `offer()`/`add()` вставляют и всплывают (O(log n)); `poll()` забирает корень и просеивает кучу (O(log n)); `peek()` читает корень (O(1)).

**Преимущества:**

- O(log n) вставка и удаление
- Всегда отдаёт следующий приоритетный элемент за O(1)

**Недостатки:**

- Не упорядочивает всю очередь — гарантирован только корень (минимум)
- Порядок итератора не отсортирован

**Когда использовать:** Планировщики задач, задачи «top K», алгоритмы вроде Дейкстры.

---

### Интерфейс Deque

**Определение:** Двусторонняя очередь — поддерживает вставку и удаление с обоих концов.

**Цель:** Одна структура, пригодная и как очередь (FIFO — First-In-First-Out), и как стек (LIFO — Last-In-First-Out).

Основные методы `Deque`: `addFirst`/`addLast`, `removeFirst`/`removeLast`, `peekFirst`/`peekLast`, плюс `push`/`pop` для работы как стек.

---

### ArrayDeque

**Определение:** `Deque` на основе расширяемого **кольцевого массива** (circular array).

**Цель:** Быстрая очередь и стек общего назначения.

**Преимущества:**

- O(1) амортизированно для операций с обоих концов
- Нет node-объектов на каждый элемент — меньше памяти, чем у `LinkedList`
- Быстрее `LinkedList` для задач очереди/стека

**Недостатки:**

- Медленные вставки/удаления в середине (для этого не предназначен)

**Когда использовать:** Везде, где нужен стек или очередь.

> [!danger] Не используй устаревший класс Stack
> `Stack` наследует от `Vector` (синхронный, медленный) и нарушает LSP — он «является» `Vector`, поэтому в стек протекают методы `List` вроде `get(index)`. Используй `ArrayDeque` с `push()`/`pop()` для LIFO или `offer()`/`poll()` для FIFO.

```java
// Стек (LIFO):
Deque<String> stack = new ArrayDeque<>();
stack.push("a"); stack.pop();

// Очередь (FIFO):
Queue<String> queue = new ArrayDeque<>();
queue.offer("a"); queue.poll();
```

> [!info] ArrayDeque vs LinkedList как Deque
> `ArrayDeque` практически всегда предпочтительнее: он быстрее и экономичнее по памяти (кольцевой массив против node-объекта на элемент). `LinkedList` стоит брать, только если нужен list-подобный доступ по индексу в середине, а этого `Deque` не даёт.

---

## Immutable и Unmodifiable коллекции

В Java есть несколько способов получить коллекцию, которую нельзя изменить. Они неодинаковы.

**1. `List.of(...)` / `Set.of(...)` / `Map.of(...)` (Java 9+)**

Возвращают **полностью immutable** коллекции. Любой `add`, `remove` или `set` бросает `UnsupportedOperationException`. Также они не допускают `null`.

**2. `Collections.unmodifiableList(list)` (и аналоги)**

Оборачивают **существующую** коллекцию в представление только для чтения. Оригинал остаётся изменяемым, и его изменения **видны** через представление.

**3. `List.copyOf(list)` / `Set.copyOf()` / `Map.copyOf()` (Java 10+)**

Создают **immutable копию** — настоящий снапшот, который больше не отражает источник.

```java
List<String> mutable = new ArrayList<>();
mutable.add("a");
List<String> unmod = Collections.unmodifiableList(mutable);
mutable.add("b");          // unmod тоже увидит "b"!

List<String> copy = List.copyOf(mutable); // снапшот, больше не меняется
```

> [!tip] Arrays.asList() vs List.of() vs new ArrayList()
> `new ArrayList<>()` полностью изменяем. `Arrays.asList()` — список **фиксированного размера**, **подкреплённый** массивом: `set()` работает, но `add`/`remove` бросают `UnsupportedOperationException`, а изменения в массиве отражаются на списке. `List.of()` полностью **immutable** и не допускает `null`.

## Утилитные классы

Классы `Arrays` и `Collections` содержат статические вспомогательные методы. Они работают с массивами и коллекциями снаружи — это не методы-экземпляры.

---

### Arrays Utility

Вспомогательные методы для массивов.

| Вызов | Что делает |
|---|---|
| `Arrays.sort(arr)` | Отсортировать массив на месте |
| `Arrays.sort(arr, from, to)` | Отсортировать диапазон `from..to-1` |
| `Arrays.sort(arr, comparator)` | Сортировка с пользовательским `Comparator` |
| `Arrays.toString(arr)` | Читаемое строковое представление |
| `Arrays.deepToString(arr)` | Строковое представление для многомерных массивов |
| `Arrays.equals(a, b)` | Поэлементная проверка на равенство |
| `Arrays.deepEquals(a, b)` | Равенство для вложенных массивов |
| `Arrays.fill(arr, val)` | Заполнить весь массив значением |
| `Arrays.fill(arr, from, to, val)` | Заполнить диапазон |
| `Arrays.copyOf(arr, newLength)` | Копирование с обрезкой или добиванием |
| `Arrays.copyOfRange(arr, from, to)` | Копирование диапазона |

| `Arrays.stream(arr)` | Преобразование в `Stream` |
| `Arrays.binarySearch(arr, key)` | Бинарный поиск, возвращает индекс или отрицательное число |
| `Arrays.mismatch(a, b)` | Первый индекс, где массивы различаются, или `-1` |

```java
int[] nums = {3, 1, 2};
```

### Collections Utility

Вспомогательные методы для коллекций.

| Вызов                                                   | Что делает                                |
| ------------------------------------------------------- | ----------------------------------------- |
| `Collections.sort(list)`                                | Отсортировать список на месте             |
| `Collections.reverse(list)`                             | Обратный порядок                          |
| `Collections.shuffle(list)`                             | Случайное перемешивание                   |
| `Collections.max(list)` / `min(list)`                   | Наибольший / наименьший элемент           |
| `Collections.max(list, comp)` / `min(list, comp)`       | С пользовательским компаратором           |
| `Collections.frequency(list, o)`                        | Подсчёт вхождений                         |
| `Collections.addAll(c, elements...)`                    | Добавить несколько элементов за раз       |
| `Collections.disjoint(a, b)`                            | `true`, если нет общих элементов          |
| `Collections.rotate(list, k)`                           | Циклический сдвиг на `k` позиций          |
| `Collections.swap(list, i, j)`                          | Поменять местами два элемента по индексу  |
| `Collections.binarySearch(list, key)`                   | Бинарный поиск по отсортированному списку |
| `Collections.binarySearch(list, key, comp)`             | С пользовательским компаратором           |

| `Collections.synchronizedList(list)`                    | Потокобезопасная обёртка                  |

| `Collections.fill(list, o)`                             | Заменить все элементы на `o`              |
| `Collections.reverseOrder()`                            | Компаратор для обратной сортировки        |

```java
List<Integer> scores = new ArrayList<>(List.of(30, 10, 20));
Collections.sort(scores);
Collections.reverse(scores);
System.out.println(scores); // [30, 20, 10]
```

> [!info] Collection vs Collections
> `Collection` — корневой **интерфейс** иерархии (`List`, `Set`, `Queue`). `Collections` (множественное число) — **утилитный класс** со статическими методами: `sort`, `reverse`, `max`, `unmodifiableList`.

## Нотация Big O

Временная сложность основных операций. `*` = амортизированный средний случай.

### Реализации List

| Операция | ArrayList | LinkedList |
|---|---|---|
| `add(e)` (конец) | O(1)* | O(1) |
| `add(i, e)` (середина) | O(n) | O(n) |
| `get(i)` | O(1) | O(n) |
| `remove(i)` | O(n) | O(n) |
| `contains(o)` | O(n) | O(n) |
| `size()` | O(1) | O(1) |

> [!caution] Сложность add в ArrayList
> `ArrayList.add(e)` работает за O(1) в среднем, потому что большинство добавлений идут в конец внутреннего массива. Когда массив заполняется, он увеличивается (выделение нового массива + копирование), что стоит O(n) — но в пересчёте на множество вызовов это O(1). `LinkedList.add(e)` всегда O(1), потому что просто привязывается новый узел.

### Реализации Set

| Операция | HashSet | TreeSet | LinkedHashSet |
|---|---|---|---|
| `add(o)` | O(1)* | O(log n) | O(1)* |
| `remove(o)` | O(1)* | O(log n) | O(1)* |
| `contains(o)` | O(1)* | O(log n) | O(1)* |
| `size()` | O(1) | O(1) | O(1) |

> [!warning] Худший случай для HashSet
> `HashSet` работает за O(1) в среднем. В худшем случае (много коллизий хешей) Java 8+ преобразует цепочки бакетов в сбалансированные деревья на глубине 8, давая O(log n). Сломанный `hashCode()`, возвращающий одно значение для всех элементов, ухудшает до O(log n), а не до O(n).

### Реализации Map

| Операция | HashMap | TreeMap | LinkedHashMap |
|---|---|---|---|
| `put(k, v)` | O(1)* | O(log n) | O(1)* |
| `get(k)` | O(1)* | O(log n) | O(1)* |
| `remove(k)` | O(1)* | O(log n) | O(1)* |
| `containsKey(k)` | O(1)* | O(log n) | O(1)* |
| `containsValue(v)` | O(n) | O(n) | O(n) |
| `size()` | O(1) | O(1) | O(1) |

> [!tip] Когда что выбирать
> - `HashMap` — выбор по умолчанию, самые быстрые операции в среднем, нет гарантии порядка
> - `LinkedHashMap` — когда нужен порядок вставки или порядка доступа (LRU-кэш)
> - `TreeMap` — когда нужны отсортированные ключи или запросы диапазона (`subMap`, `headMap`, `tailMap`)

### Deque (ArrayDeque / LinkedList как Deque)

| Операция | ArrayDeque | LinkedList |
|---|---|---|
| `addFirst(e)` / `addLast(e)` | O(1)* | O(1) |
| `removeFirst()` / `removeLast()` | O(1)* | O(1) |
| `peekFirst()` / `peekLast()` | O(1) | O(1) |
| `size()` | O(1) | O(1) |

## Итерация и основные операции

По массивам и коллекциям очень часто нужно проходить в цикле.

### For Loop

Подходит, когда нужен индекс.

```java
String[] names = {"Ann", "Bob", "Kate"};
for (int i = 0; i < names.length; i++) {
    System.out.println(i + ": " + names[i]);
}
```

### Enhanced for

Подходит, когда нужны только значения.

```java
List<String> cities = List.of("Minsk", "Warsaw", "Berlin");
for (String city : cities) {
    System.out.println(city);
}
```

### Iterator и ListIterator

**`Iterator`** проходит коллекцию по одному элементу за раз. Любая `Collection` предоставляет метод `iterator()`.

| Метод | Что делает |
|---|---|
| `hasNext()` | `true`, если есть ещё элементы |
| `next()` | Возвращает следующий элемент |
| `remove()` | Удаляет последний элемент, возвращённый `next()` (опционально) |

```java
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    String s = it.next();
    if (s.isEmpty()) it.remove(); // безопасное удаление во время итерации
}
```

**`ListIterator`** расширяет `Iterator` и доступен только у `List`. Он ходит в обе стороны и может менять список:

| Метод | Что делает |
|---|---|
| `hasPrevious()` / `previous()` | Движение назад |
| `nextIndex()` / `previousIndex()` | Текущий индекс |
| `add(e)` / `set(e)` | Вставить / заменить последний возвращённый элемент |

> [!info] Iterator vs ListIterator
> `Iterator` ходит только вперёд и работает с любой `Collection`. `ListIterator` ходит в обе стороны, поддерживает `add`/`set` и доступен только у `List`.

### Реалистичный пример

Соберём результаты тестов и обработаем их.

```java
List<String> results = new ArrayList<>(List.of("pass", "fail", "pass", "skip", "fail"));

results.set(0, "PASS");                       // замена по индексу
results.removeIf(r -> r.equals("skip"));      // безопасное удаление по условию

System.out.println(results.indexOf("fail"));  // 1
System.out.println(results.contains("PASS")); // true

Map<String, Integer> stats = new HashMap<>();
for (String r : results) {
    stats.merge(r, 1, Integer::sum);
}
System.out.println(stats); // {PASS=1, fail=2, pass=1}
```

> [!danger] ConcurrentModificationException
> Не удаляй элементы из коллекции внутри enhanced `for` через `list.remove()`. Итератор хранит `modCount` и проверяет его на каждом `next()` — если коллекция структурно изменена, будет брошен `ConcurrentModificationException` (fail-fast). Используй `Iterator.remove()`, `removeIf()` или собери элементы для удаления в новый список.

```java
// Правильно: removeIf
list.removeIf(s -> s.length() < 3);

// Правильно: Iterator
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().length() < 3) it.remove();
}
```

## Массивы и коллекции в AQA

В автотестировании массивы и коллекции встречаются постоянно.

### Типичные примеры

```java
List<String> browsers = List.of("chrome", "firefox", "edge");
Map<String, String> headers = Map.of(
    "Content-Type", "application/json",
    "Authorization", "Bearer token"
);
Set<String> uniqueUserIds = new HashSet<>();
```

### Где они используются

- `List<WebElement>` из Selenium: несколько элементов на странице
- `Map<String, Object>` для API payload или разобранного JSON
- `Set<String>` для проверки уникальности ID, ссылок или значений
- Массивы в параметрах методов, например `String[] args`

### Частые ошибки

- Использовать массив, когда размер часто меняется
- Использовать `HashMap` и ждать стабильного порядка
- Забывать, что `Set` убирает дубликаты
- Обращаться по неверному индексу и получать `ArrayIndexOutOfBoundsException`
- Вызывать `get()` у пустого списка

> [!info] Хорошее правило для AQA
> Если нужна упорядоченная гибкая группа элементов, начинай с `ArrayList`. Если нужна уникальность, используй `HashSet`. Если нужны пары ключ-значение, используй `HashMap`.

> [!tip] Выбор коллекции под задачу
> | Задача | Лучший выбор |
> |---|---|
> | Упорядоченный гибкий список | `ArrayList` |
> | Уникальные элементы | `HashSet` |
> | Уникальные элементы, сохранить порядок вставки | `LinkedHashSet` |
> | Поиск значения по ключу | `HashMap` |
> | Ключ-значение, порядок вставки / LRU | `LinkedHashMap` |
> | Отсортированные уникальные элементы | `TreeSet` |
> | Отсортированные ключи | `TreeMap` |
> | Стек (LIFO) | `ArrayDeque` |
> | Очередь (FIFO) | `ArrayDeque` |
> | Сначала элемент с наивысшим приоритетом | `PriorityQueue` |
> | Потокобезопасная map | `ConcurrentHashMap` |

## Вопросы для собеседования

### Вопросы начального уровня

**1. В чём разница между массивом и коллекцией?**
Массив имеет фиксированный размер и может хранить примитивы или объекты. Коллекция имеет гибкий размер и предоставляет готовые методы вроде `add()`, `remove()` и `contains()`. Коллекции хранят объекты, а не примитивы.

**2. В чём разница между `List` и `Set`?**
`List` сохраняет порядок и допускает дубликаты. `Set` хранит только уникальные элементы и обычно не гарантирует порядок, если не использовать специальную реализацию.

**3. В чём разница между `ArrayList` и `LinkedList`?**
`ArrayList` лучше подходит для чтения по индексу и обычно быстрее в большинстве задач. `LinkedList` лучше подходит для частых вставок и удалений по краям, но хуже для случайного доступа.

**4. Может ли коллекция хранить примитивные типы?**
Нет. Коллекции работают с объектами. Используй классы-обёртки: `Integer`, `Double`, `Boolean`.

**5. В чём разница между `HashMap` и `HashSet`?**
`HashMap` хранит пары ключ-значение. `HashSet` хранит только уникальные значения. Внутри `HashSet` построен на основе `HashMap` (каждый элемент — ключ с общим объектом-заглушкой в качестве значения).

**6. Что будет, если обратиться к неверному индексу массива?**
Java выбросит `ArrayIndexOutOfBoundsException`.

**7. В чём разница между `size()` и `length`?**
У массивов используется поле `length`. У коллекций используется метод `size()`.

**8. Сохраняет ли `HashMap` порядок добавления?**
Нет. Если нужен порядок добавления, используй `LinkedHashMap`.

**9. Может ли `ArrayList` содержать дубликаты и `null`?**
Да. `ArrayList` допускает дубликаты и может содержать `null`.

**10. Когда использовать `Map`?**
Когда нужно находить значение по ключу, например поля пользователя, параметры конфига, headers или тестовые данные.

**11. Что такое Java Collections Framework?**
Единая архитектура интерфейсов и классов для хранения и обработки групп объектов. Предоставляет готовые структуры данных (`List`, `Set`, `Queue`, `Map`), чтобы не реализовывать их самому.

**12. Какие основные интерфейсы входят в Collections Framework?**
`List` (упорядоченная, дубликаты), `Set` (уникальные элементы), `Queue` (порядок обработки) и `Map` (пары ключ-значение). `List`, `Set` и `Queue` наследуют `Collection`, а `Map` — нет.

**13. Входит ли `Map` в интерфейс `Collection`?**
Нет. `Map` — часть Java Collections Framework, но он не наследует `Collection`, потому что хранит пары ключ-значение, а не одиночные элементы.

**14. В чём разница между `Collection` и `Collections`?**
`Collection` — корневой интерфейс иерархии. `Collections` (множественное число) — утилитный класс со статическими методами вроде `sort()`, `reverse()` и `max()`.

**15. Как объявить и инициализировать массив?**
Несколькими способами: `int[] a = new int[5];`, `int[] a = {1, 2, 3};` или `int[] a = new int[]{1, 2, 3};`. Размер фиксируется при создании.

**16. Какие значения по умолчанию у элементов массива?**
`0` для числовых примитивов, `false` для `boolean`, `'\u0000'` для `char` и `null` для массивов объектов.

**17. В чём разница между `length` и `length()`?**
`length` — поле массива; `length()` — метод `String`. У коллекций используется метод `size()`.

**18. Может ли коллекция содержать `null`?**
Большинство может — `ArrayList`, `HashMap` и `HashSet` допускают `null`. Исключения: `TreeSet`, `TreeMap`, `ConcurrentHashMap` и `PriorityQueue` не допускают `null`.

**19. Может ли `HashMap` иметь `null` в качестве ключа?**
Да, ровно один. `Hashtable` и `ConcurrentHashMap` не допускают ни `null`-ключей, ни `null`-значений.

**20. Как преобразовать массив в List?**
Используй `Arrays.asList(arr)` (фиксированного размера, подкреплённый массивом) или `new ArrayList<>(Arrays.asList(arr))` (полностью изменяемый). Для по-настоящему immutable списка — `List.of(...)`.

**21. Как преобразовать List в массив?**
Используй `list.toArray(new String[0])` (типизированный) или `list.toArray()` (возвращает `Object[]`).

**22. Как отсортировать List?**
Используй `Collections.sort(list)` или `list.sort(comparator)`. Элементы должны быть `Comparable`, либо нужно передать `Comparator`.

**23. В чём разница между `Comparable` и `Comparator`?**
`Comparable` задаёт естественный порядок внутри класса (`compareTo`). `Comparator` — внешняя стратегия, которую передают в `sort()` для кастомного порядка.

**24. Какая начальная capacity по умолчанию у `ArrayList`? У `HashMap`?**
`ArrayList` стартует пустым и выделяет 10 при первом `add`. `HashMap` по умолчанию 16 бакетов с load factor 0.75.

**25. Как растёт `ArrayList`?**
Когда внутренний массив заполняется, создаётся новый массив примерно в 1.5 раза больше и элементы копируются. Разовый resize стоит O(n), амортизированно O(1) на `add`.

**26. В чём разница между `HashSet` и `TreeSet`?**
`HashSet` — без порядка, O(1) в среднем, использует `hashCode`/`equals`. `TreeSet` — хранит элементы отсортированными, O(log n), использует `Comparable`/`Comparator` и не допускает `null`.

**27. Что такое `PriorityQueue`?**
Очередь, в которой голова всегда наименьший элемент (min-heap по умолчанию). `offer` и `poll` — O(log n), `peek` — O(1).

**28. Почему стоит избегать класса `Stack`?**
Он наследует `Vector` (синхронный и медленный) и протекает методами `List` в стек. Используй `ArrayDeque` с `push()`/`pop()`.

**29. В чём разница между `ArrayDeque` и `LinkedList`?**
`ArrayDeque` — кольцевой массив: быстрее и экономичнее, идеален для стека/очереди. `LinkedList` использует node на элемент и расходует больше памяти; предпочитай его только для list-подобного доступа по индексу.

**30. Как создать immutable коллекцию?**
Используй `List.of(...)`, `Set.of(...)` или `Map.of(...)` (Java 9+, полностью immutable), `Collections.unmodifiableList(list)` (представление существующей коллекции) или `List.copyOf(list)` (immutable снапшот).

**31. Зачем нужны `equals()` и `hashCode()`?**
Коллекции вроде `HashSet` и `HashMap` используют `hashCode()`, чтобы быстро найти бакет, а `equals()` — чтобы проверить, совпадает ли элемент внутри бакета. Если не переопределить их, два одинаковых по смыслу объекта будут считаться разными.

**32. Что такое `Iterator` и зачем он нужен?**
`Iterator` проходит коллекцию по одному элементу и доступен у любой `Collection`. Его главное преимущество — можно безопасно удалять элементы во время обхода через `iterator.remove()`.

**33. Что такое autoboxing и unboxing?**
Autoboxing — автоматическое преобразование примитива в класс-обёртку (`int` → `Integer`). Unboxing — обратное. Коллекции хранят только объекты, поэтому при добавлении примитив упаковывается автоматически: `list.add(5)` превращает `5` в `Integer.valueOf(5)`.

**34. Чем `TreeMap` отличается от `HashMap`?**
`HashMap` не гарантирует порядок и работает за O(1) в среднем. `TreeMap` хранит ключи отсортированными через `Comparable`/`Comparator`, работает за O(log n) и не допускает `null`-ключи.

**35. Как отсортировать массив и вывести его в консоль?**
`Arrays.sort(arr)` сортирует массив на месте. `Arrays.toString(arr)` возвращает читаемое строковое представление: `[1, 2, 3]`. Для многомерных массивов — `Arrays.deepToString(arr)`.

**36. В чём разница между `Queue` и `Deque`?**
`Queue` — обычная очередь: добавляешь в конец, забираешь с начала (FIFO). `Deque` — двусторонняя: можно добавлять и забирать с обоих концов. `Deque` может работать и как очередь, и как стек.

**37. Зачем нужны дженерики в коллекциях?**
Дженерики (`List<String>`, `Map<Integer, String>`) указывают тип элементов коллекции. Компилятор проверяет типы на этапе компиляции, и при чтении не нужен каст.

**38. Что возвращает `subList()` — копию или представление?**
`subList(from, to)` возвращает **представление** (view) оригинального списка, а не копию. Изменения в subList (включая `clear()`) отражаются на оригинальном списке.

### Вопросы среднего уровня

**1. Является ли `Map` подтипом `Collection`?**
Нет. `Map` относится к Java Collections Framework, но не реализует `Collection`.

**2. Почему `ArrayList` чаще лучше `LinkedList` на практике?**
Потому что многие реальные задачи требуют быстрого чтения, а процессорный кеш хорошо работает с непрерывными массивами. У `LinkedList` больше накладных расходов на объекты и медленнее доступ по индексу.

**3. Может ли `Set` содержать дубликаты, если `equals()` и `hashCode()` реализованы неправильно?**
Да. Если у кастомного класса сломаны `equals()` и `hashCode()`, хеш-ориентированный set может хранить дубликаты.

**4. В чём разница между `remove(int index)` и `remove(Object o)` в `List`?**
Один вариант удаляет по позиции, другой — по значению. С `List<Integer>` это путают: `remove(1)` удаляет элемент по индексу `1`, а не значение `1`.

**5. Почему `Arrays.asList()` может быть опасным?**
Он возвращает список фиксированного размера, подкреплённый массивом. Элементы менять можно через `set()`, но добавлять или удалять нельзя — будет `UnsupportedOperationException`.

**6. В чём разница между `peek()` и `poll()` у `Queue`?**
`peek()` возвращает голову без удаления; `poll()` удаляет и возвращает голову. Оба возвращают `null`, если очередь пуста.

**7. Что делает `retainAll()`?**
Оставляет только те элементы, которые есть и в переданной коллекции, — то есть выполняет пересечение (in-place).

**8. Можно ли отсортировать `HashMap` напрямую?**
Нет — `HashMap` неупорядочена. Для работы с отсортированными данными используй `TreeMap` (отсортированные ключи) или скопируй `entrySet()` в `List` и отсортируй.

**9. Почему `TreeSet` не допускает `null`?**
Потому что он упорядочивает элементы через `compareTo()` или `Comparator`, а сравнение с `null` бросает `NullPointerException`.

**10. Если обернуть список в `Collections.unmodifiableList()`, может ли исходный список всё ещё меняться?**
Да. Обёртка — это представление только для чтения; исходный список остаётся изменяемым, и любые его изменения видны через обёртку.

**11. В чём разница между `offer()` и `add()` у `Queue`?**
`offer()` возвращает `false`, если элемент нельзя добавить; `add()` в этом случае бросает исключение.

**12. Когда стоит выбрать `LinkedHashSet` вместо `HashSet`?**
Когда нужна уникальность **и** предсказуемый порядок обхода: `LinkedHashSet` сохраняет порядок вставки, а `HashSet` не гарантирует порядок.

**13. Обходит ли `PriorityQueue` элементы в отсортированном порядке?**
Нет. Только голова гарантированно наименьший элемент; остальное при итерации не упорядочено полностью.

**14. Что делает `computeIfAbsent()`?**
Возвращает значение по ключу, если он есть; иначе вычисляет значение через переданную функцию, сохраняет его и возвращает.

**15. В чём разница между `iterator()` и `listIterator()`?**
`iterator()` ходит только вперёд и работает с любой `Collection`; `listIterator()` работает только с `List`, ходит в обе стороны и поддерживает `add()`/`set()`.

**16. В чём разница между ordered и sorted коллекцией?**
Ordered — элементы идут в предсказуемой последовательности (например, порядок вставки); sorted — упорядочены по значению через `Comparable`/`Comparator`.

**17. В чём разница между `contains()` и `containsAll()`?**
`contains(o)` проверяет один элемент; `containsAll(c)` проверяет, присутствует ли каждый элемент коллекции `c`.

**18. Может ли `HashMap` хранить несколько `null`-значений?**
Да — значений `null` может быть сколько угодно, но `null`-ключ максимум один.

**19. В чём разница между `put()` и `putIfAbsent()`?**
`put()` всегда перезаписывает значение; `putIfAbsent()` записывает значение, только если ключа ещё нет.

**20. Почему `TreeMap` не нуждается в `hashCode()` и `equals()` для упорядочивания?**
Потому что он упорядочивает и идентифицирует ключи через `compareTo()` (или `Comparator`), а не через хеш-контракт.

### Вопросы повышенного уровня

**1. Как работает `HashMap` внутри?**
Записи хранятся в массиве **бакетов**, каждый содержит `Node<K,V>` (key, value, hash, next). При `put` `hashCode()` ключа мапится на бакет. Если бакет пуст — кладём node; если занят — сравниваем ключи через `equals()`: совпали → перезаписываем value, не совпали → цепляем новый node в связный список (коллизия). Начиная с Java 8 бакет с более чем 8 записями превращается в сбалансированное красно-чёрное дерево, улучшая поиск в худшем случае с O(n) до O(log n). `get` работает так же: hash → бакет → поиск через `equals()`.

**2. Что такое capacity и load factor в `HashMap`?**
Capacity — количество бакетов (по умолчанию 16). Load factor (по умолчанию 0.75) — порог заполнения: когда `size > capacity × loadFactor`, карта удваивает capacity и перехеширует все записи. Указание начальной capacity избегает повторных resize для больших карт.

**3. Почему `equals()` и `hashCode()` обязаны соглашаться?**
Хеш-коллекции находят бакет через `hashCode()`, а потом сравнивают через `equals()`. Если два «равных» объекта возвращают разные хеш-коды, они попадают в разные бакеты, и коллекция считает их разными — появляются дубликаты или неудачный поиск. Контракт: равные объекты обязаны иметь равные хеш-коды.

**4. Как `HashSet` хранит элементы?**
Он подкреплён `HashMap`: каждый элемент хранится как **ключ**, а все записи используют один общий объект-заглушку как **значение**. Поэтому поведение `HashSet` (хеширование, load factor, отсутствие порядка) напрямую следует из `HashMap`.

**5. Что будет, если hash code ключа в `HashMap` изменится после вставки?**
Запись останется в исходном бакете, но при поиске вычислится новый hash и будет искаться в другом бакете — найти запись больше не получится. Не используй изменяемые объекты как ключи; бери неизменяемые вроде `String` или `Integer`.

**6. `HashMap` vs `LinkedHashMap` vs `TreeMap`?**
`HashMap` — нет порядка, O(1) в среднем. `LinkedHashMap` — сохраняет порядок добавления (или доступа) через связный список, O(1) в среднем. `TreeMap` — ключи отсортированы, O(log n), на основе красно-чёрного дерева.

**7. `HashSet` vs `TreeSet`?**
`HashSet` — O(1) add/remove/contains, без порядка, требует корректного `hashCode()`. `TreeSet` — O(log n), хранит элементы отсортированными (через `Comparable`/`Comparator`), не допускает `null`, для уникальности игнорирует `hashCode`/`equals`. Бери `HashSet` для чистой уникальности и скорости, `TreeSet` — когда нужен отсортированный набор или диапазонные запросы.

**8. `HashMap` vs `ConcurrentHashMap`?**
`HashMap` не синхронизирована и не thread-safe. `ConcurrentHashMap` потокобезопасна: блокирует на уровне бакета (а не всей карты), допускает конкурентное чтение без блокировок и не бросает `ConcurrentModificationException` при итерации. Не допускает `null` ни для ключей, ни для значений.

**9. Почему не стоит использовать `Stack`? Что вместо него?**
`Stack` — устаревший класс, наследующий от `Vector` (синхронный, но медленный) и нарушающий LSP — он «является» `Vector`, поэтому позволяет вызывать методы `List` вроде `get(index)`. Используй `ArrayDeque` с `push()`/`pop()`: быстрее, легче и специально спроектирован для LIFO.

**10. `ArrayDeque` vs `LinkedList` как `Deque`?**
`ArrayDeque` — кольцевой массив: быстрее и экономичнее, без node-объектов на элемент. `LinkedList` использует node на элемент (prev + next + value), поэтому расходует в 2–3 раза больше памяти. Предпочитай `ArrayDeque`, если только не нужен индексный доступ в середине, которого `Deque` не даёт.

**11. Как работает `PriorityQueue`?**
Основана на min-куче (бинарное дерево, где каждый родитель меньше потомков). `offer()`/`add()` вставляют и всплывают (O(log n)); `poll()` забирает корень и просеивает (O(log n)); `peek()` читает корень (O(1)). Гарантирован только корень (минимум) — остальное не упорядочено. Используется в планировщиках, задачах «top K», алгоритме Дейкстры.

**12. В чём разница между `Comparable` и `Comparator`?**
`Comparable` задаёт естественный порядок **внутри** класса (`compareTo(T o)`) — один порядок по умолчанию. `Comparator` — внешняя стратегия (`compare(a, b)`): можно сделать несколько разных для одного класса и передавать в `sort()` или `TreeSet`.

**13. Что такое fail-fast итератор и как возникает `ConcurrentModificationException`?**
Большинство коллекций (`ArrayList`, `HashMap`) хранят `modCount`. Их итераторы снимают его и проверяют на каждом `next()`. Если коллекция структурно изменена во время итерации (например, `list.remove()` внутри for-each), `modCount` не совпадает, и итератор бросает `ConcurrentModificationException`. Используй `Iterator.remove()` или `removeIf()`. Это обнаружение best-effort, а не гарантия.

**14. Fail-fast vs fail-safe итераторы?**
Fail-fast (стандартные коллекции) бросают исключение при конкурентной модификации. Fail-safe итераторы (`CopyOnWriteArrayList`, `ConcurrentHashMap`) обходят снапшот или tolerate изменения, поэтому никогда не бросают — но могут не отражать последнее состояние.

**15. Как удалить элементы из коллекции во время итерации?**
Три безопасных способа: (1) `Iterator.remove()` — удаляет текущий элемент; (2) `Collection.removeIf(predicate)` — лаконично, Java 8+; (3) собрать элементы для удаления в отдельный список и вызвать `removeAll()` после цикла. Вызов `list.remove()` внутри enhanced for неверен и бросает `ConcurrentModificationException`.

**16. В чём разница между `Arrays.asList()`, `List.of()` и `new ArrayList<>()`?**
`new ArrayList<>()` полностью изменяем. `Arrays.asList()` — фиксированного размера, подкреплён массивом: `set()` работает, но `add`/`remove` бросают `UnsupportedOperationException`, и допускает `null`. `List.of()` полностью **immutable** — любая модификация бросает исключение, и `null` не допускает.

**17. Как создать immutable коллекцию?**
`List.of(...)` / `Set.of(...)` / `Map.of(...)` (Java 9+) — immutable и не допускают null. `Collections.unmodifiableList(list)` оборачивает существующую коллекцию — оригинал всё ещё может меняться, и представление это видит. `List.copyOf(list)` (Java 10+) делает настоящий immutable снапшот.

**18. Разница между `Collection` и `Collections`?**
`Collection` — корневой **интерфейс** иерархии (`List`, `Set`, `Queue`). `Collections` — **утилитный класс** со статическими методами: `sort()`, `reverse()`, `max()`, `frequency()`, `unmodifiableList()`, `synchronizedList()` и т.д.

**19. Как растёт `ArrayList` внутри?**
При создании через конструктор по умолчанию внутренний массив пуст. Первый `add()` выделяет `DEFAULT_CAPACITY = 10`. При заполнении `grow()` создаёт новый массив размером `oldCapacity + (oldCapacity >> 1)` (примерно в 1.5 раза) и копирует элементы через `Arrays.copyOf()`. Разовый resize стоит O(n), амортизированно O(1) на add.

**20. `ArrayList` vs `LinkedList` на уровне памяти?**
`ArrayList` — один непрерывный массив ссылок, примерно 4–8 байт на элемент. `LinkedList` хранит объект `Node` на каждый элемент (value + prev + next), поэтому использует в несколько раз больше памяти. На 1 миллион элементов `ArrayList` — несколько MB, а `LinkedList` — десятки MB.

**21. Почему `ArrayList` предпочтительнее `LinkedList` даже для вставок?**
`LinkedList` даёт O(1) вставку, только если ты уже держишь узел. Вставка по индексу всё равно стоит O(n) — нужно дойти до позиции, плюс накладные расходы на node и плохая локальность для кеша. На практике `ArrayList` обычно выигрывает даже для вставок в середину на реальных размерах.

**22. Когда использовать `WeakHashMap`? `EnumMap`/`EnumSet`?**
`WeakHashMap` удерживает ключи через слабые ссылки, поэтому записи исчезают, когда на ключ нет сильных ссылок снаружи — полезно для кэшей и реестров слушателей. `EnumMap`/`EnumSet` специализированы для enum-ключей/элементов: используют массив по `ordinal()` (или битовый вектор), давая O(1) операции и минимальную память.

### Вопросы с кодом

**1. Что выведет этот код?**
```java
List<String> list = new ArrayList<>();
list.add("A"); list.add("B"); list.add("C");
for (String s : list) {
    if (s.equals("B")) list.remove(s);
}
System.out.println(list);
```
**Ответ:** `ConcurrentModificationException`. Enhanced for неявно использует итератор, а `list.remove()` изменяет коллекцию без уведомления итератора. Правильно: `list.removeIf(s -> s.equals("B"))`.

**2. Что выведет этот код?**
```java
Set<Short> set = new HashSet<>();
for (short i = 0; i < 10; i++) {
    set.add(i);
    set.remove(i - 1); // (int) i - 1
}
System.out.println(set.size());
```
**Ответ:** 10. Ошибка в том, что `i - 1` — это `int`, а не `short`. `set.remove(Integer.valueOf(i-1))` ищет `Integer`, а в Set хранятся `Short`. Ничего не удаляется.

**3. Что выведет этот код?**
```java
Map<String, String> map = new HashMap<>();
map.put("a", "1");
map.put("b", "2");
map.put("c", "3");
map.computeIfAbsent("a", k -> "10");
map.computeIfAbsent("d", k -> "40");
System.out.println(map.get("a") + map.get("d"));
```
**Ответ:** `"140"`. `computeIfAbsent` выполняется только если ключа нет. `"a"` уже есть — ничего не меняется. `"d"` нет — добавляется `"40"`.

**4. Что выведет этот код?**
```java
List<Integer> list = new ArrayList<>(List.of(1, 2, 3, 4, 5));
list.subList(1, 3).clear();
System.out.println(list);
```
**Ответ:** `[1, 4, 5]`. `subList()` возвращает **представление** оригинального списка, а не копию. `clear()` на subList удаляет эти элементы из оригинального списка.

**5. Что выведет этот код?**
```java
Map<String, Integer> map = new HashMap<>();
map.put("x", 1);
map.put("y", 2);
map.merge("x", 10, Integer::sum);
map.merge("z", 30, Integer::sum);
System.out.println(map.get("x") + " " + map.get("z"));
```
**Ответ:** `11 30`. `merge()` для существующего ключа применяет функцию (1 + 10 = 11). Для отсутствующего — просто вставляет значение (30).

### 1. Reverse Array — развернуть массив

**Условие:** Дан массив целых чисел. Разверни его **на месте** (in-place), не используя дополнительный массив.

```java
public void reverse(int[] arr) {
    int left = 0, right = arr.length - 1;
    while (left < right) {
        int tmp = arr[left];
        arr[left] = arr[right];
        arr[right] = tmp;
        left++;
        right--;
    }
}
```

**Разбор:** Техника двух указателей. Один с начала, другой с конца — меняем местами и сдвигаемся к центру.

| Метрика | Значение |
|---|---|
| Сложность | O(n) время, O(1) память |
| Ключевой приём | Два указателя (left/right) |

### 2. Find Missing Number — найти пропущенное число

**Условие:** Дан массив из `n` чисел в диапазоне `[0, n]`, одно число пропущено. Найди его.

```java
// Решение 1 — через HashSet (для понимания коллекций):
public int missingNumberSet(int[] nums) {
    Set<Integer> set = new HashSet<>();
    for (int n : nums) set.add(n);
    for (int i = 0; i <= nums.length; i++) {
        if (!set.contains(i)) return i;
    }
    return -1;
}

// Решение 2 — математическое (O(1) по памяти):
public int missingNumberMath(int[] nums) {
    int n = nums.length;
    int expected = n * (n + 1) / 2;
    int sum = 0;
    for (int num : nums) sum += num;
    return expected - sum;
}
```

**Разбор:** HashSet даёт O(1) проверку наличия — очевидный подход. Математический вариант через сумму арифметической прогрессии экономнее по памяти.

| Метрика | Значение |
|---|---|
| Сложность | O(n) время, O(n) или O(1) память |
| Ключевая структура | `HashSet<Integer>` или сумма |

### 3. Remove Duplicates — убрать дубликаты из ArrayList

**Условие:** Дан `ArrayList<String>` с дубликатами. Верни новый список только из уникальных значений, сохранив порядок первого появления.

```java
public List<String> removeDuplicates(List<String> input) {
    Set<String> seen = new LinkedHashSet<>(input); // сохраняет порядок
    return new ArrayList<>(seen);
}

// Ручной вариант (чтобы показать логику):
public List<String> removeDuplicatesManual(List<String> input) {
    Set<String> seen = new HashSet<>();
    List<String> result = new ArrayList<>();
    for (String s : input) {
        if (seen.add(s)) {          // add() вернёт false, если уже есть
            result.add(s);
        }
    }
    return result;
}
```

**Разбор:** `LinkedHashSet` — лучшее из двух миров: уникальность HashSet + порядок ArrayList. Второй вариант показывает механику.

| Метрика | Значение |
|---|---|
| Сложность | O(n) время, O(n) память |
| Ключевые структуры | `LinkedHashSet<T>`, `HashSet<T>` + `ArrayList<T>` |

### 4. Merge Two Sorted Lists — слить два отсортированных списка

**Условие:** Даны два отсортированных `List<Integer>`. Верни новый отсортированный список, содержащий все элементы из обоих.

```java
public List<Integer> mergeSorted(List<Integer> a, List<Integer> b) {
    List<Integer> result = new ArrayList<>(a.size() + b.size());
    int i = 0, j = 0;

    while (i < a.size() && j < b.size()) {
        result.add(a.get(i) <= b.get(j) ? a.get(i++) : b.get(j++));
    }
    while (i < a.size()) result.add(a.get(i++));
    while (j < b.size()) result.add(b.get(j++));

    return result;
}
```

**Разбор:** Техника двух указателей на ArrayList. Меньший элемент забираем, указатель сдвигаем. Когда один список кончился — дописываем хвост второго. `get(i)` у ArrayList работает за O(1).

| Метрика | Значение |
|---|---|
| Сложность | O(a + b) время, O(a + b) память |
| Ключевые структуры | `ArrayList<Integer>` + два указателя |

### 5. Rotate List — сдвинуть ArrayList на k позиций

**Условие:** Дан `ArrayList<Integer>` и число `k`. Сдвинь все элементы вправо на `k` позиций. Последние k элементов становятся первыми.

```java
public void rotate(List<Integer> list, int k) {
    int n = list.size();
    if (n == 0) return;
    k = k % n;             // если k больше размера
    if (k == 0) return;

    // Разворачиваем весь список, потом части
    Collections.reverse(list);           // [7,6,5,4,3,2,1] для n=7,k=3
    Collections.reverse(list.subList(0, k));   // первые k
    Collections.reverse(list.subList(k, n));   // остальные
    // Результат: [5,6,7,1,2,3,4]
}
```

**Разбор:** Тройной reverse — элегантный трюк, популярный на собесах. Работает за O(n) и in-place. `Collections.reverse()` + `subList()` — полезная комбинация.

| Метрика | Значение |
|---|---|
| Сложность | O(n) время, O(1) память |
| Ключевые методы | `Collections.reverse()`, `List.subList()` |

### 6. Find Middle — найти середину списка

**Условие:** Дан `List<Integer>`. Верни элемент, находящийся в середине. Для чётного размера — левый из двух центральных.

```java
// Простой способ (только для ArrayList):
public int findMiddleSimple(List<Integer> list) {
    return list.get((list.size() - 1) / 2);
}

// Универсальный способ (два указателя, работает с LinkedList):
public int findMiddleTwoPointer(List<Integer> list) {
    int slow = 0, fast = 0;
    while (fast < list.size() - 1) {
        slow++;           // медленный — шаг 1
        fast += 2;        // быстрый — шаг 2
    }
    return list.get(slow);
}
```

**Разбор:** `get(index)` у ArrayList — O(1), так что ответ в лоб. Но техника двух указателей (fast/slow) обязательна к знанию: она работает для односвязных списков (LinkedList), где нет быстрого доступа по индексу.

| Метрика | Значение |
|---|---|
| Сложность | O(1) / O(n) время, O(1) память |
| Ключевой приём | Два указателя (fast / slow) |

### 7. Intersection of Two Arrays — пересечение через Set

**Условие:** Даны два массива. Верни их пересечение — элементы, которые встречаются в обоих, без дубликатов.

```java
public Set<Integer> intersection(int[] a, int[] b) {
    Set<Integer> setA = new HashSet<>();
    for (int n : a) setA.add(n);

    Set<Integer> result = new HashSet<>();
    for (int n : b) {
        if (setA.contains(n)) {
            result.add(n);
        }
    }
    return result;
}

// Компактнее через retainAll (в одну строку):
public Set<Integer> intersectionRetain(int[] a, int[] b) {
    Set<Integer> setA = Arrays.stream(a).boxed().collect(Collectors.toSet());
    Set<Integer> setB = Arrays.stream(b).boxed().collect(Collectors.toSet());
    setA.retainAll(setB);
    return setA;
}
```

**Разбор:** `retainAll()` — специальный метод Set для пересечения. Сначала собираем один массив в HashSet (даёт O(1) проверку), потом фильтруем второй.

| Метрика | Значение |
|---|---|
| Сложность | O(a + b) время, O(min(a, b)) память |
| Ключевая структура | `HashSet<Integer>` + `retainAll()` |

### 8. Two Sum — найти индексы двух чисел, дающих target

**Условие:** Дан массив `nums` и `target`. Верни индексы двух чисел, сумма которых равна `target`. Гарантируется ровно одно решение.

```java
public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> map = new HashMap<>(); // значение -> индекс
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (map.containsKey(complement)) {
            return new int[]{map.get(complement), i};
        }
        map.put(nums[i], i);
    }
    return new int[0]; // не найдено
}
```

**Разбор:** Самая известная задача на HashMap. Проходим массив один раз, на каждом шаге проверяем, не встречали ли число-дополнение. O(1) поиск в Map делает решение линейным вместо квадратичного перебора.

| Метрика | Значение |
|---|---|
| Сложность | O(n) время, O(n) память |
| Ключевая структура | `HashMap<Integer, Integer>` |

### 9. Count Character Frequency — частота символов в строке

**Условие:** Дана строка. Верни `Map<Character, Integer>`, где ключ — символ, значение — сколько раз он встречается.

```java
public Map<Character, Integer> charFrequency(String s) {
    Map<Character, Integer> freq = new HashMap<>();
    for (char c : s.toCharArray()) {
        freq.merge(c, 1, Integer::sum);  // лаконичный подсчёт
    }
    return freq;
}

// Понятный вариант (для новичков):
public Map<Character, Integer> charFrequencyClassic(String s) {
    Map<Character, Integer> freq = new HashMap<>();
    for (char c : s.toCharArray()) {
        if (freq.containsKey(c)) {
            freq.put(c, freq.get(c) + 1);
        } else {
            freq.put(c, 1);
        }
    }
    return freq;
}
```

**Разбор:** `merge(c, 1, Integer::sum)` заменяет целый if-else блок. Если ключа нет — вставляет 1, если есть — прибавляет к текущему значению.

| Метрика | Значение |
|---|---|
| Сложность | O(n) время, O(n) память |
| Ключевая структура | `HashMap<Character, Integer>` + `merge()` |

### 10. Most Frequent Element — самый частый элемент

**Условие:** Дан массив целых чисел. Верни элемент, который встречается чаще всех. Если таких несколько — верни любой.

```java
public int mostFrequent(int[] nums) {
    Map<Integer, Integer> freq = new HashMap<>();
    for (int n : nums) {
        freq.merge(n, 1, Integer::sum);
    }
    return Collections.max(freq.entrySet(), Map.Entry.comparingByValue()).getKey();
}
```

**Разбор:** Два шага: подсчёт через HashMap, поиск максимума через `Collections.max()` с компаратором по значению. В production для больших данных лучше PriorityQueue, но на собесе это лаконичное решение проходит.

| Метрика | Значение |
|---|---|
| Сложность | O(n) время, O(n) память |
| Ключевые структуры | `HashMap<Integer, Integer>` + `Collections.max()` |

### 11. Valid Parentheses — правильная скобочная последовательность

**Условие:** Дана строка из скобок `(){}[]`. Определи, правильно ли они открыты, закрыты и вложены.

```java
public boolean isValid(String s) {
    Map<Character, Character> pairs = Map.of(')', '(', '}', '{', ']', '[');
    Deque<Character> stack = new ArrayDeque<>();

    for (char c : s.toCharArray()) {
        if (pairs.containsValue(c)) {           // открывающая скобка
            stack.push(c);
        } else {                                  // закрывающая
            if (stack.isEmpty() || stack.pop() != pairs.get(c)) {
                return false;
            }
        }
    }
    return stack.isEmpty();
}
```

**Разбор:** `Deque` (интерфейс) с `ArrayDeque` — современная замена устаревшему `Stack`. Открывающие скобки кладём, на закрывающих проверяем верхушку. Совпадение не сошлось или стек пуст — ошибка.

| Метрика | Значение |
|---|---|
| Сложность | O(n) время, O(n) память |
| Ключевая структура | `Deque<Character>` (`ArrayDeque`) |

### 12. Top K Frequent Elements — K самых частых элементов

**Условие:** Дан массив целых чисел и `k`. Верни `k` наиболее часто встречающихся элементов.

```java
public List<Integer> topKFrequent(int[] nums, int k) {
    // 1. Подсчёт частоты
    Map<Integer, Integer> freq = new HashMap<>();
    for (int n : nums) freq.merge(n, 1, Integer::sum);

    // 2. Сортировка по убыванию частоты
    List<Map.Entry<Integer, Integer>> entries = new ArrayList<>(freq.entrySet());
    entries.sort((a, b) -> b.getValue() - a.getValue());

    // 3. Первые k элементов
    List<Integer> result = new ArrayList<>();
    for (int i = 0; i < k && i < entries.size(); i++) {
        result.add(entries.get(i).getKey());
    }
    return result;
}
```

**Разбор:** Комбинация всех трёх структур: HashMap для подсчёта, ArrayList для сортировки entrySet() по значениям. В production стоит PriorityQueue (O(n log k) вместо O(n log n)), но эта версия нагляднее для собеса.

| Метрика | Значение |
|---|---|
| Сложность | O(n log n) время, O(n) память |
| Ключевые структуры | `HashMap` → `ArrayList<Entry>` → сортировка |

---

> [!tip] Что запомнить для собеса
> - **HashMap** — для «найди быстро» и «подсчитай»
> - **HashSet** — для уникальности и пересечений
> - **ArrayList** — лучший выбор по умолчанию, get() за O(1)
> - **Deque/ArrayDeque** — для стека и очереди (скобки, палиндромы)
> - **Два указателя** — приём для массивов и списков (reverse, merge, find middle)
> - **Всегда называй O(...)** после решения — показывает глубину понимания
