## Зачем нужны дженерики

До дженериков (Java < 5) коллекции хранили `Object` — приходилось приводить тип каждый раз:

```java
// Без дженериков — компилируется, но может упасть в runtime
List list = new ArrayList();
list.add("hello");
list.add(42);                         // нет ошибки при компиляции
String s = (String) list.get(1);      // ClassCastException в runtime!
```

Проблема: компилятор понятия не имеет, какие типы в списке. Ошибки появляются только во время выполнения, а это слишком поздно.

Дженерики решают это, позволяя указать тип:

```java
// С дженериками — ошибка ловится при компиляции
List<String> list = new ArrayList<>();
list.add("hello");
// list.add(42);                      // ошибка компиляции — поймана сразу
String s = list.get(0);               // приведение не нужно
```

> [!danger] Ключевая идея
> Дженерики переносят проверку типов из **runtime** в **compile time**. Компилятор гарантирует типобезопасность — если скомпилировалось, `ClassCastException` не будет. Это называется "типобезопасность".

## Обобщённые классы

Обобщённый класс имеет один или несколько **параметров типа** — заполнителей для реальных типов, которые указываются при создании экземпляра.

```java
public class Box<T> {
    private T value;

    public Box(T value) { this.value = value; }
    public T getValue() { return value; }
    public void setValue(T value) { this.value = value; }
}
```

`T` — **параметр типа**, заполнитель. При создании `Box` ты заменяешь `T` реальным типом:

```java
Box<String> stringBox = new Box<>("Hello");
String s = stringBox.getValue();      // приведение не нужно — компилятор знает, что это String

Box<Integer> intBox = new Box<>(42);
int n = intBox.getValue();            // авто-распаковка в int

// Box<int> x = ...;                  // ошибка компиляции — примитивы не допускаются
```

### Несколько параметров типа

```java
public class Pair<K, V> {
    private K key;
    private V value;

    public Pair(K key, V value) {
        this.key = key;
        this.value = value;
    }

    public K getKey() { return key; }
    public V getValue() { return value; }
}

Pair<String, Integer> entry = new Pair<>("age", 25);
String key = entry.getKey();       // "age"
Integer value = entry.getValue();  // 25
```

### Соглашение об именовании

Имена параметров типа — одиночные заглавные буквы по соглашению:

| Буква | Обычное значение |
|---|---|
| `T` | Type (общий) |
| `E` | Element (коллекции) |
| `K` | Key (словари) |
| `V` | Value (словари) |
| `R` | Return type |
| `N` | Number |

## Обобщённые методы

Метод может иметь собственный параметр типа, независимый от класса.

```java
public class Utils {
    // <T> перед типом возвращаемого значения объявляет параметр типа
    public static <T> T getFirst(List<T> list) {
        if (list == null || list.isEmpty()) return null;
        return list.get(0);
    }
}

String first = Utils.getFirst(List.of("a", "b", "c"));  // "a"
Integer num = Utils.getFirst(List.of(1, 2, 3));          // 1
```

Компилятор **выводит** `T` из аргумента. Можно указать и явно:

```java
String first = Utils.<String>getFirst(List.of("a", "b"));
```

### Зачем нужны обобщённые методы?

Они позволяют написать один метод, работающий с любым типом, при этом сохраняя типобезопасность. Без дженериков пришлось бы писать отдельные методы для `List<String>`, `List<Integer>`, `List<User>` и т.д. — или использовать `List<Object>` и потерять типобезопасность.

## Ограниченные типы

Иногда нужно ограничить, какие типы можно использовать как параметр. Например, метод сравнения значений нуждается в типах, реализующих `Comparable`.

### Верхняя граница: `extends`

`<T extends SomeClass>` означает "T должен быть SomeClass или его подклассом".

```java
// T должен реализовывать Comparable
public static <T extends Comparable<T>> T findMax(List<T> list) {
    T max = list.get(0);
    for (T item : list) {
        if (item.compareTo(max) > 0) {
            max = item;
        }
    }
    return max;
}

findMax(List.of(3, 1, 4, 1, 5));         // 5
findMax(List.of("banana", "apple"));      // "banana"
// findMax(List.of(new Object()));         // ошибка компиляции — Object не Comparable
```

### Несколько границ

```java
// T должен наследовать Number И реализовывать Comparable
public static <T extends Number & Comparable<T>> T findMax(List<T> list) { ... }
```

> [!info] extends в дженериках
> В дженериках `extends` означает и `extends` (классы), и `implements` (интерфейсы). Всегда пишется `extends`, даже для интерфейсов: `<T extends Comparable>`, а не `<T implements Comparable>`.

## Подстановочные символы

Подстановочные символы (`?`) представляют **неизвестный тип**. Используются в параметрах методов, когда не нужно именовать тип.

### Неограниченный: `?`

"Любой тип." Используется, когда метод не зависит от конкретного типа.

```java
public void printAll(List<?> list) {
    for (Object item : list) {
        System.out.println(item);
    }
}

printAll(List.of("a", "b"));     // работает
printAll(List.of(1, 2, 3));       // работает
printAll(List.of(new User()));    // работает
```

### С верхней границей: `? extends T`

"T или любой подтип T." Используй, когда **читаешь** из коллекции.

```java
// принимает List<Integer>, List<Double>, List<Number> и т.д.
public double sum(List<? extends Number> numbers) {
    double total = 0;
    for (Number n : numbers) {
        total += n.doubleValue();
    }
    return total;
}

sum(List.of(1, 2, 3));           // List<Integer> — работает
sum(List.of(1.5, 2.5));          // List<Double> — работает
```

### С нижней границей: `? super T`

"T или любой супертип T." Используй, когда **пишешь** в коллекцию.

```java
// принимает List<Integer>, List<Number>, List<Object>
public void addNumbers(List<? super Integer> list) {
    list.add(1);
    list.add(2);
    list.add(3);
}

List<Number> numbers = new ArrayList<>();
addNumbers(numbers);              // работает — Number является супертипом Integer
```

### Принцип PECS

**P**roducer **E**xtends, **C**onsumer **S**uper.

| Ты хочешь... | Используй | Почему |
|---|---|---|
| **Читать** из коллекции (она производит значения) | `? extends T` | Ты знаешь, что получишь как минимум `T` |
| **Писать** в коллекцию (она потребляет значения) | `? super T` | Ты знаешь, что она может принять `T` |
| **И читать, и писать** | `T` (без подстановки) | Нужен точный тип |

> [!tip] Не зубри — пойми
> `? extends Number` -> "Я буду читать Number из этого списка, мне неважно, Integer они или Double."
> `? super Integer` -> "Я буду класть Integer в этот список, мне неважно, хранит ли он Number или Object."

## Стирание типов

Дженерики существуют только на этапе **компиляции**. Во время выполнения JVM не знает о параметрах типа — они **стираются**.

```java
// Что ты пишешь:
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);

// Что видит JVM (после стирания):
List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0);    // компилятор вставляет приведение
```

### Последствия стирания типов

```java
// 1. Нельзя создать массив обобщённого типа
// T[] arr = new T[10];              // ошибка компиляции

// 2. Нельзя использовать instanceof с дженериками
// if (list instanceof List<String>)  // ошибка компиляции

// 3. Нельзя перегрузить метод только по параметру типа
// void process(List<String> list) { }
// void process(List<Integer> list) { }  // ошибка компиляции — одинаковое стирание

// 4. В runtime List<String> и List<Integer> — один класс
List<String> strings = new ArrayList<>();
List<Integer> ints = new ArrayList<>();
System.out.println(strings.getClass() == ints.getClass());  // true
```

> [!info] Зачем существует стирание
> Когда дженерики добавили в Java 5, разработчикам нужна была обратная совместимость с существующим кодом на "сырых" типах (`List` без `<>`). Стирание типов позволило новому обобщённому коду работать со старыми библиотеками без изменений. Цена — отсутствие информации о типах в runtime.

## Дженерики на практике

### AQA: обобщённый метод Page Object

```java
public abstract class BasePage {
    protected WebDriver driver;

    public BasePage(WebDriver driver) { this.driver = driver; }

    // Обобщённый метод — возвращает фактический тип страницы для цепочки вызовов
    @SuppressWarnings("unchecked")
    protected <T extends BasePage> T clickAndReturn(By locator) {
        driver.findElement(locator).click();
        return (T) this;
    }
}

public class LoginPage extends BasePage {
    public LoginPage(WebDriver driver) { super(driver); }

    public LoginPage enterUsername(String name) {
        driver.findElement(By.id("user")).sendKeys(name);
        return this;
    }

    public DashboardPage clickLogin() {
        driver.findElement(By.id("login")).click();
        return new DashboardPage(driver);
    }
}
```

### Обобщённая утилита для тестовых данных

```java
public class TestDataLoader {
    // Загрузка любого типа из JSON-файла
    public static <T> T loadFromJson(String path, Class<T> type) {
        String json = Files.readString(Path.of(path));
        return new Gson().fromJson(json, type);
    }
}

User user = TestDataLoader.loadFromJson("data/user.json", User.class);
Config config = TestDataLoader.loadFromJson("data/config.json", Config.class);
```

### Коллекции, которые ты используешь каждый день

```java
List<String> names = new ArrayList<>();           // список строк
Map<String, Integer> scores = new HashMap<>();    // ключи String, значения Integer
Set<User> uniqueUsers = new HashSet<>();          // множество объектов User
Optional<String> maybeName = Optional.of("Alice"); // возможно содержит String
```

> [!tip] Когда видишь `<>` в коде
> Каждый раз, когда ты видишь угловые скобки в Java — это дженерики. `List<String>`, `Map<K, V>`, `Optional<User>`, `Comparable<T>` — все они используют тот же механизм, который ты изучил здесь.

## Вопросы для собеседования

### Вопросы начального уровня

**1. Что такое дженерики в Java? Зачем они нужны?**
Дженерики позволяют писать классы и методы, работающие с любым типом, сохраняя типобезопасность. Компилятор проверяет типы на этапе компиляции, поэтому ошибки ловятся рано, а не выбрасывают `ClassCastException` в runtime.

**2. Что такое параметр типа?**
Заполнитель для реального типа, объявляемый в угловых скобках: `<T>`. При создании экземпляра (`Box<String>`) `T` заменяется на `String`. Компилятор гарантирует, что используются только значения `String`.

**3. Можно ли использовать примитивные типы с дженериками?**
Нет. Дженерики работают только со ссылочными типами. Используй классы-обёртки: `List<Integer>`, а не `List<int>`, `Box<Double>`, а не `Box<double>`. Автоупаковка (autoboxing) обеспечивает преобразование.

**4. Что такое ограниченный параметр типа?**
`<T extends Comparable<T>>` ограничивает `T` типами, реализующими `Comparable`. Это позволяет вызывать `compareTo()` внутри метода. Без ограничения компилятор не знает, что у `T` есть такой метод.

**5. Что такое "сырой" тип? Почему он плох?**
Сырой тип — обобщённый класс без параметров типа: `List list = new ArrayList();`. Он отключает проверку типов, теряет всю безопасность дженериков, и компилятор выдаёт предупреждения. Всегда указывай параметры типа.

**6. Что такое "ромб" (`<>`) в `new ArrayList<>()`?**
Оператор ромб (Java 7+) говорит компилятору вывести тип из левой части: `List<String> list = new ArrayList<>();` — компилятор выводит `ArrayList<String>`. До Java 7 приходилось писать `new ArrayList<String>()`.

### Вопросы среднего уровня

**1. В чём разница между `<T>` и `<?>`?**
`<T>` объявляет именованный параметр типа — на `T` можно ссылаться в теле метода/класса. `<?>` — подстановочный символ, означающий "неизвестный тип", используется, когда не нужно ссылаться на тип по имени.

**2. Что такое подстановка с верхней границей? Приведи пример.**
`<? extends Number>` означает "Number или любой подкласс". Используется при чтении из коллекции: `double sum(List<? extends Number> list)` принимает `List<Integer>`, `List<Double>` и т.д.

**3. Что такое принцип PECS?**
Producer Extends, Consumer Super. Если коллекция предоставляет значения — используй `? extends`. Если принимает значения — используй `? super`. Если и то, и другое — используй конкретный тип.

**4. Что такое стирание типов?**
В runtime информация о типах дженериков удаляется. `List<String>` становится `List`. Компилятор вставляет приведения типов где нужно. Это сделано для обратной совместимости с кодом до Java 5.

**5. Является ли `List<Object>` супертипом `List<String>`?**
Нет. Дженерики **инвариантны**. `String` наследует `Object`, но `List<String>` НЕ наследует `List<Object>`. Это предотвращает небезопасные операции: если бы это было допустимо, можно было бы добавить `Integer` в `List<String>` через ссылку `List<Object>`.

**6. Почему нельзя написать `if (list instanceof List<String>)`?**
Из-за стирания типов. В runtime JVM видит только `List` — часть `<String>` уже удалена. Можно проверить `list instanceof List<?>` (проверка сырого типа), но не конкретный параметр типа.

**7. Может ли обобщённый класс наследовать `Throwable`?**
Нет. Нельзя создать обобщённый класс исключений (`class MyException<T> extends Exception`). Механизм catch нуждается в точных типах в runtime, а стирание типов их удаляет.

**8. В чём разница между `List<Object>` и `List<?>`?**
`List<Object>` — **конкретный** тип, хранит любые объекты. Можно читать как `Object` и **писать** любые типы. `List<?>` — **неизвестный** тип. Можно читать как `Object`, но **нельзя писать** ничего кроме `null`.

### Вопросы повышенного уровня

**1. Что такое вариативность (variance) дженериков?**
Дженерики **инвариантны** по умолчанию: `List<String>` не является подтипом `List<Object>`, хотя `String` — подтип `Object`. Это блокирует небезопасные операции.
**Ковариантность** (`? extends T`) разрешает подтипы: `List<Integer>` — подтип `List<? extends Number>`.
**Контравариантность** (`? super T`) разрешает супертипы: `List<Number>` — подтип `List<? super Integer>`.

```java
List<String> strs = new ArrayList<>();
// List<Object> objs = strs;          // ❌ инвариантность — ошибка

List<? extends Number> nums = new ArrayList<Integer>(); // ✅ ковариантность
List<? super Integer> ints = new ArrayList<Number>();   // ✅ контравариантность
```

**2. Почему нельзя вставить элемент в `List<? extends Number>`?**
Тип неизвестен. Это может быть `List<Integer>`, `List<Double>` или `List<Number>`. Компилятор не знает, совместим ли новый элемент с конкретным подтипом. Можно только **читать** — гарантированно получать `Number`.

```java
List<? extends Number> list = new ArrayList<Integer>();
// list.add(3.14);    // ❌ — Double не может быть Integer
// list.add(42);      // ❌ — компилятор не знает, Integer ли внутри
Number n = list.get(0); // ✅ — чтение безопасно
```

**3. Почему нельзя перегрузить метод только по параметру типа?**
Из-за стирания типов. Два метода с одинаковой сигнатурой после стирания — один и тот же метод. Компилятор не может их различить.

```java
// ❌ Не компилируется — одинаковое стирание:
void process(List<String> list) { }
void process(List<Integer> list) { } // оба process(List) после стирания
```

**4. В чём разница между reifiable и non-reifiable типами?**
**Reifiable** — тип, информация о котором полностью доступна в runtime: примитивы (`int`), непараметризованные типы (`String`), сырые типы (`List`), массивы (`String[]`), и `?` (unbounded wildcard).
**Non-reifiable** — типы, информация о которых частично стёрта: `List<String>`, `List<Integer>`. Для них нельзя использовать `instanceof` и создавать массивы.

```java
boolean a = "hello" instanceof String;         // ✅ reifiable
boolean b = list instanceof List<?>;            // ✅ unbounded wildcard
// boolean c = list instanceof List<String>;    // ❌ non-reifiable — ошибка
```

**5. Можно ли создать массив обобщённого типа?**
Нет. `new T[10]` недопустимо из-за стирания типов — JVM не знает, какой тип `T` в runtime. Используй `List<T>` вместо массива, или передай `Class<T>` и используй `Array.newInstance()`.

**6. Что такое `Class<T>` как параметр типа (type token)?**
Позволяет передать информацию о типе в runtime, обходя стирание. Используется в библиотеках вроде Jackson, Gson, Spring для создания экземпляров нужного типа.

```java
public <T> T fromJson(String json, Class<T> type) {
    return new Gson().fromJson(json, type);
}
User user = fromJson("{\"name\":\"Alice\"}", User.class);
// Class<User> передаёт runtime-информацию для создания правильного типа
```

**7. Когда оправдано `@SuppressWarnings("unchecked")`?**
Когда ты, как программист, точно знаешь, что приведение типа безопасно, но компилятор не может это доказать из-за стирания типов. Типичные случаи: (1) массивы в обобщённых коллекциях, (2) raw-типы в legacy коде после миграции на дженерики, (3) касты в `(T[])` и `(T)`. **Никогда** не используй без понимания, почему ты уверен в безопасности.

### Вопросы с кодом

**1. Что выведет этот код?**
```java
List<String> strings = new ArrayList<>();
List<Integer> integers = new ArrayList<>();
System.out.println(strings.getClass() == integers.getClass());
```
**Ответ:** `true`. В runtime оба — `ArrayList`. Информация о `<String>` и `<Integer>` стёрта. `getClass()` возвращает одинаковый объект `Class` для обоих.

**2. Скомпилируется ли этот код?**
```java
public void addToList(List<?> list) {
    list.add("hello");
}
```
**Ответ:** Нет. `List<?>` означает «неизвестный тип». Можно читать как `Object`, но единственное, что можно добавить — `null`. Компилятор блокирует запись, чтобы не нарушить типобезопасность.

**3. Скомпилируется ли этот код?**
```java
List<? extends Number> nums = new ArrayList<Integer>();
nums.add(42);
```
**Ответ:** Нет. `nums` ковариантен — может быть `List<Integer>`, `List<Double>` или любым другим подтипом. Компилятор не может гарантировать, что `42` — правильный тип для неизвестного подтипа `Number`. Читать — можно. Писать — нет.

**4. Что выведет этот код?**
```java
public class Test {
    public static void main(String[] args) {
        Box<Integer> box = new Box<>();
        System.out.println(box.get());
    }
}
class Box<T> {
    public T get() { return null; }
}
```
**Ответ:** `null`. После стирания `Box.get()` возвращает `Object`, и компилятор вставляет приведение к `Integer`. Но значение внутри — `null`. Приведение `null` к любому типу работает без ошибок.

**5. Скомпилируется ли этот код?**
```java
class Container<T> {
    private T[] items = new T[10];
    public T get(int i) { return items[i]; }
}
```
**Ответ:** Нет. `new T[10]` не компилируется из-за стирания типов — JVM не знает, какой массив создавать. Решение: `(T[]) new Object[10]` с `@SuppressWarnings("unchecked")`, или `List<T>` вместо массива.

### 1. Напиши generic метод поиска максимума

**Задание:** Напиши статический generic метод `findMax`, который принимает `List<T>` и возвращает максимальный элемент. T должен быть ограничен Comparable.

```java
public static <T extends Comparable<T>> T findMax(List<T> list) {
    T max = list.get(0);
    for (T item : list) {
        if (item.compareTo(max) > 0) max = item;
    }
    return max;
}

// Использование:
System.out.println(findMax(List.of(3, 1, 4, 1, 5)));       // 5
System.out.println(findMax(List.of("apple", "banana")));    // "banana"
```

**Разбор:** `<T extends Comparable<T>>` — bounded type parameter. Без него нельзя вызвать `compareTo()`. Метод работает с любым типом, реализующим Comparable.

| Сложность | O(n) время, O(1) память |

### 2. PECS — что скомпилируется?

**Задание:** Отметь, какие строки скомпилируются, а какие — нет. Объясни почему.

```java
List<? extends Number> readers = new ArrayList<Integer>();
// readers.add(42);                        // 1 — ?
Number n = readers.get(0);                 // 2 — ?

List<? super Integer> writers = new ArrayList<Number>();
writers.add(42);                           // 3 — ?
Object o = writers.get(0);                 // 4 — ?
```

**Ответ:** 1 — ❌ не скомпилируется (ковариантность — тип неизвестен, нельзя писать). 2 — ✅ скомпилируется (читаем как Number). 3 — ✅ скомпилируется (контравариантность — можно писать Integer). 4 — ✅ скомпилируется, но тип — Object, не Integer.

**Разбор:** PECS: Producer Extends (только читаем), Consumer Super (только пишем).

### 3. Generic класс Pair

**Задание:** Реализуй generic класс `Pair<K, V>` с методами `getKey()` и `getValue()`.

```java
public class Pair<K, V> {
    private final K key;
    private final V value;

    public Pair(K key, V value) { this.key = key; this.value = value; }
    public K getKey() { return key; }
    public V getValue() { return value; }
}

// Использование:
Pair<String, Integer> pair = new Pair<>("age", 25);
String key = pair.getKey();     // "age" — без каста
Integer val = pair.getValue();  // 25 — без каста
```

**Разбор:** K и V — два независимых параметра типа. Компилятор выводит типы из конструктора (ромб `<>`).

### 4. Wildcard capture — исправь код

**Задание:** Этот код не компилируется. Исправь его через wildcard capture.

```java
public void swap(List<?> list, int i, int j) {
    list.set(i, list.get(j)); // ❌ не компилируется
}
```

**Решение:**

```java
private <T> void swapHelper(List<T> list, int i, int j) {
    T tmp = list.get(i);
    list.set(i, list.get(j));
    list.set(j, tmp);
}
public void swap(List<?> list, int i, int j) {
    swapHelper(list, i, j); // компилятор захватывает ? как T
}
```

**Разбор:** Проблема — компилятор не знает, что `?` в `get()` и `set()` — один и тот же тип. Вспомогательный метод с `<T>` захватывает (capture) неизвестный тип и делает оба вызова типобезопасными.

### 5. Type erasure — проверь на практике

**Задание:** Что выведет этот код? Почему?

```java
List<String> strings = new ArrayList<>();
List<Integer> integers = new ArrayList<>();
System.out.println(strings.getClass() == integers.getClass());

// Можно ли сделать так?
// if (strings instanceof List<String>) { }
```

**Ответ:** `true`. В runtime оба — `ArrayList`. Информация о `<String>` и `<Integer>` стёрта. `instanceof List<String>` не скомпилируется — non-reifiable тип нельзя проверить в runtime.

**Разбор:** Стирание типов — цена обратной совместимости с Java 4. Компилятор вставляет неявные касты, а в runtime все generic-типы выглядят как raw types.
