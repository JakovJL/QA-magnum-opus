## Что такое контроль версий

**Контроль версий** — это система, которая отслеживает изменения в файлах. Она помогает команде работать над одним проектом и не терять историю.

**Цель:** хранить историю проекта, сравнивать изменения, возвращаться к старым версиям и безопасно работать в ветках.

Распространённые системы:

- `Git` — самая популярная распределённая система контроля версий
- `TFS` / Azure DevOps — старая Microsoft-экосистема и проектная платформа
- `SVN` — старая централизованная система контроля версий

## Основы Git

Git хранит изменения как commits.

| Термин | Значение |
|---|---|
| repository | папка проекта под контролем Git |
| working tree | текущие файлы на диске |
| staging area | место, где изменения готовятся к commit |
| commit | сохранённый снимок изменений с сообщением |
| branch | отдельная линия разработки |

Частые команды:

```bash
git status
git add file.md
git commit -m "Add test cases"
git log --oneline
git diff
```

**Пример:** перед изменением automation framework проверь `git status`, создай ветку, делай небольшие логические commits и открой pull request.

## Ветки и слияние

**Branch** позволяет работать над изменением, не мешая основной линии кода.

Частые имена веток:

- `feature/login-tests`
- `fix/flaky-wait`
- `chore/update-dependencies`

```bash
git switch -c feature/login-tests
git switch main
git merge feature/login-tests
```

**Преимущества:**
- безопасная работа над фичами и исправлениями
- удобнее code review
- понятная история по каждой задаче

**Недостатки:**
- могут возникать merge conflicts
- долгоживущие ветки тяжело сливать

> [!caution] Держи ветки короткими
> Ветку, которая живёт неделями, часто тяжело слить. Регулярно подтягивай изменения из основной ветки.

## Удалённые репозитории

**Удалённый репозиторий** — это общая копия проекта, обычно на GitHub, GitLab, Bitbucket или Azure DevOps.

Частые команды:

```bash
git clone https://example.com/project.git
git pull
git push
```

**Pull request** или **merge request** — это шаг проверки перед слиянием кода в основную ветку.

## Git в AQA

AQA-инженер использует Git, чтобы:

- хранить код автотестов
- проверять изменения в тестах
- отслеживать изменения framework
- связывать изменения с задачами или багами
- запускать CI pipelines после push или pull request

Хорошие практики:

- commit должен содержать одно логическое изменение
- пиши понятные commit messages
- не коммить secrets, tokens или passwords
- не коммить сгенерированные отчёты, если команда не договорилась хранить их
- проверяй diff перед push

## Вопросы для собеседования

### Вопросы начального уровня

**1. Что такое контроль версий?**
Контроль версий отслеживает изменения файлов и хранит историю проекта.

**2. Что такое commit?**
Commit — это сохранённый снимок изменений с сообщением.

### Вопросы среднего уровня

**3. Зачем команде ветки?**
Ветки позволяют безопасно работать над разными задачами и проверять изменения перед слиянием.

**4. Что такое pull request?**
Pull request — это запрос на проверку и слияние изменений из одной ветки в другую.

### Вопросы повышенного уровня

**5. Почему QA-инженеру нельзя коммитить secrets?**
Secrets могут дать доступ к системам. Tokens и passwords должны храниться в environment variables или secret managers.

### Вопросы с кодом

**6. Что показывает эта команда?**

```bash
git diff
```

Она показывает unstaged changes в tracked files.
