## Что такое интерфейс

**Интерфейс** — это контракт. Определяет **что** класс должен делать, но не **как**.

```java
public interface Drawable {
    void draw();
    double getArea();
}

public class Circle implements Drawable {
    private double radius;
    public Circle(double radius) { this.radius = radius; }

    @Override
    public void draw() { System.out.println("Рисуем круг"); }

    @Override
    public double getArea() { return Math.PI * radius * radius; }
}
```

> [!info] Синтаксис
> - `interface` вместо `class`
> - Методы без тела
> - Поля — только `public static final`
> - Класс использует `implements`

## Реализация интерфейсов

Класс может реализовать несколько интерфейсов.

```java
public interface Printable { void printInfo(); }
public interface Resizable { void resize(double factor); }

public class Document implements Printable, Resizable {
    @Override
    public void printInfo() { System.out.println("Документ"); }

    @Override
    public void resize(double factor) { /* ... */ }
}
```

### Интерфейс как тип

```java
Printable doc = new Document("Hello");
doc.printInfo();   // работает
```

> [!tip] Программируй на уровне интерфейсов
> `List<String> list = new ArrayList<>();` — гибко, можно переключить реализацию.

## Default и статические методы (Java 8+)

```java
public interface Vehicle {
    void start();
    default void honk() { System.out.println("Бип!"); }
}

public class Car implements Vehicle {
    @Override
    public void start() { System.out.println("Машина заведена"); }
    // honk() наследуется автоматически
}
```

## Функциональные интерфейсы

Один абстрактный метод. Основа лямбд.

```java
@FunctionalInterface
public interface Greeting {
    String greet(String name);
}

Greeting hello = name -> "Привет, " + name;
```

### Встроенные

| Интерфейс | Метод | Назначение |
|---|---|---|
| `Predicate<T>` | `boolean test(T t)` | Проверить условие |
| `Consumer<T>` | `void accept(T t)` | Сделать что-то |

## Абстрактные классы vs Интерфейсы

| Особенность | Абстрактный класс | Интерфейс |
|---|---|---|
| Поля | Любые | Только константы |
| Конструкторы | ✅ | ❌ |
| Множественное наследование | ❌ | ✅ |

### Когда что выбирать

Абстрактный класс — общее состояние + частичное поведение.
Интерфейс — контракт/способность.

## Вопросы для собеседования

### Вопросы начального уровня

**Что такое интерфейс в Java?**
Интерфейс — это ссылочный тип, определяющий контракт: какие методы класс должен реализовать, но не как. Все методы неявно `public` и `abstract`. Класс реализует интерфейс через ключевое слово `implements`.

**Может ли класс реализовать несколько интерфейсов?**
Да. Это способ множественного наследования в Java: `class MyClass implements A, B { }`. Класс должен реализовать все методы из всех интерфейсов.

**Что такое функциональный интерфейс?**
Интерфейс ровно с одним абстрактным методом. Основа лямбда-выражений. Аннотация `@FunctionalInterface` необязательна, но рекомендуется.

**Что такое default методы в интерфейсах? (Java 8+)**
Default методы — методы с телом, которые можно добавить в интерфейс без поломки существующих реализаций. Классы-реализаторы наследуют их автоматически и могут переопределить при необходимости.

**Могут ли интерфейсы иметь поля? Если да, то какие?**
Да, но неявно `public static final` — только константы. Instance-поля в интерфейсе недопустимы. Пример: `int MAX_VALUE = 100;` эквивалентно `public static final int MAX_VALUE = 100;`.

### Вопросы среднего уровня

**В чём разница между абстрактным классом и интерфейсом?**
Абстрактный класс может иметь любые поля, конструкторы и конкретные методы. Интерфейс — только `public static final` поля, без конструкторов, и (с Java 8) `default` и `static` методы. Класс может наследовать один абстрактный класс, но реализовать несколько интерфейсов.

**Чем отличается extends для интерфейсов от implements для классов?**
`extends` — интерфейс наследует другой интерфейс: `interface B extends A`. `implements` — класс выполняет контракт интерфейса: `class C implements A`.

**Зачем нужны static методы в интерфейсах? (Java 8+)**
Static методы в интерфейсах предоставляют утилитные/вспомогательные методы, относящиеся к концепции интерфейса. Вызываются через имя интерфейса: `Validator.isValidEmail(email)`. Не наследуются классами-реализаторами.

**Почему интерфейсы полезны в тестовой автоматизации?**
Позволяют отделить логику тестов от реализации. `WebDriver driver = new ChromeDriver()` — можно переключиться на Firefox или Edge без изменения тестов. Page Object через интерфейсы позволяет заменить Selenium на Playwright сменой одного класса реализации.

**Что такое проблема ромба (diamond problem) с интерфейсами? Как Java её решает?**
Если класс реализует два интерфейса с одинаковым `default` методом, класс обязан переопределить этот метод, чтобы разрешить конфликт. Иначе компилятор выдаёт ошибку и заставляет выбрать реализацию.

### Вопросы повышенного уровня

**Может ли интерфейс расширять несколько интерфейсов?**
Да. В отличие от классов, интерфейсы могут наследовать несколько: `interface C extends A, B { }`. Это безопасно, так как интерфейсы содержат только сигнатуры методов — реализации для конфликта нет.

**Может ли default метод в интерфейсе быть final?**
Нет. Default методы созданы для переопределения классами-реализаторами. `final` противоречит их назначению, компилятор не разрешит.

**Что будет, если класс реализует два интерфейса с одинаковым default методом?**
Класс обязан переопределить метод. Иначе — ошибка компиляции (diamond problem). Класс должен предоставить свою реализацию или явно вызвать один из родительских: `InterfaceA.super.methodName()`.

**Можно ли создать экземпляр интерфейса напрямую через new?**
Нет. Интерфейсы нельзя инстанциировать. Но можно создать анонимный класс или лямбду (для функциональных интерфейсов), которые реализуют интерфейс на лету.

**Могут ли интерфейсы иметь private методы? (Java 9+)**
Да. Private методы в интерфейсах (Java 9+) используются для общего кода между `default` методами внутри того же интерфейса. Не видны реализующим классам.

### Вопросы с кодом

```java
public interface Vehicle {
    void start();
    default void honk() { System.out.println("Бип!"); }
}

public class Car implements Vehicle {
    @Override
    public void start() { System.out.println("Машина заведена"); }
}

Vehicle v = new Car();
v.start();
v.honk();
```
**Что выведет этот код?**
**Ответ:**
```
Машина заведена
Бип!
```
`Car` реализует `start()`, а `default` метод `honk()` наследуется автоматически от интерфейса, без явного определения в `Car`.

---

```java
@FunctionalInterface
public interface Greeting {
    String greet(String name);
}

Greeting hello = name -> "Привет, " + name;
System.out.println(hello.greet("Alice"));
```
**Что выведет код и почему лямбда работает?**
**Ответ:** Выводит `Привет, Alice`. У `Greeting` ровно один абстрактный метод, значит это функциональный интерфейс, поэтому лямбда `name -> "Привет, " + name` задаёт тело метода `greet(String)`.

---

```java
public interface Printable { void printInfo(); }
public interface Resizable { void resize(double factor); }

public class Document implements Printable, Resizable {
    @Override
    public void printInfo() { System.out.println("Документ"); }
    // resize() не реализован
}
```
**Скомпилируется ли это? Почему?**
**Ответ:** Нет. `Document` заявляет, что реализует `Resizable`, но не предоставляет метод `resize(double)`. Конкретный класс обязан реализовать все абстрактные методы всех своих интерфейсов — или быть объявленным `abstract`.
