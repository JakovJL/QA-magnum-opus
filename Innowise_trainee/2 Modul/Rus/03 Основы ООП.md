## Классы и объекты

**Класс** — это шаблон. **Объект** — экземпляр класса.

```java
public class Car {
    String brand;
    String color;
    int year;
    void startEngine() {
        System.out.println("Двигатель " + brand + " запущен");
    }
}

Car myCar = new Car();
myCar.brand = "Toyota";
myCar.startEngine();
```

### Анатомия класса

| Часть | Что делает |
|---|---|
| **Поля** | Хранят состояние объекта |
| **Конструкторы** | Инициализируют объект при создании |
| **Методы** | Определяют поведение |
| **Статические члены** | Принадлежат классу, а не объекту |

## Конструкторы

```java
public class Student {
    String name;
    int age;

    public Student(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public Student() {
        this("Неизвестно", 0);   // цепочка конструкторов
    }
}

Student s = new Student("Alice", 20);
```

> [!tip] Цепочка конструкторов
> `this(...)` вызывает один конструктор из другого. Должен быть первой строкой.

## Ключевое слово this

`this` ссылается на текущий объект.

```java
public class Employee {
    String name;
    public Employee(String name) {
        this.name = name;   // this.name = поле, name = параметр
    }
}
```

**Три用途:** 1) различить поле и параметр 2) вызвать другой конструктор 3) вернуть текущий объект.

## Статические члены (static)

Принадлежат классу, а не объектам. Загружаются **один раз** при первом обращении к классу.

### Static-поля

```java
public class Counter {
    static int totalCount = 0;   // общее для всех объектов
    int instanceCount = 0;       // своё для каждого объекта

    public Counter() {
        totalCount++;
        instanceCount++;
    }
}

Counter c1 = new Counter();  // totalCount = 1, instanceCount = 1
Counter c2 = new Counter();  // totalCount = 2, instanceCount = 1
System.out.println(Counter.totalCount);   // 2 — обращение через класс
```

### Static-методы

```java
public class MathUtils {
    public static int add(int a, int b) {
        return a + b;
    }
}

int sum = MathUtils.add(5, 3);   // 8 — вызов без создания объекта
```

> [!info] Static vs Instance
> - Static-методы могут обращаться **только** к static-полям напрямую
> - Instance-методы могут обращаться и к static, и к instance полям
> - `this` нельзя использовать в static-методах (нет объекта)

### Static-блоки

Static-блок выполняется **один раз** при загрузке класса — до создания любых объектов.

```java
public class Config {
    static String dbUrl;
    static Properties props;

    static {
        System.out.println("Загрузка конфигурации...");
        dbUrl = "jdbc:mysql://localhost:3306/app";
        // props.load(...) — загрузка файла
    }
}
```

**Используется для:** инициализации сложных static-полей, загрузки конфигов, подключения к БД.

### Порядок инициализации

Когда JVM впервые загружает класс, всё выполняется **строго сверху вниз**:

```java
public class Demo {
    static int a = 1;          // 1. static поля
    static {                   // 2. static блоки
        System.out.println("static: a=" + a + ", b=" + b);
    }
    static int b = 2;          // 3. static поля (продолжение)
    int x = 3;                 // 4. instance поля
    {                          // 5. instance блоки
        System.out.println("instance: x=" + x);
    }
    public Demo() {            // 6. конструктор
        System.out.println("constructor");
    }
}

// При new Demo():
// static: a=1, b=0   ← b ещё не инициализирован!
// instance: x=3
// constructor
```

**Общий порядок загрузки объекта:**
```
1. static поля (сверху вниз)
2. static блоки (сверху вниз)
3. instance поля (сверху вниз)
4. instance блоки (сверху вниз)
5. конструктор
```

### При наследовании

```
1. static родителя (все static поля + блоки)
2. static потомка  (все static поля + блоки)
3. instance родителя → конструктор родителя
4. instance потомка → конструктор потомка
```

```java
class Parent {
    static { System.out.println("Parent static"); }
    { System.out.println("Parent instance"); }
    Parent() { System.out.println("Parent constructor"); }
}

class Child extends Parent {
    static { System.out.println("Child static"); }
    { System.out.println("Child instance"); }
    Child() { System.out.println("Child constructor"); }
}

// new Child() выведет:
// Parent static
// Child static
// Parent instance
// Parent constructor
// Child instance
// Child constructor
```

> [!warning] Важно
> Проверяемые исключения внутри static-блоков должны быть пойманы внутри блока. Непроверяемые исключения оборачиваются в `ExceptionInInitializerError`.
> Порядок полей и блоков в коде **имеет значение** — JVM выполняет строго сверху вниз.

## equals() и hashCode()

По умолчанию `equals()` из `Object` сравнивает **ссылки** (как `==`). Если нужно сравнивать **содержимое**, его надо переопределить.

### Зачем переопределять equals()?

```java
User a = new User("Alice", 25);
User b = new User("Alice", 25);

System.out.println(a == b);       // false — разные объекты
System.out.println(a.equals(b));  // false — equals() по умолчанию использует ==
```

После переопределения:

```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (o == null || getClass() != o.getClass()) return false;
    User user = (User) o;
    return age == user.age && Objects.equals(name, user.name);
}
```

Теперь `a.equals(b)` вернёт `true`, потому что содержимое совпадает.

### Зачем переопределять hashCode() тоже?

**Контракт** гласит: если два объекта равны (`equals()` возвращает `true`), у них должен быть одинаковый `hashCode()`. Это требование для `HashMap`, `HashSet` и любых hash-коллекций.

```java
// Без переопределения hashCode():
Set<User> users = new HashSet<>();
users.add(new User("Alice", 25));
users.contains(new User("Alice", 25));  // false! — разный hashCode

// С правильным hashCode():
@Override
public int hashCode() {
    return Objects.hash(name, age);
}
// Теперь contains() вернёт true
```

**Что сломается, если переопределить только equals():**
- `HashMap` использует `hashCode()` для поиска корзины (bucket). Если у двух равных объектов разные хеш-коды, map положит их в разные корзины и никогда не найдёт один по другому.
- `HashSet` работает по тому же принципу — "равные" объекты выглядят как дубликаты.

> [!danger] Контракт
> 1. Если `a.equals(b)` — `true`, то `a.hashCode() == b.hashCode()` обязан быть `true`
> 2. Если хеш-коды равны, `equals()` может быть и `true`, и `false` (коллизии допустимы)
> 3. Если `a.equals(b)` — `false`, хеш-коды *могут* совпадать (но лучше бы отличались для производительности)

> [!tip] IntelliJ генерирует оба
> `Alt + Insert` → **equals() and hashCode()** → выбрать поля. Всегда генерируй их вместе.

## Вопросы для собеседования

### Вопросы начального уровня

**В чём разница между классом и объектом?**
Класс — это шаблон (чертёж), описывающий структуру (поля) и поведение (методы) чего-либо. Объект — это конкретный экземпляр класса, созданный с помощью ключевого слова `new`. Один класс может порождать множество независимых объектов.

**Что такое конструктор? Может ли он возвращать значение?**
Конструктор — это специальный метод, инициализирующий объект при его создании через `new`. Он имеет то же имя, что и класс, и не имеет возвращаемого типа (даже `void`). Он не может возвращать значение.

**Что такое инкапсуляция? Зачем она нужна?**
Инкапсуляция — это сокрытие внутренних данных и предоставление доступа только через публичные методы. Она защищает состояние объекта от некорректных значений и снижает связанность между компонентами. Достигается пометкой полей как `private` и предоставлением публичных геттеров/сеттеров.

**Какие модификаторы доступа есть в Java? Перечислите от самого строгого к самому мягкому.**
`private` (только класс), default/package-private (только пакет), `protected` (пакет + наследники), `public` (везде).

**Что такое метод `toString()` и зачем его переопределять?**
`toString()` наследуется из `Object` и возвращает строковое представление по умолчанию (имя класса + хэш-код). Переопределение даёт содержательную информацию о состоянии объекта, что полезно для отладки и логирования.

**Что такое статическая константа? Как она объявляется?**
Статическая константа — это значение уровня класса, которое нельзя изменить. Объявляется как `public static final ТИП ИМЯ = значение;` — например, `public static final int MAX_SIZE = 100;`. Соглашение об именах — UPPER_SNAKE_CASE.

### Вопросы среднего уровня

**Что такое перегрузка конструкторов и цепочка конструкторов?**
Перегрузка конструкторов — наличие нескольких конструкторов с разными списками параметров. Цепочка конструкторов — вызов одного конструктора из другого через `this(...)` — это избавляет от дублирования кода инициализации.

**На что ссылается ключевое слово `this`?**
`this` ссылается на текущий объект — объект, чей метод или конструктор вызывается. Используется, чтобы различить поля и параметры (`this.name = name`), вызвать другой конструктор (`this(params)`) и вернуть текущий объект.

**В чём разница между `static` и instance-членами?**
Static-члены принадлежат самому классу и общие для всех экземпляров. Доступ к ним — через имя класса (`ClassName.field`). Instance-члены принадлежат каждому отдельному объекту и требуют экземпляра для доступа. Static-методы не могут напрямую обращаться к instance-полям.

**В чём смысл геттеров и сеттеров?**
Геттеры предоставляют контролируемое чтение приватных полей. Сеттеры предоставляют контролируемую запись — они могут валидировать данные, выполнять преобразования или запускать побочные эффекты. Без сеттеров прямой доступ к полям позволил бы задать некорректные значения вроде `user.age = -5`.

**В чём разница между `==` и `.equals()` для объектов?**
`==` сравнивает ссылки (указывают ли две переменные на один объект в памяти). `.equals()`, унаследованный от `Object`, по умолчанию тоже сравнивает ссылки — но при переопределении сравнивает содержимое. Поэтому `new User("a") == new User("a")` даёт `false`, а при правильном переопределении `equals()` `.equals()` вернёт `true`.

**Зачем переопределять `hashCode()`, если переопределяешь `equals()`?**
Контракт гласит: у равных объектов должен быть одинаковый хэш-код. `HashMap` и `HashSet` используют `hashCode()` для поиска корзины (bucket). Если у двух равных объектов разные хэш-коды, коллекция положит их в разные корзины и никогда не найдёт один через другой — `contains()` вернёт `false` даже для равного объекта.

### Вопросы повышенного уровня

**Что будет, если не написать ни одного конструктора в классе?**
Java автоматически добавит конструктор по умолчанию без параметров, который ничего не делает (`public ClassName() { }`). Он инициализирует все поля значениями по умолчанию (0 для чисел, null для объектов).

**Что будет, если написать хотя бы один конструктор? Сохранится ли конструктор по умолчанию?**
Нет. Как только ты определяешь любой конструктор, конструктор по умолчанию больше НЕ создаётся автоматически. Если безаргументный конструктор всё ещё нужен, его придётся написать явно. Это частая ошибка — наследование класса, у родителя которого только параметризованный конструктор, вызывает ошибку компиляции.

**Может ли static-метод обращаться к нестатическим полям? Почему?**
Нет. Static-метод не принадлежит какому-либо конкретному объекту. Поскольку нестатические поля требуют существования объекта, у static-метода нет объекта для доступа. Ошибка компиляции: "Cannot make a static reference to the non-static field".

**Можно ли использовать `this` внутри static-метода?**
Нет. `this` ссылается на текущий объект. Static-методы не связаны ни с каким объектом, поэтому `this` в них недоступен.

**Каков порядок инициализации при загрузке класса и создании объекта?**
Static-поля и блоки выполняются первыми, строго сверху вниз, при загрузке класса (один раз). Затем для каждого `new`: instance-поля, instance-блоки и наконец конструктор — все сверху вниз. При наследовании: static родителя до static потомка, затем instance+конструктор родителя, затем instance+конструктор потомка.

**Что будет, если внутри static-блока выбрасывается проверяемое исключение?**
Проверяемые исключения внутри static-блоков должны быть пойманы внутри блока. Непроверяемые исключения оборачиваются в `ExceptionInInitializerError`.

### Вопросы с кодом

```java
public class Counter {
    static int totalCount = 0;
    int instanceCount = 0;

    public Counter() {
        totalCount++;
        instanceCount++;
    }
}

Counter c1 = new Counter();
Counter c2 = new Counter();
System.out.println(Counter.totalCount);
System.out.println(c1.instanceCount);
System.out.println(c2.instanceCount);
```
**Что выведет этот код?**
**Ответ:** `2`, затем `1`, затем `1`. `totalCount` — static и общий, поэтому доходит до 2. `instanceCount` — свой для каждого объекта, поэтому у каждого экземпляра своя копия, равная 1.

```java
User a = new User("Alice", 25);
User b = new User("Alice", 25);
System.out.println(a == b);
System.out.println(a.equals(b));
```
**Предположим, `equals()` НЕ переопределён. Что выведет код? Что изменится после переопределения?**
**Ответ:** Без переопределения и `==`, и `equals()` используют сравнение по ссылке, поэтому `false` и `false`. После переопределения `equals()` для сравнения `name` и `age` — `a == b` по-прежнему `false`, но `a.equals(b)` становится `true`.

```java
public class Demo {
    static int a = 1;
    static { System.out.println("static: b=" + b); }
    static int b = 2;

    public static void main(String[] args) {
        new Demo();
    }
}
```
**Что выведет код и почему?**
**Ответ:** `static: b=0`. Static-поля и блоки выполняются сверху вниз при загрузке класса. Блок выполняется до того, как `b` присваивается значение 2, поэтому `b` всё ещё содержит значение по умолчанию `0`.
