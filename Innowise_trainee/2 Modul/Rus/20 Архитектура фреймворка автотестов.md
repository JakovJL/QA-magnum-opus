## Что такое framework

**Automation framework** — это структурированный проект для написания, запуска и поддержки автотестов.

**Цель:** сделать тесты читаемыми, переиспользуемыми, стабильными и удобными для запуска локально или в CI.

Framework обычно включает:

- test runner
- page или API clients
- assertions
- configuration
- test data
- logging и reporting
- utilities

---

## Основные слои

Обычный UI framework имеет несколько слоёв.

| Слой | Ответственность |
|---|---|
| tests | описывают test scenarios |
| pages / components | прячут UI locators и UI actions |
| services / clients | вызывают API или backend helpers |
| config | browser, URL, timeouts, environment |
| utils | маленькие переиспользуемые helpers |
| reporting | logs, screenshots, Allure steps |

**Пример:** тест должен говорить `loginPage.loginAs(user)`, а не повторять locators и низкоуровневые clicks везде.

---

## Конфигурация и тестовые данные

Configuration управляет тем, как запускаются тесты.

Примеры:

- base URL
- browser
- timeout
- environment name
- источник API token

Test data могут приходить из:

- builders
- factories
- JSON files
- databases
- API setup steps

> [!caution] Не хардкодь environment secrets
> Passwords и tokens должны приходить из environment variables или CI secrets, а не из source code.

## Page Object Model

**Page Object Model (POM)** — это pattern, где у каждой важной страницы или component есть свой class.

**Цель:** хранить locators и page actions в одном месте.

Преимущества:

- меньше duplication
- проще поддержка при изменении UI
- более читаемые tests

Недостатки:

- плохие page objects могут стать слишком большими
- business assertions не стоит прятать слишком глубоко

→ см. [[17 Основы Selenium WebDriver]] для основ Selenium и примеров POM.

---

## Правила хорошей архитектуры

Хороший framework code:

- отделяет test intent от технических деталей
- оставляет одну причину для изменения class
- избегает duplicated locators и waits
- использует понятные имена
- падает с полезными error messages
- сохраняет reports и logs для debugging

Простая структура:

```text
src/test/java
  tests/
  pages/
  config/
  data/
  utils/
```

## Вопросы для собеседования

### Вопросы начального уровня

**1. Что такое automation framework?**
Это структурированный проект для автотестов.

**2. Зачем нужны слои?**
Слои разделяют test logic, UI actions, configuration и utilities.

### Вопросы среднего уровня

**3. Какую проблему решает POM?**
POM хранит locators и page actions в одном месте, поэтому tests проще поддерживать.

**4. Где должны храниться настройки окружения?**
В configuration files, environment variables или CI secrets.

### Вопросы повышенного уровня

**5. Что делает framework сложным в поддержке?**
Duplicated locators, hidden waits, hardcoded data, большие classes и непонятные failures.

### Вопросы с кодом

**6. Какая строка лучше для теста и почему?**

```java
loginPage.loginAs(user);
driver.findElement(By.id("login")).sendKeys(user.login());
```

Первая строка лучше в тесте, потому что показывает intent и прячет низкоуровневые UI details в page object.
