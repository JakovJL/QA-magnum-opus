## Что такое MongoDB

**MongoDB** — документо-ориентированная база данных. Она хранит данные в документах, похожих на JSON; внутри MongoDB используется формат BSON.

**Цель:** удобно хранить данные с гибкой или вложенной структурой.

MongoDB не требует одинакового набора полей у каждого документа, но в production-проекте правила структуры всё равно должны быть определены контрактом или валидацией.

---

## Документы и коллекции

- **Document** — один объект данных, похожий на JSON-объект;
- **Collection** — набор документов одного типа;
- **Database** — набор коллекций;
- `_id` — уникальный идентификатор документа.

```json
{
  "_id": "user-42",
  "name": "Ana",
  "roles": ["customer"],
  "address": {"city": "Minsk"}
}
```

В реляционной базе адрес и роли часто находятся в отдельных таблицах. В MongoDB их можно сохранить внутри одного документа, если они обычно читаются и изменяются вместе.

## Запросы и индексы

Запрос ищет документы по полям и условиям:

```javascript
db.users.find({ name: "Ana", roles: "customer" })
```

**Индекс** ускоряет поиск и сортировку, но замедляет запись и занимает место. Индексы нужны для частых запросов, особенно по полям фильтрации и сортировки.

> [!caution] Индекс не заменяет проверку запроса
> Даже быстрый запрос может вернуть неправильные данные, если фильтр, тип поля или вложенный путь указаны неверно.

---

## Вложенные данные

MongoDB поддерживает два основных подхода:

- **вложение** — хранить связанные данные внутри документа;
- **ссылка** — хранить идентификатор другого документа.

Вложение удобно для небольших данных, которые используются вместе. Ссылка лучше, когда данные большие, часто переиспользуются или меняются независимо.

---

## Что проверять тестировщику

- создан ли документ после API-запроса или события;
- правильно ли сохранены типы: число, строка, дата, массив, вложенный объект;
- обновляются ли только нужные поля;
- не создаётся ли дубликат при повторной операции;
- корректно ли работают фильтрация, сортировка и пагинация;
- есть ли индекс для критичного запроса, если это требование производительности.

## Типичные ошибки

| Ошибка | Как сделать лучше |
|---|---|
| Сравнивать весь документ с динамическими полями | Проверять значимые поля и типы |
| Не учитывать вложенные объекты и массивы | Добавлять отдельные проверки для их структуры |
| Считать отсутствие схемы отсутствием правил | Проверять контракт и валидацию данных |
| Использовать один тестовый набор для всех сценариев | Изолировать и очищать тестовые данные |
