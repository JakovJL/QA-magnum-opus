## Почему сборочные инструменты важны

Java-проекту обычно нужен не только исходный код. Ему также нужны:

- внешние библиотеки
- тестовые зависимости
- компиляция
- запуск тестов
- упаковка
- плагины для отчётов и качества кода

Делать всё это вручную было бы медленно и легко приводило бы к ошибкам. **Сборочные инструменты** автоматизируют такие задачи.

Два самых популярных сборочных инструмента в Java:

- `Maven`
- `Gradle`

---

## Какие задачи решают Maven и Gradle

Оба инструмента помогают управлять проектом структурированно.

Они умеют:

- скачивать зависимости из репозиториев
- компилировать исходный код
- запускать тесты
- упаковывать проект в JAR/WAR
- запускать плагины вроде Allure или Checkstyle

Проще говоря, build tool отвечает на вопрос: "Как собрать, протестировать и запустить проект повторяемым способом?"

### Частые термины

| Термин | Смысл |
|---|---|
| dependency | внешняя библиотека, нужная проекту |
| plugin | дополнительный инструмент, который build system умеет запускать |
| repository | место, откуда скачиваются зависимости |
| build lifecycle | упорядоченные шаги процесса сборки |

## Управление пакетами и зависимостями

**Package manager** скачивает libraries, разрешает их versions и делает одинаковые dependencies доступными для всей команды.

В Java AQA projects Maven и Gradle работают и как build tools, и как dependency managers.

Они помогают:

- скачивать Selenium, JUnit, REST Assured и Allure
- хранить dependency versions в одном месте
- отделять test dependencies от production dependencies
- делать local и CI runs повторяемыми

**Пример:** если проект использует JUnit 5, dependency описывается в `pom.xml` или `build.gradle`. Каждый разработчик и CI runner получает одну и ту же library из repository.

> [!tip] Идея package manager
> Не храни внешние `.jar` files вручную в проекте. Описывай dependencies в Maven или Gradle, чтобы build был повторяемым.

## Основы Maven

**Maven** — это декларативный build tool. Ты описываешь проект в `pom.xml`, а Maven следует стандартному lifecycle.

### Главный файл

```xml
<project>
    <modelVersion>4.0.0</modelVersion>
    <groupId>com.example</groupId>
    <artifactId>demo</artifactId>
    <version>1.0.0</version>
</project>
```

### Важные координаты

- `groupId` - группа проекта, часто в стиле компании или package
- `artifactId` - имя проекта
- `version` - версия проекта

Вместе они уникально определяют артефакт.

### Пример зависимостей

```xml
<dependencies>
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <version>5.10.2</version>
        <scope>test</scope>
    </dependency>
</dependencies>
```

### Частые Maven-команды

```bash
mvn clean
mvn compile
mvn test
mvn package
```

> [!tip] Идея Maven
> Maven любит convention over configuration. Если проект следует стандартной структуре, часто нужно меньше кастомной настройки.

## Основы Gradle

**Gradle** — это сборочный инструмент, который более гибок и построен вокруг скриптов. Обычно используются `build.gradle` или `build.gradle.kts`.

### Простой пример

```groovy
plugins {
    id 'java'
}

repositories {
    mavenCentral()
}

dependencies {
    testImplementation 'org.junit.jupiter:junit-jupiter:5.10.2'
}
```

### Частые Gradle-команды

```bash
gradle clean
gradle test
gradle build
```

Часто проекты используют wrapper:

```bash
./gradlew test
gradlew.bat test
```

Wrapper полезен тем, что фиксирует одинаковую версию Gradle для всей команды.

### Почему людям нравится Gradle

- больше гибкости
- мощный scripting
- хорошо подходит для сложных сборок
- популярен во многих современных проектах

> [!info] Groovy и Kotlin DSL
> Скрипты Gradle могут быть написаны на Groovy (`build.gradle`) или Kotlin DSL (`build.gradle.kts`).

## Зависимости плагины и жизненный цикл

### Зависимости

Зависимости — это внешние библиотеки, которые нужны проекту.

Примеры в AQA:

- JUnit
- Selenium
- REST Assured
- Allure
- Lombok

Build tool скачивает их из репозиториев вроде Maven Central.

### Плагины

Плагины расширяют процесс сборки.

Примеры:

- Surefire/Failsafe для тестов в Maven
- Allure plugin
- compiler plugin
- JaCoCo для coverage

### Maven Lifecycle

У Maven есть стандартный упорядоченный lifecycle.

Частые фазы:

- `validate`
- `compile`
- `test`
- `package`
- `verify`
- `install`
- `deploy`

Если ты запускаешь `mvn test`, Maven сначала выполнит все необходимые более ранние фазы.

### Gradle Tasks

Gradle работает с **tasks**, а не с такой же жёсткой идеей lifecycle.

Примеры:

- `clean`
- `compileJava`
- `test`
- `build`

Сборка в Gradle — это граф задач с зависимостями между ними.

> [!warning] Scope зависимостей важен
> В Maven scope вроде `test`, `compile` и `provided` влияет на то, где зависимость будет доступна. Например, тестовая зависимость не должна быть нужна в production runtime.

## Maven vs Gradle

Оба инструмента решают похожие задачи, но стиль у них разный.

> [!tip] Быстрое сравнение
> | Инструмент | Сильная сторона | Типичный компромисс |
> |---|---|---|
> | Maven | простой, стандартизованный, предсказуемый | меньше гибкости |
> | Gradle | гибкий, мощный, скриптуемый | может стать сложнее |

### Maven

- XML-конфигурация
- более сильные соглашения
- легче читать в стандартных проектах
- очень распространён в backend и enterprise-проектах

### Gradle

- конфигурация через скрипты
- проще глубоко кастомизировать
- часто быстрее в сложных сборках
- распространён в современных Java- и Android-проектах

На собеседовании хороший короткий ответ обычно такой:

- Maven проще и более стандартизован
- Gradle гибче и программируемее

## Сборочные инструменты в AQA

Build tools очень важны для automation projects.

### Что они обычно управляют

- зависимостями Selenium
- JUnit или TestNG
- REST Assured
- Allure-отчётами
- запуском smoke/regression suites
- командами для CI

### Maven-пример для AQA

```xml
<dependencies>
    <dependency>
        <groupId>org.seleniumhq.selenium</groupId>
        <artifactId>selenium-java</artifactId>
        <version>4.22.0</version>
    </dependency>
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <version>5.10.2</version>
        <scope>test</scope>
    </dependency>
</dependencies>
```

### Почему это важно в AQA

- одной командой можно запустить весь test suite
- CI может запускать ту же команду на каждом билде
- новым разработчикам проще стартовать в проекте
- версии зависимостей остаются под контролем

### Хорошие практики

- держи версии зависимостей понятными
- убирай неиспользуемые зависимости
- используй wrapper в Gradle-проектах
- избегай случайных ручных jar-файлов в проекте
- держи build-конфиг читаемым

> [!caution] Version conflicts
> Если две библиотеки требуют разные версии одной и той же зависимости, проект может скомпилироваться, но упасть в runtime. Управление зависимостями — одна из главных причин, почему build tools так важны.

## Вопросы для собеседования

### Вопросы начального уровня

**1. Что такое build tool в Java?**
Build tool автоматизирует задачи вроде управления зависимостями, компиляции, тестирования, упаковки и запуска плагинов.

**2. Что такое Maven?**
Maven — это декларативный Java build tool, основанный на соглашениях и файле `pom.xml`.

**3. Что такое Gradle?**
Gradle — это гибкий build tool, который использует скрипты и задачи для описания сборки.

**4. Что такое dependency?**
Это внешняя библиотека, которая нужна проекту, например JUnit или Selenium.

**5. Что такое `pom.xml`?**
Это главный файл конфигурации Maven, где объявляются координаты проекта, зависимости, плагины и другие настройки.

**6. Что такое plugin в build tool?**
Плагин расширяет возможности сборки, например запуском тестов, генерацией отчётов или проверкой качества кода.

---

### Вопросы среднего уровня

**1. В чём разница между Maven и Gradle?**
Maven более стандартизован и основан на соглашениях. Gradle более гибкий и скриптуемый.

**2. Что такое Gradle wrapper?**
Это скрипт, который поставляется вместе с проектом и запускает Gradle нужной версии, чтобы разработчикам не приходилось ставить точную версию вручную.

**3. Почему build tools важны в AQA?**
Они управляют automation-зависимостями, запуском тестов, отчётами и интеграцией с CI.

**4. Что происходит при запуске `mvn test`?**
Maven выполняет все нужные предыдущие фазы lifecycle и затем запускает фазу тестов.

---

### Вопросы повышенного уровня

**1. Почему Maven часто проще для начинающих?**
Потому что у него сильные соглашения и предсказуемая структура, значит решений приходится принимать меньше.

**2. Почему Gradle может быть мощнее в сложных проектах?**
Потому что он более программируемый и гибкий, значит сложные workflow легче описывать.

**3. Чем плохи неиспользуемые зависимости?**
Они увеличивают размер сборки, могут замедлять скачивание, создавать конфликты и делают проект сложнее для поддержки.

**4. Почему лучше использовать зависимости из репозиториев, а не локальные jar-файлы?**
Потому что зависимости из репозиториев версионируются, воспроизводимы, удобны для CI и проще управляются всей командой.

---

### Вопросы с кодом

**1. Какова роль выделенного `<scope>` в этом фрагменте `pom.xml`?**

```xml
<dependencies>
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <version>5.10.2</version>
        <scope>test</scope>
    </dependency>
</dependencies>
```

**Ответ:** `<scope>test</scope>` означает, что JUnit доступен только на test classpath и не попадает в финальную сборку. Так Maven убирает тестовые библиотеки из production runtime.

**2. Эта конфигурация Gradle объявляет зависимость. Что означает `testImplementation` по сравнению с Maven `compile`?**

```groovy
dependencies {
    testImplementation 'org.junit.jupiter:junit-jupiter:5.10.2'
}
```

**Ответ:** `testImplementation` — это аналог Maven `<scope>test</scope>` в Gradle: зависимость доступна только тестовому коду, а не основному коду или финальной сборке.

**3. Разработчик запускает `mvn test`, но проект ещё ни разу не компилировался. Упадёт ли команда?**

```bash
mvn test
```

**Ответ:** Нет. Maven следует стандартному lifecycle и выполняет все предыдущие фазы перед `test`, поэтому он сначала компилирует код, а потом запускает тесты.
