## Почему String особенный

В Java `String` — не примитивный тип, это класс. Но язык обращается с ним иначе, чем с другими классами:

- Можно создать String без `new`: `String s = "hello";`
- Оператор `+` перегружен для конкатенации строк
- Строковые литералы автоматически **интернируются** (хранятся в специальном пуле памяти)
- Строки **неизменяемы** — после создания их нельзя изменить

Эти особенности существуют потому, что строки — самый часто используемый тип данных. Практически каждая программа работает с текстом — пользовательский ввод, пути к файлам, URL, тестовые данные, логи. Java активно оптимизирует строки именно поэтому.

> [!danger] Ключевое правило
> `String` — это `final` класс. Его нельзя наследовать. Каждая "модификация" создаёт новый объект String. Это сделано намеренно, а не является ограничением.

## String Pool

Когда ты пишешь строковый литерал в коде, Java не всегда создаёт новый объект. Сначала она проверяет специальную область памяти — **String Pool** (пул строк).

```java
String a = "hello";    // создаёт "hello" в пуле
String b = "hello";    // переиспользует ТОТ ЖЕ объект из пула
String c = new String("hello");  // создаёт НОВЫЙ объект в куче

System.out.println(a == b);     // true  — один объект в пуле
System.out.println(a == c);     // false — разные объекты
System.out.println(a.equals(c)); // true — одинаковое содержимое
```

**Как это работает:**
1. `"hello"` (литерал) -> JVM проверяет пул. Не найден? Создаёт там. Найден? Возвращает существующую ссылку.
2. `new String("hello")` -> Всегда создаёт новый объект в куче, даже если `"hello"` уже есть в пуле.

### Зачем нужен пул?

Строки появляются повсюду в коде — имена классов, методов, значения конфигов. Многие из них повторяются. Без пула каждый `"hello"` создавал бы отдельный объект. Пул экономит память, переиспользуя идентичные строки.

```java
// Можно вручную добавить строку в пул:
String c = new String("hello").intern();
System.out.println(a == c);   // true — intern() возвращает версию из пула
```

> [!tip] Практическое правило
> Используй литералы (`"hello"`) когда возможно. Используй `new String()` только когда явно нужен отдельный объект (что почти никогда не нужно). Всегда сравнивай через `.equals()`, не через `==`.

### Свёртка констант на этапе компиляции

Когда компилятор видит выражение, состоящее **только из литералов и `final`-констант**, он вычисляет его на этапе компиляции и сохраняет единый результат в пуле. Поэтому `"ab" + "c"` превращается в ту же пуловую строку `"abc"`, что и литерал `"abc"`:

```java
System.out.println("abc" == "ab" + "c");   // true — свёрнуто при компиляции
```

Но если часть выражения — **переменная** (не константа времени компиляции), конкатенация происходит в runtime и создаёт новый объект:

```java
String a = "ab";
System.out.println("abc" == a + "c");       // false — конкатенация в runtime
```

Именно поэтому `==` для строк ненадёжен: результат зависит от того, как строки были получены, а не только от их содержимого.

## Неизменяемость

Объект `String` нельзя изменить после создания. Каждый метод, который кажется "изменяющим" строку, на самом деле возвращает **новый** String.

```java
String name = "Alice";
String upper = name.toUpperCase();

System.out.println(name);    // "Alice"  — оригинал не изменился
System.out.println(upper);   // "ALICE"  — новый объект
```

Это значит, что конкатенация строк в цикле создаёт множество временных объектов:

```java
// ПЛОХО — создаёт ~1000 временных объектов String
String result = "";
for (int i = 0; i < 1000; i++) {
    result = result + i;   // каждый + создаёт новый String
}

// ХОРОШО — один StringBuilder, без временных объектов
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i);
}
String result = sb.toString();
```

### Почему String неизменяем?

Это намеренное решение, и причины важны:

1. **String Pool работает только благодаря неизменяемости.** Если `a` и `b` указывают на один `"hello"` в пуле и кто-то мог бы изменить `a` на `"world"`, то `b` тоже стал бы `"world"`. Неизменяемость это предотвращает.
2. **Потокобезопасность.** Несколько потоков могут использовать один и тот же String без синхронизации.
3. **Безопасность.** Строки хранят чувствительные данные — имена классов, пути к файлам, URL баз данных. Если бы строку можно было изменить после проверки безопасности, проверка была бы бессмысленной.
4. **Кеширование хеш-кода.** String кеширует свой `hashCode()` при первом вызове. Это делает поиск в HashMap по строковым ключам очень быстрым.

> [!warning] Частая ошибка
> `name.toUpperCase();` ничего не делает, если не присвоить результат. Оригинальный `name` не изменился. Нужно писать `name = name.toUpperCase();`.

## StringBuilder и StringBuffer

Когда нужно многократно строить или изменять текст, используй `StringBuilder`. Это **изменяемая** последовательность символов — модификации происходят на месте, без создания новых объектов.

```java
StringBuilder sb = new StringBuilder("Hello");
sb.append(" World");      // "Hello World"
sb.insert(5, ",");        // "Hello, World"
sb.replace(0, 5, "Hi");  // "Hi, World"
sb.delete(2, 4);          // "Hiorld"  — удаляет индексы 2..3
sb.reverse();              // "dlroiH"

String result = sb.toString();  // преобразование обратно в String
```

### StringBuilder vs StringBuffer

| Свойство | StringBuilder | StringBuffer |
|---|---|---|
| Потокобезопасность | Нет | Да (synchronized) |
| Скорость | Быстрее | Медленнее (из-за синхронизации) |
| Когда использовать | Однопоточный код (99% случаев) | Многопоточный код с общим буфером |

> [!tip] Выбор по умолчанию
> Всегда используй `StringBuilder`. Используй `StringBuffer` только когда несколько потоков пишут в один буфер — что крайне редко на практике.

### Когда что использовать

| Ситуация | Используй |
|---|---|
| Простая конкатенация (`a + b`) | `String` (компилятор оптимизирует) |
| Конкатенация в цикле | `StringBuilder` |
| Построение сложного текста (HTML, SQL, отчёты) | `StringBuilder` |
| Только чтение текста, параметры методов | `String` |

## Полезные методы String

Эти методы постоянно встречаются в автотестировании — валидация текста, извлечение данных, сравнение результатов.

### Проверка содержимого

```java
String text = "Test Case #42: Login Validation";

text.isEmpty()               // false
text.isBlank()               // false (Java 11+ — проверяет и пробелы)
text.contains("Login")       // true
text.startsWith("Test")      // true
text.endsWith("Validation")  // true
text.matches(".*#\\d+.*")   // true (regex-совпадение)
```

### Извлечение и преобразование

```java
String text = "  Hello, World!  ";

text.trim()                  // "Hello, World!" — убирает пробелы по краям
text.strip()                 // "Hello, World!" — Java 11+, работает с Unicode-пробелами
text.substring(2, 7)         // "llo, " — от индекса 2 до 6 (конец не включается)
text.charAt(7)               // 'W'
text.indexOf("World")        // 9
text.toLowerCase()           // "  hello, world!  "
text.toUpperCase()           // "  HELLO, WORLD!  "
text.replace("World", "Java") // "  Hello, Java!  "
```

### Разбиение и объединение

```java
String csv = "Alice,Bob,Charlie";
String[] names = csv.split(",");     // ["Alice", "Bob", "Charlie"]

String joined = String.join(" | ", names);  // "Alice | Bob | Charlie"

// Разбиение с лимитом
"a.b.c.d".split("\\.", 2);          // ["a", "b.c.d"] — только 2 части
```

> [!caution] split() принимает regex
> `"file.txt".split(".")` вернёт пустой массив, потому что `.` в regex означает "любой символ". Экранируй: `split("\\.")`.

## Форматирование строк

### String.format()

Работает как `printf` в C — подстановочные символы заменяются значениями.

```java
String name = "Alice";
int score = 95;
double percent = 95.5;

String msg = String.format("Студент %s набрал %d (%.1f%%)", name, score, percent);
// "Студент Alice набрал 95 (95.5%)"
```

| Подстановка | Тип | Пример |
|---|---|---|
| `%s` | String | `"Alice"` |
| `%d` | Integer | `42` |
| `%f` | Float/Double | `3.140000` |
| `%.2f` | Float с 2 знаками | `3.14` |
| `%n` | Перевод строки (платформозависимый) | |
| `%%` | Литерал `%` | `%` |

### formatted() (Java 15+)

```java
String msg = "Привет, %s! У вас %d сообщений.".formatted("Alice", 5);
```

### Text Blocks (Java 15+)

Для многострочных строк — удобно для SQL, JSON, HTML в тестах:

```java
String json = """
        {
            "name": "Alice",
            "age": 25,
            "active": true
        }
        """;
```

> [!tip] Применение в AQA
> Text blocks идеальны для ожидаемого JSON в API-тестах и SQL-запросов в тестах БД. Больше никакой уродливой конкатенации с `\n`.

## Основы регулярных выражений

Регулярные выражения (regex) позволяют искать шаблоны в тексте. В автотестировании их используют для валидации форматов, извлечения данных и парсинга логов.

### Часто используемые шаблоны

| Шаблон | Совпадает с | Пример |
|---|---|---|
| `\\d` | Цифра | `"5"` |
| `\\d+` | Одна или более цифр | `"123"` |
| `\\w` | Буква, цифра или `_` | `"a"` |
| `\\s` | Пробельный символ | `" "` |
| `.` | Любой символ | `"x"` |
| `[abc]` | Один из a, b, c | `"b"` |
| `[a-z]` | Любая строчная буква | `"m"` |
| `^` | Начало строки | |
| `$` | Конец строки | |
| `*` | Ноль или более | `"aaa"` или `""` |
| `+` | Один или более | `"aaa"` |
| `?` | Ноль или один | `"a"` или `""` |

### Использование regex со String

```java
// Проверка совпадения
String email = "user@example.com";
boolean valid = email.matches("[\\w.]+@[\\w.]+\\.[a-z]{2,}");  // true

// Замена по шаблону
String text = "Order #123 and Order #456";
String cleaned = text.replaceAll("#\\d+", "#XXX");
// "Order #XXX and Order #XXX"

// Разбиение по шаблону
String data = "Alice   Bob\tCharlie";
String[] names = data.split("\\s+");   // ["Alice", "Bob", "Charlie"]
```

### Regex в AQA

```java
// Валидация формата сообщения об ошибке
String error = "Error 404: Page not found";
assertTrue(error.matches("Error \\d{3}: .+"));

// Извлечение числа из текста
String price = "Total: $42.99";
String number = price.replaceAll("[^\\d.]", "");  // "42.99"
```

> [!info] Когда углубляться
> Для сложного regex (группы захвата, lookahead, именованные группы) используй классы `Pattern` и `Matcher`. Для простой валидации и замены хватает методов String.

## Вопросы для собеседования

### Вопросы начального уровня

**Почему String неизменяем в Java?**
Для безопасности String Pool (общие ссылки испортились бы, будь строки изменяемыми), потокобезопасности (не нужна синхронизация), безопасности (чувствительные строки нельзя изменить после проверки) и производительности (hashCode кешируется). Неизменяемость обеспечивается `final` классом + `private final` внутренним массивом.

**Что такое String Pool?**
Специальная область памяти, где Java хранит строковые литералы. Когда ты пишешь `"hello"`, JVM сначала проверяет пул. Если `"hello"` уже есть — возвращает существующую ссылку вместо создания нового объекта. Это экономит память.

**В чём разница между `==` и `.equals()` для строк?**
`==` сравнивает ссылки (адреса в памяти). `.equals()` сравнивает содержимое (символы). Две строки с одинаковым содержимым могут быть разными объектами: `new String("a") == new String("a")` — `false`, но `.equals()` — `true`.

**Что такое text blocks? (Java 15+)**
Многострочные строковые литералы с тройными кавычками (`"""`). Они сохраняют переносы строк и отступы, убирая общий ведущий пробел. Удобны для JSON, SQL, HTML-шаблонов в тестах.

**Для чего используется `String.format()`?**
Создание строк с подстановками: `String.format("Имя: %s, Возраст: %d", name, age)`. Подстановки: `%s` (String), `%d` (int), `%f` (float), `%.2f` (2 знака после запятой). Чище, чем конкатенация для сложных сообщений.

### Вопросы среднего уровня

**В чём разница между `StringBuilder` и `StringBuffer`?**
Оба — изменяемые последовательности символов. `StringBuilder` не потокобезопасен, но быстрее. `StringBuffer` потокобезопасен (synchronized), но медленнее. Используй `StringBuilder` в 99% случаев.

**Когда использовать `StringBuilder` вместо конкатенации String?**
При конкатенации в цикле или сборке текста из многих частей. Конкатенация `String` каждый раз создаёт новый объект. `StringBuilder` модифицирует на месте. Для простого `a + b + c` (без цикла) компилятор оптимизирует сам — `StringBuilder` не нужен.

**В чём разница между `trim()` и `strip()`?**
`trim()` убирает ASCII-пробелы (символы <= ` `). `strip()` (Java 11+) убирает все Unicode-пробелы, включая неразрывные. `strip()` — современный выбор.

**Как работает `split()`? Какой подвох?**
`split(regex)` разделяет строку по regex-шаблону. Подвох: аргумент — это regex, а не обычная строка. `"a.b".split(".")` вернёт пустой массив, потому что `.` в regex означает "любой символ". Используй `split("\\.")` для литеральной точки.

**Что делает `intern()`?**
`intern()` добавляет строку в String Pool (или возвращает существующую версию из пула). После `String s = new String("hi").intern();` `s` указывает на пуловую `"hi"`. Полезно в редких случаях, когда есть много дубликатов строк из внешних источников.

### Вопросы повышенного уровня

**Сколько объектов String создаёт `String s = new String("hello");`?**
До двух. Литерал `"hello"` создаёт один объект в String Pool (если его ещё не было). `new String(...)` создаёт второй объект в куче. Переменная `s` ссылается на объект в куче.

**Допустимо ли `"" + 5`? Что произойдёт?**
Да. Компилятор преобразует это в `new StringBuilder().append("").append(5).toString()`, что даёт `"5"`. Любой тип, конкатенированный со String, становится String.

**Почему `"abc" == "ab" + "c"` возвращает `true`?**
Компилятор оптимизирует константные выражения на этапе компиляции. `"ab" + "c"` состоит из литералов, поэтому вычисляется в `"abc"` при компиляции, и результат использует ту же строку из пула. Но `String a = "ab"; a + "c" == "abc"` — `false`, потому что `a` — переменная, и конкатенация происходит в runtime, создавая новый объект.

**Что произойдёт при вызове `split("")` на строке?**
Разделит каждый символ: `"abc".split("")` вернёт `["a", "b", "c"]`. В Java 8+ это не включает ведущую пустую строку (в более старых версиях Java включало).

### Вопросы с кодом

```java
String a = "hello";
String b = "hello";
String c = new String("hello");

System.out.println(a == b);
System.out.println(a == c);
System.out.println(a.equals(c));
```
**Что выведет этот код?**
**Ответ:**
```
true
false
true
```
`a` и `b` переиспользуют один и тот же литерал из пула, поэтому `==` даёт `true`. `c` — отдельный объект в куче из `new String(...)`, поэтому `a == c` даёт `false`. `.equals()` сравнивает содержимое, поэтому `a.equals(c)` даёт `true`.

---

```java
String name = "Alice";
name.toUpperCase();
System.out.println(name);
```
**Что выведет код и почему?**
**Ответ:** Выводит `Alice`. Строки неизменяемы, поэтому `toUpperCase()` возвращает **новый** String и не изменяет `name`. Нужно сохранить результат: `name = name.toUpperCase();`.

---

```java
String log = "Order #123 and Order #456";
String cleaned = log.replaceAll("#\\d+", "#XXX");
System.out.println(cleaned);

String row = "alice,bob,kate";
String[] users = row.split(",");
System.out.println(users.length);
```
**Что выведет этот код?**
**Ответ:**
```
Order #XXX and Order #XXX
3
```
`replaceAll` принимает regex, поэтому `#\\d+` совпадает с каждой группой «#цифры». `split(",")` разбивает по каждой запятой на три части.
