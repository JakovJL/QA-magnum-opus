## Что такое Appium

**Appium** — инструмент для автоматизации мобильных приложений на Android и iOS. Его API похож на Selenium WebDriver, поэтому подходы к ожиданиям, Page Object и проверкам во многом совпадают.

**Цель:** управлять настоящим устройством или эмулятором и проверять пользовательские сценарии мобильного приложения.

Appium поддерживает нативные, гибридные и мобильные веб-приложения.

## Как устроен запуск теста

```text
Тест на Java → Appium Server → драйвер платформы → устройство или эмулятор → приложение
```

Для Android обычно используется драйвер UiAutomator2, для iOS — XCUITest. Перед запуском нужны установленный Appium Server, инструменты платформы и доступное устройство либо эмулятор.

---

## Capabilities

Capabilities — это настройки сессии: какая платформа, устройство и приложение должны использоваться.

```java
UiAutomator2Options options = new UiAutomator2Options()
    .setDeviceName("Android Emulator")
    .setPlatformName("Android")
    .setApp("C:/apps/demo.apk");

AndroidDriver driver = new AndroidDriver(appiumUrl, options);
```

В реальном проекте путь к приложению, адрес сервера и имя устройства лучше читать из конфигурации, а не хранить в коде.

## Локаторы

Предпочтительный порядок локаторов:

1. `accessibility id` — стабильный и доступный для пользователей с ассистивными технологиями;
2. `id` — если он стабилен;
3. Android UiSelector или iOS predicate — когда стандартных локаторов нет;
4. XPath — только в крайнем случае.

```java
driver.findElement(AppiumBy.accessibilityId("login_button")).click();
```

> [!caution] Не привязывайся к тексту и координатам
> Текст меняется при локализации, а координаты — на экранах разного размера. Такие тесты часто падают без реального дефекта.

## Жесты и ожидания

Мобильный интерфейс часто загружается медленнее веб-интерфейса. Используй явные ожидания нужного состояния элемента.

Для прокрутки и свайпов применяй W3C Actions или специальные команды Appium. После жеста проверяй видимый результат: элемент появился, экран изменился или значение обновилось.

Не используй `Thread.sleep()`: он делает тесты медленными и не решает проблему нестабильной синхронизации.

---

## Типичные ошибки

| Ошибка | Как сделать лучше |
|---|---|
| Тестировать только эмулятор | Регулярно запускать важные сценарии на реальных устройствах |
| Использовать XPath для всех элементов | Сначала попросить стабильные `accessibility id` |
| Не сбрасывать данные приложения | Подготавливать состояние перед каждым тестом |
| Не учитывать разрешения и системные окна | Явно обрабатывать их в сценарии |
