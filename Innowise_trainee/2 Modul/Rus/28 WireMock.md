## Что такое WireMock

**WireMock** — библиотека, которая поднимает тестовый HTTP-сервер и возвращает заранее заданные ответы. Такой сервер называют заглушкой, или stub.

**Цель:** тестировать клиент или сервис без настоящей внешней системы.

WireMock полезен, когда внешний API платный, нестабилен, недоступен в тестовом окружении или слишком медленный.

---

## Когда он нужен

Используй WireMock, когда нужно проверить, как приложение обрабатывает:

- успешный ответ внешнего API;
- ошибку `404`, `429` или `500`;
- медленный ответ или тайм-аут;
- конкретные заголовки, параметры и тело запроса;
- повторные попытки выполнения запроса.

WireMock не проверяет настоящий контракт внешнего поставщика. Для этого нужны отдельные интеграционные или контрактные тесты.

## Stub и проверка запроса

Stub задаёт условие запроса и ответ, который вернёт тестовый сервер.

```java
stubFor(get(urlEqualTo("/users/42"))
    .willReturn(aResponse()
        .withStatus(200)
        .withHeader("Content-Type", "application/json")
        .withBody("{\"id\":42,\"name\":\"Ana\"}")));
```

После действия приложения полезно проверить, что запрос действительно был отправлен:

```java
verify(getRequestedFor(urlEqualTo("/users/42")));
```

Проверяй также важные заголовки, query-параметры и тело. Иначе тест может пройти, хотя приложение отправляет неверный запрос.

## Запуск в тесте

В JUnit 5 сервер обычно запускают перед тестом и останавливают после него. Адрес WireMock передают в конфигурацию тестируемого клиента вместо настоящего адреса API.

```java
WireMockServer server = new WireMockServer(options().dynamicPort());
server.start();
String baseUrl = server.baseUrl();
```

Динамический порт помогает избежать конфликтов, особенно при параллельном запуске тестов.

## Сценарии ошибок

Хороший набор тестов включает не только успешный ответ:

- `401` или `403` — приложение должно сообщить об отсутствии доступа;
- `404` — правильно обработать отсутствие ресурса;
- `429` — корректно применить повторную попытку или вывести понятную ошибку;
- `500` — не раскрывать пользователю технические детали;
- задержка ответа — сработал ли настроенный тайм-аут.

---

## Типичные ошибки

| Ошибка | Как сделать лучше |
|---|---|
| Заглушка отвечает на любой запрос | Ограничить метод, URL и важные параметры |
| Проверять только результат метода | Дополнительно вызвать `verify()` для запроса |
| Использовать WireMock вместо интеграционных тестов | Сохранять отдельные тесты с реальной системой |
| Оставлять сервер запущенным | Останавливать его в `@AfterEach` или `@AfterAll` |
