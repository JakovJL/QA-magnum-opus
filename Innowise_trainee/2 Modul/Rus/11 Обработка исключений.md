## Почему исключения важны

Программы не всегда работают по счастливому сценарию. Файл может отсутствовать, локатор может быть неверным, сетевой запрос может упасть, а ввод пользователя может оказаться некорректным. Java использует **исключения**, чтобы сигнализировать о таких проблемах.

Обработка исключений важна, потому что она помогает:

- Отделять основную логику от логики ошибок
- Избегать тихих сбоев
- Давать полезные сообщения для отладки
- Восстанавливаться после ожидаемых проблем, если это возможно

Без обработки исключений программа может завершиться с ошибкой без понятного объяснения, что именно произошло.

## Иерархия исключений

Все проблемы в Java наследуются от `Throwable`.

```text
Throwable
├── Error
└── Exception
    └── RuntimeException
```

### Error

`Error` означает серьёзные проблемы на уровне JVM. Обычно прикладной код не должен пытаться их обрабатывать.

Примеры:

- `OutOfMemoryError`
- `StackOverflowError`

### Exception

`Exception` означает проблемы, которые прикладной код может захотеть обработать.

Примеры:

- `IOException`
- `SQLException`
- `ParseException`

### RuntimeException

`RuntimeException` — это особая группа исключений, чаще всего связанная с ошибками в коде.

Примеры:

- `NullPointerException`
- `IllegalArgumentException`
- `IndexOutOfBoundsException`
- `ArithmeticException`

> [!danger] Ключевое правило
> `Error` обычно не обрабатывают. `Exception` обрабатывают или объявляют. `RuntimeException` часто означает баг в коде или некорректные данные.

## Checked и unchecked исключения

Это одна из самых частых тем на собеседованиях.

### Checked исключения

Checked-исключения проверяются компилятором. Ты обязан либо обработать их через `try-catch`, либо объявить через `throws`.

```java
public String readFile(String path) throws IOException {
    return Files.readString(Path.of(path));
}
```

Примеры:

- `IOException`
- `SQLException`
- `ClassNotFoundException`

### Unchecked исключения

Unchecked-исключения являются потомками `RuntimeException`. Компилятор не заставляет тебя их обрабатывать.

```java
public int divide(int a, int b) {
    return a / b; // может выбросить ArithmeticException, если b == 0
}
```

Примеры:

- `NullPointerException`
- `IllegalStateException`
- `NumberFormatException`

> [!tip] Быстрое сравнение
> | Тип | Проверяется компилятором | Типичная причина |
> |---|---|---|
> | Checked exception | Да | Внешняя проблема: файл, БД, сеть |
> | Unchecked exception | Нет | Ошибка в коде или плохой ввод |

Если совсем просто:

- Checked exception -> "что-то снаружи кода может не сработать"
- Unchecked exception -> "код неправильно использует данные"

## try catch и finally

Базовый синтаксис обработки исключений — `try-catch-finally`.

```java
try {
    int result = 10 / 0;
} catch (ArithmeticException e) {
    System.out.println("Cannot divide by zero");
} finally {
    System.out.println("This block always runs");
}
```

### `try`

Блок `try` содержит код, который может выбросить исключение.

### `catch`

Блок `catch` обрабатывает конкретное исключение.

```java
try {
    Integer.parseInt("abc");
} catch (NumberFormatException e) {
    System.out.println("Invalid number: " + e.getMessage());
}
```

### `finally`

Блок `finally` почти всегда выполняется независимо от того, было исключение или нет. Обычно он нужен для очистки ресурсов.

```java
FileInputStream input = null;
try {
    input = new FileInputStream("data.txt");
    // work with file
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (input != null) {
        try {
            input.close();
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

### Несколько catch-блоков

```java
try {
    String text = args[0];
    int number = Integer.parseInt(text);
} catch (ArrayIndexOutOfBoundsException e) {
    System.out.println("No argument passed");
} catch (NumberFormatException e) {
    System.out.println("Argument is not a number");
}
```

> [!warning] Порядок важен
> Сначала ставь более специфичные исключения, потом более общие. Если сначала поймать `Exception`, компилятор не разрешит следующий `catch` для `IOException`, потому что он станет недостижимым.

## throw и throws

Эти два ключевых слова часто путают.

### `throw`

`throw` используется внутри метода, чтобы создать и выбросить один объект исключения.

```java
public void setAge(int age) {
    if (age < 0) {
        throw new IllegalArgumentException("Age cannot be negative");
    }
}
```

### `throws`

`throws` используется в сигнатуре метода, чтобы объявить: метод может передать исключение вызывающему коду.

```java
public String loadConfig(String path) throws IOException {
    return Files.readString(Path.of(path));
}
```

> [!info] Короткое правило
> `throw` -> действие внутри метода. `throws` -> объявление в сигнатуре метода.

## Пользовательские исключения

Иногда стандартные исключения слишком общие. В таком случае можно создать собственный класс исключения.

```java
public class InvalidUserDataException extends RuntimeException {
    public InvalidUserDataException(String message) {
        super(message);
    }
}
```

```java
public void register(User user) {
    if (user.getEmail() == null) {
        throw new InvalidUserDataException("Email is required");
    }
}
```

### Когда использовать пользовательские исключения

- Когда ошибка имеет понятный бизнес-смысл
- Когда нужны более хорошие логи и сообщения
- Когда нужен один явный тип для одной области проблем

В автотестах пользовательские исключения делают падения понятнее.

## Try-With-Resources

Если ты работаешь с файлами, потоками, соединениями с БД или другими ресурсами, которые нужно закрывать, используй **try-with-resources**.

```java
try (BufferedReader reader = Files.newBufferedReader(Path.of("data.txt"))) {
    String line = reader.readLine();
    System.out.println(line);
} catch (IOException e) {
    e.printStackTrace();
}
```

Почему это лучше:

- Ресурс закрывается автоматически
- Код короче
- Меньше риск утечки памяти или ресурсов

Это работает с классами, которые реализуют `AutoCloseable`, например:

- streams
- readers/writers
- JDBC connections, statements, result sets

> [!tip] Best Practice
> Предпочитай try-with-resources ручной очистке через `finally`, когда работаешь с закрываемыми ресурсами.

## Обработка исключений в AQA

В автоматизации исключения происходят постоянно. Цель не в том, чтобы их скрывать. Цель — обрабатывать их правильно и оставлять понятные сообщения о падении.

### Частые примеры

```java
try {
    driver.findElement(By.id("login")).click();
} catch (NoSuchElementException e) {
    throw new AssertionError("Login button was not found", e);
}
```

### Типичные исключения в AQA

- `NoSuchElementException` - элемент не найден
- `TimeoutException` - условие ожидания не выполнилось
- `StaleElementReferenceException` - DOM изменился
- `AssertionError` - ожидаемый результат не совпал с фактическим

### Хорошие практики

- Добавляй понятное сообщение при повторном выбрасывании исключения
- Не лови все исключения подряд без причины
- Не скрывай падения тестов пустыми `catch`-блоками
- Лови исключение только тогда, когда можешь восстановиться или улучшить сообщение

### Плохой пример

```java
try {
    driver.findElement(By.id("login")).click();
} catch (Exception e) {
    // ignore
}
```

Это плохо, потому что тест может продолжить работу в сломанном состоянии и упасть позже уже с непонятной причиной.

> [!caution] Пустой catch опасен
> Пустой `catch` скрывает настоящую проблему. В тестах это создаёт ложные успешные результаты или неясные падения. Если ты ловишь исключение, залогируй его, заверни или урони тест с понятным сообщением.

## Вопросы для собеседования

### Вопросы начального уровня

**1. Что такое исключение в Java?**
Исключение — это объект, который описывает проблему во время выполнения программы. Java использует его, чтобы прервать обычный поток выполнения и передать управление коду обработки ошибок.

**2. В чём разница между `throw` и `throws`?**
`throw` используется внутри метода для выбрасывания объекта исключения. `throws` используется в сигнатуре метода для объявления возможных исключений.

**3. Для чего нужен `finally`?**
Он нужен для кода очистки, который должен выполниться независимо от того, произошло исключение или нет, например для закрытия ресурса.

**4. Можно ли ловить несколько исключений?**
Да. Можно использовать несколько `catch`-блоков или multi-catch синтаксис вида `catch (IOException | SQLException e)`.

**5. Какой класс является родителем для всех исключений?**
Технически верхний класс — `Throwable`. `Exception` является родителем для большинства прикладных исключений.

**6. Что такое try-with-resources?**
Это синтаксис, который автоматически закрывает ресурсы после использования. Он работает с классами, реализующими `AutoCloseable`.

**7. Когда стоит создавать пользовательское исключение?**
Когда проблема имеет понятный бизнес-смысл и отдельный тип исключения делает код и логи понятнее.

---

### Вопросы среднего уровня

**1. В чём разница между checked и unchecked исключениями?**
Checked-исключения проверяются компилятором и должны быть обработаны или объявлены. Unchecked-исключения наследуются от `RuntimeException`, и компилятор не заставляет их обрабатывать.

**2. Является ли `RuntimeException` checked или unchecked?**
Unchecked. Компилятор не заставляет его обрабатывать.

**3. В чём разница между `Exception` и `Error`?**
`Exception` описывает проблемы, которые прикладной код может обработать. `Error` описывает серьёзные системные проблемы, которые обычно не считаются восстанавливаемыми в обычном коде.

**4. Почему обычно плохо ловить `Throwable`?**
Потому что тогда ловятся и `Error`, а серьёзные проблемы JVM обычно не нужно обрабатывать как обычные ошибки бизнеса.

**5. Нужно ли ловить `Exception` везде?**
Нет. Слишком широкий `catch` скрывает реальные проблемы и усложняет отладку. По возможности лучше ловить конкретные исключения.

**6. Можно ли иметь `try` без `catch`?**
Да, если есть `finally`. Также можно использовать try-with-resources с `catch` и/или `finally`.

**7. Когда ловить исключение, а когда пробрасывать `throws`?**
Лови, когда: (1) можешь восстановиться (retry, fallback, default value), (2) хочешь добавить контекст (exception chaining), (3) должен освободить ресурсы (close, rollback).
Пробрасывай, когда: (1) не знаешь, как обработать (пусть caller решает), (2) в библиотечном коде (не твоё дело решать), (3) аннотация `throws` уже есть и логично её расширить.

### Вопросы повышенного уровня

**1. Multi-catch vs несколько `catch`-блоков — в чём разница?**
Несколько `catch`-блоков ловят исключения по порядку — первое совпадение выигрывает. Multi-catch (`catch (IOException | SQLException e)`) ловит несколько типов в одном блоке. Ограничение: типы в multi-catch не могут быть в отношении родитель-потомок (компилятор заставит указать только родителя).

```java
// ✅ Несколько catch — разные действия:
try { ... }
catch (FileNotFoundException e) { log("file missing"); }
catch (IOException e) { log("io error"); }

// ✅ Multi-catch — одинаковое действие:
try { ... }
catch (IOException | SQLException e) { log("db or io error", e); }

// ❌ Ошибка — FileNotFoundException extends IOException:
// catch (FileNotFoundException | IOException e) { }
```

**2. Как JVM обрабатывает исключение «под капотом»?**
При исключении JVM: (1) создаёт объект исключения (стек-трейс заполняется), (2) ищет подходящий `catch`-блок от самого внутреннего `try` к внешним (по стеку вызовов), (3) если ни один `catch` не найден — исключение идёт выше по стеку вызовов, пока не достигнет main, после чего поток завершается. Если исключение не обработано нигде — программа падает с сообщением и стек-трейсом в stderr.

**3. Как читать и анализировать стек-трейс?**
Стек-трейс читается **сверху вниз**. Самая верхняя строка — точное место выброса исключения. Чаще всего проблема в твоём коде (не в библиотечном), ищи строку с твоим пакетом.

```java
Exception in thread "main" java.lang.NullPointerException
    at com.myapp.service.UserService.findUser(UserService.java:25) // ← тут проблема
    at com.myapp.controller.UserController.getUser(UserController.java:12)
    at com.myapp.Main.main(Main.java:5)
```

`e.getMessage()` — короткое описание. `e.getCause()` — исходное исключение (при chaining). `e.printStackTrace()` — полный стек-трейс.

**4. Что такое exception chaining (цепочка исключений)?**
Оборачивание одного исключения в другое с сохранением исходной причины. Позволяет не терять контекст при пробросе через слои приложения.

```java
try {
    parseFile(path);
} catch (IOException e) {
    throw new RuntimeException("Failed to load config from " + path, e);
}
// Потом: e.getCause() вернёт исходный IOException
```

**5. Что такое suppressed exceptions (подавленные исключения)?**
Когда try-with-resources закрывает несколько ресурсов, каждое исключение при закрытии **подавляется** в пользу первичного исключения. Подавленные исключения можно получить через `e.getSuppressed()`. Появились в Java 7 для решения проблемы скрытых исключений в finally.

```java
try (AutoCloseable a = () -> { throw new RuntimeException("A"); };
     AutoCloseable b = () -> { throw new RuntimeException("B"); }) {
    throw new RuntimeException("Main");
}
// Основное: "Main". Подавленные: "A", "B"
```

**6. Что произойдёт, если в `finally` выбросить исключение?**
Оно **заменит** собой исходное исключение из `try` или `catch`. Это одна из главных проблем finally — потеря информации об ошибке. Именно поэтому try-with-resources безопаснее: он использует suppressed exceptions вместо замены.

```java
try {
    throw new RuntimeException("Original");
} finally {
    throw new RuntimeException("Finally"); // ❌ Original потерян
}
// Будет только "Finally"
```

**7. Можно ли использовать `return` в `try`, `catch` и `finally`? Что победит?**
`finally` **всегда** побеждает. Если в `try` есть `return`, а в `finally` — `return`, то вернётся значение из `finally`. Если в `finally` есть `return`, а в `catch` — `throw`, то `return` подавит исключение.

```java
public int test() {
    try { return 1; }
    finally { return 2; } // вернётся 2, не 1
}
```

**8. Какие исключения в Selenium WebDriver самые частые?**
| Исключение | Когда возникает |
|---|---|
| `NoSuchElementException` | Элемент не найден в DOM |
| `StaleElementReferenceException` | Элемент был в DOM, но исчез (страница обновилась) |
| `TimeoutException` | Ожидание не выполнилось за таймаут |
| `ElementClickInterceptedException` | Другой элемент перекрывает целевой |
| `InvalidElementStateException` | Элемент не в том состоянии (disabled, not visible) |
| `WebDriverException` | Общее исключение WebDriver (упал драйвер, нет соединения) |

### Вопросы с кодом

**1. Что выведет этот код?**
```java
public class Test {
    public static void main(String[] args) { System.out.println(test()); }
    static int test() {
        try { return 1; }
        catch (Exception e) { return 2; }
        finally { return 3; }
    }
}
```
**Ответ:** `3`. `finally` выполняется после `try` или `catch`, но перед фактическим возвратом. Если `finally` содержит `return` — он заменяет любой предыдущий return.

**2. Что выведет этот код?**
```java
public class Test {
    public static void main(String[] args) {
        try { System.exit(0); }
        finally { System.out.println("Finally"); }
    }
}
```
**Ответ:** Ничего. `System.exit(0)` останавливает JVM немедленно. `finally` не выполняется.

**3. Скомпилируется ли это?**
```java
try { throw new IOException(); }
catch (IOException e) { throw new RuntimeException(e); }
catch (RuntimeException e) { System.out.println("Caught"); }
```
**Ответ:** Да. `IOException` пойман, обёрнут в `RuntimeException` и выброшен снова. Второй `catch` ловит его — выведет "Caught".

**4. Что выведет этот код?**
```java
public class Test {
    public static void main(String[] args) {
        try { throw new RuntimeException("Outer"); }
        catch (RuntimeException e) { throw new RuntimeException("Inner", e); }
        finally { throw new RuntimeException("Finally"); }
    }
}
```
**Ответ:** Программа упадёт с `RuntimeException: Finally`. Исходные "Outer" и "Inner" потеряны. Исключение из `finally` заменяет все предыдущие.

**5. Что выведет этот код? (suppressed exceptions)**
```java
try (AutoCloseable res = () -> { throw new Exception("Close"); }) {
    throw new Exception("Work");
} catch (Exception e) {
    System.out.println(e.getMessage());
    for (Throwable s : e.getSuppressed()) {
        System.out.println("Suppressed: " + s.getMessage());
    }
}
```
**Ответ:**
```
Work
Suppressed: Close
```
Исключение при закрытии ресурса подавляется в пользу основного. `getSuppressed()` показывает оба.

## Практические задачи

> [!tip] Как тренироваться
> Сначала попробуй решить сам, потом сверься с решением.

### 1. try-catch-finally с return

**Задание:** Что вернёт этот метод и почему?

```java
public int calculate() {
    try {
        return 10 / 0;
    } catch (ArithmeticException e) {
        return -1;
    } finally {
        System.out.println("Finally всегда выполняется");
    }
}
```

**Ответ:** Вернёт `-1`. Деление на ноль бросает `ArithmeticException`, catch ловит и возвращает -1. `finally` выполняется до return — печатает "Finally всегда выполняется".

**Разбор:** `finally` выполняется после try/catch, но до фактического возврата из метода. Если бы в `finally` был `return`, он переопределил бы return из catch.

### 2. Пользовательское исключение с chaining

**Задание:** Создай пользовательское checked исключение `ConfigLoadException` и метод, который оборачивает `IOException` в него.

```java
public class ConfigLoadException extends Exception {
    public ConfigLoadException(String message, Throwable cause) {
        super(message, cause);
    }
}

public String loadConfig(String path) throws ConfigLoadException {
    try {
        return Files.readString(Path.of(path));
    } catch (IOException e) {
        throw new ConfigLoadException("Failed to load config from " + path, e);
    }
}

// Использование:
try {
    String cfg = loadConfig("app.properties");
} catch (ConfigLoadException e) {
    System.out.println(e.getMessage());   // "Failed to load..."
    System.out.println(e.getCause());     // IOException с деталями
}
```

**Разбор:** Exception chaining сохраняет исходную причину. `getCause()` возвращает оригинальный `IOException`. Это стандартный паттерн для обёртки исключений между слоями.

### 3. Multi-catch — исправь порядок

**Задание:** Этот код не компилируется. Исправь его.

```java
try {
    // код, который может бросить разные исключения
} catch (Exception e) {
    log("Общая ошибка");
} catch (IOException e) {
    log("IO ошибка");
}
```

**Ответ:** Не компилируется — `IOException` extends `Exception`, первый catch уже ловит всё. Фикс: поменять порядок (более специфичное — первым).

```java
try {
    // код
} catch (IOException e) {
    log("IO ошибка");
} catch (Exception e) {
    log("Общая ошибка");
}
```

**Разбор:** Более специфичное исключение должно ловиться до более общего. Компилятор проверяет это на этапе компиляции.

### 4. Try-with-resources — свой ресурс

**Задание:** Создай кастомный `AutoCloseable` ресурс и используй его в try-with-resources. Он должен печатать "Open" при создании и "Close" при закрытии.

```java
public class DatabaseConnection implements AutoCloseable {
    public DatabaseConnection() { System.out.println("Open"); }
    public void query(String sql) { System.out.println("Query: " + sql); }
    @Override public void close() { System.out.println("Close"); }
}

// Использование:
try (DatabaseConnection db = new DatabaseConnection()) {
    db.query("SELECT * FROM users");
}
// Выведет:
// Open
// Query: SELECT * FROM users
// Close
```

**Разбор:** Любой класс, реализующий `AutoCloseable`, можно использовать в try-with-resources. `close()` вызывается автоматически, даже если в try произошло исключение.

### 5. Suppressed exceptions на практике

**Задание:** Что выведет этот код?

```java
try (AutoCloseable a = () -> { throw new RuntimeException("A"); };
     AutoCloseable b = () -> { throw new RuntimeException("B"); }) {
    throw new RuntimeException("Main");
} catch (RuntimeException e) {
    System.out.println(e.getMessage());
    for (Throwable s : e.getSuppressed()) {
        System.out.println("Suppressed: " + s.getMessage());
    }
}
```

**Ответ:**
```
Main
Suppressed: A
Suppressed: B
```

Основное исключение — "Main". "A" и "B" — подавленные, потому что try-with-resources собирает исключения закрытия, не теряя основное. Без try-with-resources (через finally) "Main" был бы потерян.
