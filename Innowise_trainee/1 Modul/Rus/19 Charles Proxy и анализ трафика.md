## Что такое network sniffer и proxy

**Network sniffer** — это инструмент, который перехватывает и показывает сетевой трафик. **Proxy** — это промежуточный сервер, через который трафик проходит перед тем, как попасть к реальному серверу.

**Цель:** дать QA возможность увидеть, какие запросы и ответы приложение действительно отправляет и получает.

**Почему это важно:** UI показывает только часть картины, а настоящая проблема может быть в headers, payload, status code, SSL или timing запроса.

---

## Зачем QA используют proxy

QA используют proxy, чтобы:
- смотреть API requests и responses
- отлаживать mobile applications
- проверять headers, cookies и tokens
- воспроизводить медленную сеть
- изменять requests и responses во время тестирования

**Пример:** UI показывает `unknown error`, а Charles показывает, что backend вернул `403 Forbidden`, потому что token истёк.

---

## Обзор Charles Proxy

**Charles Proxy** — desktop application, который работает как локальный proxy между устройством и интернетом.

**Как это работает:** приложение или browser отправляет трафик в Charles, Charles его перехватывает и потом пересылает целевому серверу.

**Почему это полезно для QA:**
- удобный просмотр request и response
- проверка SSL-трафика
- повтор и редактирование запросов
- отладка трафика мобильных приложений

> [!info] Man-in-the-middle идея
> Charles работает как управляемый man-in-the-middle proxy. Поэтому для проверки HTTPS нужен сертификат.

## Настройка proxy

### Настройка proxy на desktop

Чтобы перехватывать трафик browser или desktop app, обычно делают так:
1. Запускают Charles
2. Включают system proxy settings или вручную указывают приложению Charles
3. Проверяют, что приложение отправляет трафик через Charles
4. Смотрят список сессий, request и response

**Частая деталь:** Charles часто слушает порт `8888`.

### Настройка proxy на mobile device

Чтобы перехватывать трафик с телефона:
1. Подключить телефон и компьютер к одной Wi-Fi сети
2. Узнать IP-адрес компьютера
3. Открыть настройки Wi-Fi proxy на телефоне
4. Ввести IP компьютера и порт Charles
5. Доверить сертификат, если нужно смотреть HTTPS

**Применимость для QA:** это особенно часто нужно при тестировании native mobile apps, которые работают с backend API.

## Проверка SSL-трафика

HTTPS-трафик зашифрован, поэтому Charles не может читать его, пока устройство не доверяет root certificate Charles.

**Цель:** видеть body request и response для HTTPS endpoint'ов.

**Типичные шаги:**
1. Установить root certificate Charles
2. Доверить сертификат в настройках OS или устройства
3. Включить SSL Proxying для нужных доменов

**Почему это нужно:** без доверенного сертификата Charles видит соединение, но не может расшифровать payload.

**Что проверяет QA:**
- добавлен ли правильный host для SSL proxying
- отклоняет ли приложение недоверенные сертификаты
- виден ли body после корректной настройки trust

> [!caution] Certificate pinning
> Некоторые mobile apps используют certificate pinning. В этом случае приложение может отклонить Charles даже после установки root certificate.

## Ключевые функции Charles для QA

У Charles есть несколько функций, которые особенно полезны в тестировании.

### Breakpoints

**Breakpoints** приостанавливают request или response до его пересылки дальше.

**Цель:** посмотреть или изменить трафик в середине flow.

**Пример:** остановить request, поменять параметр и потом отправить дальше.

### Map Local и Map Remote

**Map Local** подменяет ответ сервера локальным файлом.

**Map Remote** перенаправляет трафик с одного удалённого endpoint на другой.

**Цель:** симулировать специальное поведение backend без изменения реального сервера.

**Пример:** вернуть локальный JSON-файл с ошибкой, чтобы протестировать UI.

### Throttling

**Throttling** замедляет сеть.

**Цель:** симулировать плохие условия интернета, например 3G или нестабильную мобильную сеть.

**Пример:** проверить, показывает ли приложение loader и как оно обрабатывает timeouts.

### Repeat и Compose

**Repeat** повторно отправляет уже существующий request.

**Compose** позволяет вручную собрать и отправить новый request.

**Цель:** быстро повторять или создавать трафик без полного повторения UI-шагов.

## Charles для мобильного тестирования

Charles очень полезен в mobile QA, потому что мобильные приложения часто общаются с API, которые трудно увидеть только через UI.

**Частые применения:**
- посмотреть login и token refresh requests
- проверить headers и payloads
- протестировать поведение приложения на медленной или сломанной сети
- проверить обработку `401`, `403` или `500`

**Важное ограничение:** некоторые приложения блокируют proxy inspection из-за certificate pinning.

## Вопросы для собеседования

### Вопросы начального уровня

**1. Что такое Charles Proxy?**
Charles Proxy — это локальный proxy tool, который перехватывает и показывает сетевой трафик между приложением и сервером.

**2. Зачем QA используют Charles?**
Чтобы смотреть requests и responses, отлаживать mobile apps, видеть HTTPS-трафик, симулировать медленную сеть и повторять запросы.

**3. В чём разница между sniffer и proxy?**
Sniffer захватывает трафик. Proxy стоит посередине, и трафик проходит через него. Charles в основном работает как proxy.

### Вопросы среднего уровня

**1. Как подключить mobile device к Charles?**
Подключить телефон и компьютер к одной Wi-Fi сети, указать на телефоне proxy с IP компьютера и портом Charles, затем установить и доверить сертификат Charles, если нужна проверка HTTPS.

**2. Зачем нужен сертификат для проверки HTTPS?**
Потому что HTTPS-трафик зашифрован. Устройство должно доверять root certificate Charles, чтобы Charles мог расшифровать и показать трафик.

**3. Для чего нужны breakpoints в Charles?**
Они останавливают requests или responses, чтобы их можно было посмотреть или изменить перед отправкой дальше.

**4. Для чего нужен throttling?**
Чтобы симулировать медленную или нестабильную сеть и проверить поведение приложения.

### Вопросы повышенного уровня

**1. Charles настроен, но mobile app всё равно показывает SSL errors. Почему?**
Возможные причины: сертификат не доверен, неправильно настроен SSL proxying host или приложение использует certificate pinning.

**2. Почему повтор запроса полезен для QA?**
Он позволяет быстро перепроверять поведение backend без полного повторения UI-сценария каждый раз.

### Вопросы с кодом

**1. Что обычно означает такая настройка?**

```text
Proxy: 192.168.0.15:8888
```

Это значит, что устройство должно отправлять трафик на компьютер с IP `192.168.0.15` и портом `8888`, где слушает Charles.
