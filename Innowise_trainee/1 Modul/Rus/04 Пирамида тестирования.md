## Пирамида тестирования

Пирамида тестирования — это модель, описывающая идеальное распределение тестов по уровням системы. Была предложена Майком Коном (Mike Cohn).

```
        /  E2E  \           ← мало, медленные, дорогие
       /----------\
      / Integration \       ← умеренное количество
     /----------------\
    /    Unit Tests     \   ← много, быстрые, дешёвые
   /____________________\
```

**Основная идея:** чем ниже уровень, тем больше тестов должно быть. Тесты нижнего уровня быстрее, дешевле и проще в сопровождении. Тесты верхнего уровня дают больше уверенности, но дороже в написании и запуске.

---

## Слои пирамиды

→ Подробные определения каждого уровня тестирования см. [[01 Виды тестирования#Уровни тестирования]]

### Модульные тесты (Unit)

Основание пирамиды. Тестируют отдельные функции, методы или классы в изоляции.

- **Скорость:** миллисекунды на тест
- **Стоимость:** низкая для написания и сопровождения
- **Кто пишет:** разработчики
- **Цикл обратной связи:** мгновенный — запускается при каждом сохранении или коммите
- **Что ловят:** логические ошибки, ошибки в вычислениях, граничные случаи отдельных компонентов

**Пример:** Тестирование того, что `calculateTotal(items)` возвращает правильную сумму, включая случаи со скидками, пустой корзиной и отрицательными количествами.

---

### Интеграционные тесты

Средний слой. Тестируют, как несколько компонентов работают вместе — модули, сервисы, базы данных, API.

- **Скорость:** секунды на тест
- **Стоимость:** умеренная — могут требовать тестовых баз данных, мок-сервисов или контейнеров
- **Кто пишет:** разработчики или QA / инженеры по автоматизации
- **Цикл обратной связи:** минуты — обычно запускается в CI-пайплайне
- **Что ловят:** нарушенные контракты между модулями, некорректную передачу данных, проблемы конфигурации

**Пример:** Тестирование того, что эндпоинт регистрации пользователя корректно сохраняет данные в базу, отправляет email подтверждения через почтовый сервис и возвращает ожидаемый ответ.

---

### Сквозные тесты (E2E)

Вершина пирамиды. Тестируют полные пользовательские сценарии через реальный UI, имитируя взаимодействие реального пользователя с системой.

- **Скорость:** от секунд до минут на тест
- **Стоимость:** высокая — требуют полного окружения, хрупкие из-за изменений UI
- **Кто пишет:** QA / инженеры по автоматизации
- **Цикл обратной связи:** медленный — запускаются по расписанию или перед релизами
- **Что ловят:** сломанные пользовательские сценарии, интеграционные проблемы по всему стеку, проблемы развёртывания

**Пример:** Тестирование полного потока покупки: вход → поиск товара → добавление в корзину → оформление заказа → подтверждение → проверка заказа в истории.

> [!tip] Краткое сравнение
> | | Unit | Integration | E2E |
> |---|---|---|---|
> | Количество | Много | Умеренно | Мало |
> | Скорость | Миллисекунды | Секунды | Секунды–минуты |
> | Стоимость | Низкая | Средняя | Высокая |
> | Стабильность | Очень стабильные | Стабильные | Хрупкие |
> | Уверенность | Низкая (изолированно) | Средняя | Высокая (реальный путь) |
> | Кто пишет | Разработчики | Dev / QA | QA / Автоматизация |

---

## Почему форма пирамиды важна

Форма пирамиды не случайна. Она отражает ключевые компромиссы:

- **Скорость обратной связи.** Модульные тесты выполняются за секунды, давая разработчикам мгновенную обратную связь. E2E-тесты могут занимать минуты или часы — медленная обратная связь означает, что баги находятся позже и стоят дороже.
- **Стоимость сопровождения.** Изменение UI может сломать десятки E2E-тестов, но ноль модульных. Чем выше уровень, тем более хрупкими и дорогими становятся тесты.
- **Сложность отладки.** Когда модульный тест падает, баг в одной функции. Когда падает E2E-тест, баг может быть в любом месте стека — поиск занимает гораздо больше времени.
- **Уверенность.** Модульные тесты доказывают, что отдельные части работают. Только E2E-тесты доказывают, что вся система работает для реальных пользователей. Нужны оба типа, но в разных пропорциях.

> [!danger] Золотое правило
> Если что-то можно протестировать на более низком уровне — тестируйте на более низком уровне. Опускайте тесты вниз по пирамиде при любой возможности. Оставляйте E2E-тесты только для критических пользовательских сценариев, которые невозможно проверить иначе.

> [!info] Пирамида — это ориентир, а не закон
> Классическая пирамида — не единственная верная форма. Разным архитектурам соответствуют разные распределения:
> - **Testing Diamond** (Алмаз тестирования) — широкий интеграционный слой. Полезен для **микросервисов**, где главный риск — контракты *между* сервисами, а не модули внутри них.
> - **Testing Trophy** (Трофей тестирования) — делает акцент на большом автоматизированном **интеграционном** слое с меньшим числом E2E-тестов; популярен в современных веб-приложениях.
>
> Пирамида остаётся самым распространённым вариантом по умолчанию для типичных приложений, но правильная форма зависит от того, где реально находятся риски и сложность вашей системы.

## Антипаттерны

### Рожок мороженого (Ice Cream Cone)

Перевёрнутая пирамида: большинство тестов — E2E и ручные, модульных почти нет.

```
   ____________________
   \   Manual / E2E   /   ← большинство тестов здесь
    \----------------/
     \ Integration /
      \----------/
       \  Unit  /          ← почти нет
        \______/
```

**Почему возникает:**
- Тесты добавляются поздно, когда существует только UI
- QA-команда пишет тесты без участия разработчиков
- Мышление «если работает в браузере — значит работает»

**Почему это плохо:**
- Набор тестов крайне медленный — CI занимает часы
- Тесты хрупкие — небольшое изменение CSS ломает десятки тестов
- Отладка сложна — ошибка может быть в любом слое
- Разработчики не получают быстрой обратной связи и перестают доверять тестам

**Как исправить:**
- Ввести модульные тесты для всего нового кода
- Постепенно заменять E2E-тесты тестами более низкого уровня, где возможно
- Оставить E2E-тесты только для критических пользовательских сценариев

---

### Песочные часы (Hourglass)

Много модульных тестов, много E2E-тестов, но почти нет интеграционных. Средний слой отсутствует.

```
        /  E2E  \           ← много
       /----------\
      |            |        ← почти нет
       \----------/
    /    Unit Tests     \   ← много
   /____________________\
```

**Почему возникает:**
- Разработчики пишут модульные тесты, QA пишет E2E — за средний слой никто не отвечает
- Интеграционные тесты сложнее настраивать (тестовые базы данных, контейнеры, моки)
- Команды пропускают интеграционные тесты, думая, что E2E поймает всё

**Почему это плохо:**
- Модульные тесты проходят, но компоненты ломаются при объединении — нарушенные интерфейсы не обнаруживаются
- E2E-тесты ловят интеграционные баги, но медленно и с трудной отладкой
- Пробел создаёт ложное чувство безопасности: «все тесты проходят», но система ломается между слоями

**Как исправить:**
- Добавить интеграционные тесты для ключевых границ модулей и API-контрактов
- Использовать инструменты вроде Testcontainers или in-memory баз данных для упрощения настройки
- Сделать интеграционные тесты общей ответственностью разработчиков и QA

---

### Кекс (Cupcake)

Одни и те же проверки дублируются на каждом уровне пирамиды. Каждый слой тестирует одно и то же вместо того, чтобы тестировать то, что он делает лучше всего.

```
        / E2E: проверяет валидацию логина \
       /----------------------------------\
      / Integration: проверяет валидацию логина \
     /------------------------------------------\
    / Unit: проверяет валидацию логина             \
   /________________________________________________\
```

**Почему возникает:**
- Нет чёткой стратегии, какой уровень что тестирует
- Команды копируют тест-кейсы между уровнями «на всякий случай»
- Страх пропустить баги приводит к избыточному тестированию

**Почему это плохо:**
- Потраченные усилия — сопровождение одной и той же проверки в трёх местах
- Одно изменение вызывает падения на каждом уровне, усложняя поиск реальной проблемы
- Медленный CI-пайплайн без дополнительной уверенности

**Как исправить:**
- Определить чёткую стратегию тестирования: каждый уровень тестирует то, что умеет лучше всего
- Модульные тесты → логика и граничные случаи
- Интеграционные тесты → потоки данных и контракты между компонентами
- E2E-тесты → только критические пользовательские сценарии
- Удалить дублирующие проверки с верхних уровней, когда нижний уровень уже их покрывает

> [!tip] Антипаттерны одним взглядом
> | Антипаттерн | Форма | Проблема | Причина |
> |---|---|---|---|
> | Рожок мороженого | Перевёрнутая пирамида | Медленно, хрупко, сложно отлаживать | Тесты добавлены поздно, нет модульных |
> | Песочные часы | Сжатая середина | Интеграционные баги проскальзывают | Никто не отвечает за интеграционные тесты |
> | Кекс | Одно и то же везде | Потраченные усилия, медленный CI | Нет стратегии тестирования |

## Вопросы для собеседования

### Вопросы начального уровня

**1. Что такое пирамида тестирования и кто её предложил?**
Пирамида тестирования — это модель, предложенная Майком Коном, которая описывает идеальное распределение тестов: много быстрых модульных тестов в основании, меньше интеграционных в середине и небольшое количество медленных E2E-тестов на вершине.

**2. Что тестирует каждый слой пирамиды?**
Unit: отдельные функции и классы в изоляции. Integration: взаимодействия между модулями, сервисами и базами данных. E2E: полные пользовательские сценарии через реальный UI.

**3. Каково золотое правило пирамиды тестирования?**
Если что-то можно протестировать на более низком уровне — тестируйте на более низком уровне. Опускайте тесты вниз по пирамиде при любой возможности.

### Вопросы среднего уровня

**1. Почему модульных тестов должно быть больше, чем E2E-тестов?**
Модульные тесты быстрые, дешёвые, стабильные и легко отлаживаются. E2E-тесты медленные, дорогие, хрупкие и сложны в отладке. Большее количество модульных тестов даёт более быструю обратную связь при меньших затратах, покрывая при этом большую часть логики.

**2. Что такое антипаттерн «Рожок мороженого» (Ice Cream Cone)?**
Перевёрнутая пирамида, где большинство тестов — E2E или ручные, а модульных почти нет. Приводит к медленному, хрупкому и дорогому набору тестов с плохой обратной связью для разработчиков.

**3. Что такое антипаттерн «Песочные часы» (Hourglass)?**
Форма, при которой много модульных и E2E-тестов, но почти нет интеграционных. Отсутствующий средний слой означает, что интеграционные баги не обнаруживаются до E2E, где отладка медленная и сложная.

**4. Что такое антипаттерн «Кекс» (Cupcake)?**
Когда одни и те же проверки дублируются на каждом уровне пирамиды — модульные, интеграционные и E2E тестируют одно и то же. Это тратит усилия и замедляет CI без дополнительной уверенности.

**5. Почему E2E-тесты более хрупкие, чем модульные?**
E2E-тесты зависят от всего стека — UI, бэкенд, база данных, сеть, сторонние сервисы. Изменение в любом слое может сломать тест. Модульные тесты зависят только от одной функции и потому гораздо стабильнее.

### Вопросы повышенного уровня

**1. Когда допустимо иметь больше E2E-тестов, чем предлагает пирамида?**
Когда система сильно завязана на UI со сложными пользовательскими сценариями, которые невозможно проверить на более низком уровне, или на ранних этапах разработки, когда архитектура ещё недостаточно стабильна для полноценных модульных тестов. Но это должно быть временным.

**2. Как бы вы исправили набор тестов «Рожок мороженого»?**
Добавить модульные тесты для всего нового кода, постепенно заменять E2E-тесты эквивалентами более низкого уровня, где возможно, и оставить E2E-тесты только для критических пользовательских сценариев. Вовлечь разработчиков в написание тестов, а не только QA.

**3. Если все модульные тесты проходят, можно ли быть уверенным, что система работает?**
Нет. Модульные тесты проверяют изолированные компоненты, а не их взаимодействие. Функция может отлично работать отдельно, но сломаться при соединении с другими модулями. Интеграционные и E2E-тесты нужны для проверки работы системы в целом.

**4. Пирамида тестирования — единственная верная модель?**
Нет. Пирамида хорошо работает для большинства приложений, но некоторые проекты используют вариации. Например, микросервисная архитектура может выиграть от «Алмаза тестирования» (Testing Diamond) с бóльшим количеством интеграционных тестов, а современные веб-приложения часто используют «Трофей тестирования» (Testing Trophy), делающий акцент на интеграционном слое. Пирамида — это ориентир, а не строгое правило.

**5. У команды 90% покрытие модульными тестами и ноль E2E-тестов. Хороший ли это набор тестов?**
Не обязательно. Высокое покрытие модульными тестами доказывает, что отдельные компоненты работают, но ничего не говорит о работоспособности системы для пользователей. Критические пользовательские сценарии всегда должны иметь хотя бы базовое E2E-покрытие. Процент покрытия сам по себе не измеряет качество.

**6. E2E-тесты постоянно падают случайным образом. Стоит ли их просто удалить?**
Нет. Нестабильные тесты — это симптом, а не причина. Сначала выясните, почему они падают — типичные причины: проблемы с таймингами, зависимости от тестовых данных или нестабильность окружения. Исправьте первопричину. Если проверку действительно невозможно выполнить на уровне E2E, перенесите её на более низкий уровень вместо удаления.

**7. Что значит опускать тесты вниз по пирамиде?**
Это значит переносить проверку на самый низкий надёжный уровень, где её можно выполнить. Например, правила валидации обычно проверяют unit-тестами, поведение API — интеграционными тестами, а на E2E оставляют только основной пользовательский сценарий.

**8. Почему интеграционный слой важен, если уже есть unit-тесты?**
Unit-тесты доказывают, что маленькие части работают отдельно. Интеграционные тесты доказывают, что эти части работают вместе: сервисы корректно вызывают друг друга, данные правильно сохраняются и читаются, а контракты между модулями соблюдаются.

**9. Почему дублирование покрытия на нескольких уровнях — проблема?**
Дублирующие проверки повышают стоимость поддержки и замедляют CI без большого прироста уверенности. Если одно и то же правило проверяется на unit-, интеграционном и E2E-уровне, одно изменение требования может потребовать обновления трёх тестов.

### Вопросы с кодом

**1.** Пайплайн команды выглядит как «Рожок мороженого»: большинство тестов — E2E/ручные, модульных почти нет, а CI выполняется часами.

```text
- Most tests: E2E + manual
- Integration: a few
- Unit tests: almost none
- CI runtime: hours, tests break on every CSS change
```

**Вопрос:** Определите антипаттерн и опишите два конкретных шага для его исправления.

**Ответ:** Это **Ice Cream Cone**. Исправляется так: (1) ввести модульные тесты для всего нового кода, опуская проверки логики в основание пирамиды, и (2) оставить E2E-тесты только для критических пользовательских сценариев, постепенно заменяя остальные тестами более низкого уровня (интеграционными).

---

**2.** Валидация логина сейчас тестируется на трёх уровнях:

```text
- E2E:        checks login validation
- Integration: checks login validation
- Unit:        checks login validation
```

**Вопрос:** Какой это антипаттерн и как его рекомендуется исправить?

**Ответ:** Это **Cupcake** — одна и та же проверка дублируется на каждом уровне. Исправление — определить чёткую стратегию, где каждый уровень тестирует то, что умеет лучше всего: модульные тесты покрывают логику валидации и граничные случаи, интеграционные — потоки данных и контракты, а E2E — только критический пользовательский сценарий. Удалите дублирующие проверки с верхних уровней, как только нижний уровень их уже покрывает.

**3.** Команда сообщает: *«У нас 90% покрытие модульными тестами, и все они проходят, но у нас ноль E2E-тестов.»*

```text
- Unit coverage: 90%
- Integration:   some
- E2E:           none
```

**Вопрос:** В чём главный риск этого набора тестов и что бы вы добавили?

**Ответ:** Риск — ложная уверенность: модульные тесты доказывают, что отдельные компоненты работают, но ничего не говорят о том, работает ли система в целом для реальных пользователей. Добавьте хотя бы базовое E2E-покрытие для критических пользовательских сценариев (например, полный поток покупки: вход → поиск → корзина → оформление → подтверждение заказа).
