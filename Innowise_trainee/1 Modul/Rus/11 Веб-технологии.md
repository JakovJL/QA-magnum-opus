## Сетевые модели

### Модель OSI

Модель OSI (Open Systems Interconnection) — концептуальная схема, описывающая, как данные передаются по сети. Состоит из 7 уровней, каждый из которых отвечает за конкретную задачу.

| Уровень | Название                     | Ответственность                   | Примеры            |
| ------- | ---------------------------- | --------------------------------- | ------------------ |
| 7       | Прикладной (Application)     | Пользовательские протоколы        | HTTP, FTP, SMTP    |
| 6       | Представления (Presentation) | Формат данных, шифрование         | TLS, JPEG, JSON    |
| 5       | Сеансовый (Session)          | Управление соединениями           | NetBIOS, RPC       |
| 4       | Транспортный (Transport)     | Надёжная доставка                 | TCP, UDP           |
| 3       | Сетевой (Network)            | Маршрутизация между сетями        | IP, ICMP           |
| 2       | Канальный (Data Link)        | Локальная доставка, MAC-адресация | Ethernet, Wi-Fi    |
| 1       | Физический (Physical)        | Передача битов                    | Кабели, радиоволны |

> [!warning] Направление данных
> При **отправке** данные проходят **сверху вниз** (добавляя заголовки на каждом уровне).
> При **получении** — **снизу вверх** (снимая заголовки на каждом уровне).

---

### Модель TCP/IP

Модель TCP/IP — практическая модель, используемая в интернете. Объединяет некоторые уровни OSI в 4 уровня.

| Уровень TCP/IP   | Соответствующие уровни OSI | Ключевые протоколы   |
| ---------------- | -------------------------- | -------------------- |
| Прикладной (Application)       | 5, 6, 7                    | HTTP, FTP, SMTP, DNS |
| Транспортный (Transport)     | 4                          | TCP, UDP             |
| Интернет (Internet)         | 3                          | IP, ICMP             |
| Сетевого доступа (Network Access) | 1, 2                       | Ethernet, Wi-Fi      |

> [!info] OSI vs TCP/IP
> OSI используется для **обучения** — описывает идеальную модель. TCP/IP используется **на практике** — описывает реальную работу интернета. QA-инженерам нужна OSI для теоретических обсуждений, а TCP/IP — для диагностики реальных сетевых проблем.

## Протоколы

### HTTP vs HTTPS

**HTTP** (HyperText Transfer Protocol) — основа обмена данными в вебе. Протокол типа запрос-ответ: клиент отправляет запрос, сервер возвращает ответ.

**HTTPS** = HTTP + шифрование TLS. Данные зашифрованы при передаче.

| | HTTP | HTTPS |
|---|---|---|
| Порт | 80 | 443 |
| Шифрование | Нет | Да (TLS) |
| Сертификат | Не нужен | Обязателен (SSL/TLS cert) |
| Применение | Внутренние инструменты, local development | Любой публичный сайт |

**Структура HTTP-запроса:**
- Метод (GET, POST, PUT, DELETE, PATCH)
- URL
- Заголовки (Content-Type, Authorization и др.)
- Тело (для POST/PUT)

**HTTP-методы и их назначение:**

| Метод  | Назначение                   | Safe | Idempotent | Есть тело? |
| ------ | ---------------------------- | ---- | ---------- | ---------- |
| GET    | Чтение / получение данных    | Да   | Да         | Нет        |
| POST   | Создание нового ресурса      | Нет  | Нет        | Да         |
| PUT    | Замена ресурса (полностью)   | Нет  | Да         | Да         |
| PATCH  | Частичное обновление ресурса | Нет  | Нет        | Да         |
| DELETE | Удаление ресурса             | Нет  | Да         | Нет        |

- **Safe** означает, что запрос только читает данные и не меняет состояние сервера (GET — safe).
- **Idempotent** означает, что повтор одного и того же запроса даёт тот же эффект, что и однократное выполнение (GET, PUT, DELETE). POST *не* идемпотентен — отправка одного и того же POST дважды может создать два ресурса.
- Ключевое практическое отличие: **GET** передаёт параметры в строке запроса URL и не должен менять состояние, тогда как **POST** отправляет данные в теле запроса, чтобы создать или изменить что-либо. GET можно кэшировать и добавить в закладки; POST — нет.

**Структура HTTP-ответа:**
- Код статуса (200 OK, 404 Not Found, 500 Internal Server Error)
- Заголовки
- Тело (HTML, JSON и др.)

**Группы кодов статуса:**

| Диапазон | Значение | Примеры |
|---|---|---|
| 2xx | Успех | 200 OK, 201 Created, 204 No Content |
| 3xx | Перенаправление | 301 Moved Permanently, 302 Found |
| 4xx | Ошибка клиента | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found |
| 5xx | Ошибка сервера | 500 Internal Server Error, 503 Service Unavailable |

### TLS и SSL

**SSL** (Secure Sockets Layer) — устаревший протокол шифрования, признан небезопасным.
**TLS** (Transport Layer Security) — современная замена SSL. Актуальные стандарты: TLS 1.2 и 1.3.

Несмотря на устаревание SSL, термин «SSL-сертификат» по-прежнему широко используется и означает TLS-сертификат.

**Что обеспечивает TLS:**
- **Шифрование** — данные нельзя прочитать при передаче
- **Аутентификация** — сервер подтверждает свою личность через сертификат
- **Целостность** — данные нельзя изменить без обнаружения

---

### WebSocket

WebSocket — протокол, устанавливающий **постоянное двунаправленное соединение** между клиентом и сервером через одно TCP-соединение.

| | HTTP | WebSocket |
|---|---|---|
| Соединение | Новый запрос на каждый обмен | Одно соединение, остаётся открытым |
| Направление | Клиент → Сервер (запрос-ответ) | Оба направления одновременно |
| Накладные расходы | Высокие (заголовки на каждый запрос) | Низкие (небольшие фреймы) |
| Использование | Обычные веб-страницы | Чаты, live-счёт, дашборды реального времени |

---

### Протоколы электронной почты и передачи файлов

**SMTP** (Simple Mail Transfer Protocol) — используется для **отправки** электронной почты. Порт 25 (сервер-сервер) или 587 (клиент-сервер, с шифрованием).

**POP3** (Post Office Protocol v3) — используется для **скачивания** почты с сервера на локальный клиент. Обычно удаляет письма с сервера после скачивания. Порт 110 / 995 (TLS).

**IMAP** (Internet Message Access Protocol) — используется для **доступа** к почте, хранящейся на сервере. Письма синхронизируются между устройствами. Порт 143 / 993 (TLS). Современный стандарт.

**FTP** (File Transfer Protocol) — используется для передачи файлов между клиентом и сервером. Порт 21. Не шифруется. Безопасные альтернативы: FTPS, SFTP.

**XMPP** (Extensible Messaging and Presence Protocol) — открытый стандарт для мгновенных сообщений (использовался в WhatsApp, Jabber).

## Адресация

### DNS

**DNS** (Domain Name System) преобразует доменные имена в IP-адреса.

**Пример:** `google.com` → `142.250.74.46`

**Процесс разрешения DNS:**
1. Браузер проверяет локальный кэш
2. ОС проверяет свой кэш (файл hosts)
3. Запрос уходит к Recursive Resolver (обычно ISP или публичный резолвер, например 8.8.8.8)
4. Резолвер запрашивает Root Nameserver → TLD Nameserver (для `.com`) → Authoritative Nameserver
5. Authoritative Nameserver возвращает IP-адрес
6. Браузер устанавливает соединение с IP

**Основные типы DNS-записей:**

| Запись | Назначение | Пример |
| ------ | ---------- | ------ |
| A | Связывает домен с IPv4-адресом | `example.com → 93.184.216.34` |
| AAAA | Связывает домен с IPv6-адресом | `example.com → 2606:2800:220:1:248:1893:25c8:1946` |
| CNAME | Псевдоним: указывает один домен на другой | `www.example.com → example.com` |
| MX | Указывает почтовый сервер домена | `example.com → mail.example.com` |
| TXT | Хранит текстовые данные для проверки и настроек | `v=spf1 include:_spf.google.com ~all` |

---

### URI, URL и URN

**URI** (Uniform Resource Identifier) — общая концепция. Идентифицирует ресурс. URL и URN — разновидности URI.

**URL** (Uniform Resource Locator) — URI, который указывает, **где** находится ресурс и **как** к нему обратиться.

**URN** (Uniform Resource Name) — URI, который даёт ресурсу **постоянное имя**, независимо от местоположения.

> [!tip] URI vs URL vs URN
> | | URI | URL | URN |
> |---|---|---|---|
> | **Что это** | Общий идентификатор | Местоположение + способ доступа | Постоянное имя |
> | **Пример** | Оба варианта ниже | `https://example.com/page` | `urn:isbn:978-3-16-148410-0` |
> | **Меняется при перемещении ресурса?** | Зависит | Да | Нет |

**Структура URL:**

```
https://user:pass@example.com:443/path/to/page?key=value&k2=v2#section
│       │         │           │   │             │               │
схема   учётные   хост        порт путь          строка запроса  фрагмент
        данные
```

- **Схема:** протокол (`https`, `http`, `ftp`)
- **Хост:** доменное имя или IP
- **Порт:** необязателен, по умолчанию 80 (HTTP) или 443 (HTTPS)
- **Путь:** расположение ресурса на сервере
- **Строка запроса:** пары ключ-значение после `?`, разделённые `&`
- **Фрагмент:** якорь на странице, обрабатывается только браузером

## Клиентские технологии

### HTML

**HTML** (HyperText Markup Language) задаёт **структуру** и содержимое веб-страницы с помощью элементов (тегов).

**Применимость для QA:** Атрибуты HTML (`id`, `name`, `data-*`) используются как локаторы в автотестах и в шагах ручных тест-кейсов.

---

### CSS

**CSS** (Cascading Style Sheets) управляет **внешним видом** — вёрсткой, цветами, шрифтами, отступами.

**Применимость для QA:** Визуальные баги (неверные цвета, сломанная вёрстка, перекрывающиеся элементы) — это проблемы CSS. DevTools позволяет инспектировать и редактировать CSS в реальном времени.

---

### DOM

**DOM** (Document Object Model) — древовидное представление HTML-страницы в памяти браузера. JavaScript читает и изменяет DOM, обновляя страницу без перезагрузки.

**Применимость для QA:**
- Автотесты (Selenium, Playwright) взаимодействуют с элементами DOM
- Вкладка Elements в DevTools показывает живой DOM, а не исходный HTML
- Изменения DOM через JS могут отличаться от исходного кода страницы

---

### JavaScript

**JavaScript (JS)** добавляет **интерактивное поведение** — валидацию форм, динамический контент, вызовы API, анимации.

JS работает в браузере (клиентская сторона) и на сервере (Node.js).

**Применимость для QA:**
- Ошибки JS видны во вкладке Console в DevTools — всегда проверяйте её при ручном тестировании
- Динамический контент, загружаемый через JS, может отсутствовать в исходном коде страницы
- Гонки условий и асинхронное поведение — частые источники нестабильных багов

---

### Ajax

**Ajax** (Asynchronous JavaScript and XML) — техника отправки HTTP-запросов из браузера **без перезагрузки всей страницы**. Название историческое — современный Ajax использует JSON, а не XML.

**Как работает:** JS отправляет запрос к серверу в фоне, получает ответ и обновляет только нужную часть страницы.

**Применимость для QA:**
- Ajax-запросы видны во вкладке Network в DevTools (ищите XHR/Fetch запросы)
- Страница, выглядящая загруженной, может ещё иметь незавершённые Ajax-вызовы
- Данные ответа (JSON) можно посмотреть в Network → Response

## Браузер и DevTools

**Как браузер отображает страницу:**
1. DNS-запрос → получение IP-адреса
2. TCP + TLS handshake → установка защищённого соединения
3. HTTP GET → получение HTML
4. Разбор HTML → построение DOM
5. Разбор CSS → построение CSSOM
6. DOM + CSSOM → Render Tree
7. Layout → вычисление позиций элементов
8. Paint → отрисовка пикселей

**Вкладки DevTools, актуальные для QA:**

| Вкладка | Для чего использовать |
|---|---|
| **Elements** | Инспектировать и редактировать DOM и CSS в реальном времени. Находить локаторы элементов для автотестов. |
| **Console** | Видеть ошибки, предупреждения и логи JS. Выполнять JS-выражения. |
| **Network** | Инспектировать каждый HTTP-запрос и ответ — URL, метод, статус, заголовки, тело, тайминг. |
| **Application** | Просматривать и редактировать cookies, localStorage, sessionStorage. Очищать кэш. |
| **Sources** | Просматривать загруженные JS-файлы. Устанавливать точки останова для отладки. |
| **Performance** | Записывать и анализировать производительность рендеринга и скриптов. |

> [!tip] Рабочий процесс с вкладкой Network для тестирования API
> Открыть DevTools → вкладка Network → перезагрузить страницу или выполнить действие → кликнуть запрос в списке → проверить: код статуса, заголовки запроса, тело запроса, тело ответа.

## Хранилище

### Cache (Кэш)

**Кэш** хранит копии ресурсов (HTML, CSS, JS, изображения) локально, чтобы при следующем визите их не нужно было загружать повторно.

**Типы:**
- **Кэш браузера** — хранится на диске пользователя
- **CDN-кэш** — хранится на граничных серверах
- **Серверный кэш** — хранится на сервере (Redis, Memcached)

**Применимость для QA:** Кэшированный контент может вынуждать тестировать старую версию страницы. Используйте Ctrl+Shift+R (жёсткая перезагрузка) или очистку кэша в DevTools для принудительной свежей загрузки.

---

### Cookie

**Cookie** — небольшой фрагмент данных, который сервер отправляет браузеру. Браузер хранит его и отправляет обратно с каждым последующим запросом к тому же домену.

**Распространённые применения:**
- Session ID (поддерживает пользовательскую сессию после Authentication)
- Пользовательские настройки (язык, тема)
- Отслеживание и аналитика

**Атрибуты cookie:**

| Атрибут | Назначение |
|---|---|
| `Expires` / `Max-Age` | Срок действия. Если не указан — сессионный cookie. |
| `HttpOnly` | Запрещает JS читать cookie (безопасность) |
| `Secure` | Отправляется только через HTTPS |
| `SameSite` | Управляет отправкой между сайтами (Strict / Lax / None) |
| `Domain` / `Path` | Какие запросы включают этот cookie |

> [!tip] Cache vs Cookie
> | | Cache | Cookie |
> |---|---|---|
> | **Хранит** | Ресурсы (HTML, изображения, CSS) | Небольшие значения данных |
> | **Назначение** | Ускорение загрузки страниц | Поддержание состояния между запросами |
> | **Отправляется на сервер?** | Нет | Да, автоматически |
> | **Управляется** | Заголовками сервера | Сервером + клиентским JS |
> | **Просмотр в DevTools** | Вкладка Network (from disk/memory) | Вкладка Application |

---

### Session

**Session** — это server-side хранилище состояния пользователя во время работы с приложением.

**Цель:** позволить серверу “помнить” пользователя между запросами.

**Как это работает:**
- сервер создаёт session data
- браузер обычно хранит только session identifier в cookie
- по session ID сервер находит реальные данные на своей стороне

**Session vs Cookie:**
- данные session хранятся на сервере
- данные cookie хранятся на клиенте
- cookie часто хранит только session ID, а не всё состояние session

**Session timeout:** после периода неактивности или по истечении заданного времени session истекает, и пользователю может потребоваться войти заново.

**Применимость для QA:**
- проверить, что logout действительно уничтожает session
- проверить поведение после session timeout
- проверить, разрешены ли параллельные sessions
- проверить, что приложение не восстанавливает истёкшую session некорректно

> [!tip] Session vs Cookie
> Session обычно живёт на сервере, а cookie — в браузере. Во многих системах cookie хранит только session ID.

## Вопросы для собеседования

### Вопросы начального уровня

**1. Что такое DNS и что он делает?**
DNS (Domain Name System) преобразует доменные имена в IP-адреса. Например, он разрешает `google.com` в IP, чтобы браузер мог подключиться к серверу.

**2. Что такое DOM?**
DOM (Document Object Model) — это древовидное представление страницы в памяти браузера. JavaScript читает и изменяет DOM, обновляя страницу без перезагрузки. Автотесты взаимодействуют с элементами DOM.

**3. Что обеспечивает TLS?**
Шифрование (данные нельзя прочитать при передаче), Authentication сервера через сертификат и целостность (данные нельзя изменить без обнаружения).

**4. Что означают группы кодов статуса HTTP?**
2xx — успех (200 OK, 201 Created), 3xx — перенаправление (301, 302), 4xx — ошибка клиента (400, 401, 403, 404), 5xx — ошибка сервера (500, 503). Первая цифра сразу показывает категорию.

### Вопросы среднего уровня

**1. В чём разница между моделью OSI и моделью TCP/IP?**
OSI — это 7-уровневая концептуальная модель для обучения и описания того, как сети работают в теории. TCP/IP — практическая 4-уровневая модель, описывающая реальную работу интернета. QA использует OSI для теоретических обсуждений, а TCP/IP — для диагностики реальных сетевых проблем.

**2. В чём разница между HTTP и HTTPS?**
HTTP передаёт данные в открытом виде по порту 80. HTTPS — это HTTP плюс шифрование TLS по порту 443, и он требует сертификат. HTTPS защищает данные при передаче от чтения и изменения.

**3. В чём разница между URI, URL и URN?**
URI — общий идентификатор. URL — это URI, который указывает, где находится ресурс и как к нему обратиться (`https://example.com/page`). URN — это URI, дающий ресурсу постоянное имя независимо от местоположения.

**4. В чём разница между cache и cookie?**
Cache хранит копии ресурсов (HTML, CSS, изображения) для ускорения загрузки страниц и не отправляется на сервер. Cookie хранит небольшие значения данных для поддержания состояния и автоматически отправляется на сервер с каждым запросом.

**5. В чём разница между GET и POST?**
GET запрашивает данные и передаёт параметры в URL; он не должен менять состояние сервера. POST отправляет данные в теле запроса, чтобы что-то создать или изменить на сервере. GET кэшируется и идемпотентен; POST — нет.

**6. Какая вкладка DevTools наиболее полезна для QA и почему?**
Вкладка Network — она показывает каждый HTTP-запрос и ответ с URL, методом, кодом статуса, заголовками, телом и таймингом. Она позволяет QA проверять API-вызовы, находить упавшие запросы и инспектировать данные ответа при ручном тестировании.

### Вопросы повышенного уровня

**1. Страница выглядит полностью загруженной, но часть данных отсутствует. Как разобраться?**
Данные, вероятно, грузятся Ajax-запросом, который упал или ещё выполняется. Откройте DevTools → Network, отфильтруйте XHR/Fetch и проверьте статус и ответ запроса. Также DOM может отличаться от исходного кода страницы, потому что JS изменил его после загрузки.

**2. Сайт использует HTTPS, значит данные полностью защищены. Верно или нет?**
Неверно. HTTPS лишь шифрует данные при передаче. Он не защищает от слабых паролей, сломанного контроля доступа, XSS или утечки данных в ответах. Транспортная безопасность — это один слой, а не вся картина.

**3. В чём разница между cookie с атрибутом HttpOnly и обычным, и почему это важно для безопасности?**
Cookie с атрибутом HttpOnly не может быть прочитан JavaScript, что защищает его от кражи через XSS. Сессионные cookie обычно помечают HttpOnly. QA должен проверять, что у чувствительных cookie выставлены HttpOnly и Secure.

**4. Почему WebSocket может использоваться вместо обычных HTTP-запросов?**
WebSocket держит одно постоянное двунаправленное соединение открытым, так что сервер может проталкивать данные клиенту в реальном времени с низкими накладными расходами. Это подходит для чатов, live-счёта и дашбордов, где повторяющиеся HTTP-запросы были бы медленными и расточительными.

**5. Почему проблема DNS может выглядеть как баг приложения?**
Если домен не разрешается в правильный IP-адрес, браузер не сможет подключиться к серверу, даже если само приложение работает. QA должен отличать проблемы DNS/соединения от настоящих дефектов приложения.

**6. Почему DOM важнее исходного кода страницы для UI-автоматизации?**
Исходный код показывает первоначальный HTML, а DOM показывает текущее состояние страницы после изменений JavaScript. UI-автоматизация работает с живым DOM, поэтому отсутствующие или изменённые DOM-элементы могут ломать тесты.

**7. В чём разница между cache и cookie с точки зрения QA?**
Cache хранит ресурсы страницы для быстрой загрузки, поэтому устаревший cache может скрывать изменения UI или файлов. Cookie хранят небольшое состояние, часто сессию или настройки, поэтому проблемы с cookie могут влиять на логин, Authentication и пользовательское поведение.

### Вопросы с кодом

**1.** Вы выкатили исправление на production-сайт, но браузер всё ещё показывает старую (багованную) страницу.

```text
- Bug:   login button misaligned (CSS fix deployed)
- After deploy: QA still sees the old layout
- Other testers see the fix correctly
```

**Вопрос:** Какова наиболее вероятная причина и что нужно сделать, чтобы проверить новую версию?

**Ответ:** Браузер отдаёт **кэшированную копию** CSS/HTML. Сделайте жёсткую перезагрузку (Ctrl+Shift+R) или очистите кэш в DevTools → Application/Network. Заголовки Cache-Control (например, `max-age`) определяют, как долго ресурсы кэшируются, поэтому новая версия появится только после обхода кэша или истечения его срока.

---

**2.** Веб-страница загружается без видимых ошибок, но раздел со списком пользователей пуст.

```text
- Page renders, no JS errors in Console
- User list area: empty
- DevTools → Network → XHR/Fetch: GET /api/users → 500 Internal Server Error
```

**Вопрос:** С помощью DevTools как подтвердить первопричину и какая группа кодов статуса указывает, где кроется fault?

**Ответ:** Откройте **DevTools → Network**, отфильтруйте по **XHR/Fetch** и найдите Ajax-запрос, который загружает пользователей. Его статус — **500**, что относится к группе **5xx (ошибка сервера)** — это значит, что сбой на стороне сервера, а не в браузере. Посмотрите тело Response на сообщение об ошибке от сервера. Страница выглядела загруженной, потому что окружающий HTML отрисовался, а данные, запрошенные асинхронно, упали.

---

**3.** Вы проверяете безопасность сессионных cookie, которые устанавливает приложение.

```text
Set-Cookie: sessionId=abc123; Path=/; Secure
```

**Вопрос:** Этот cookie отправляется только по HTTPS, что хорошо. Какого атрибута не хватает и почему это важно?

**Ответ:** У cookie отсутствует атрибут **HttpOnly**. Без него JavaScript может прочитать cookie, что делает его уязвимым к краже через атаку **XSS**. Сессионные cookie должны быть помечены как `Secure` (только HTTPS), **так и** `HttpOnly` (без доступа из JS). QA должен проверять, что чувствительные cookie несут оба атрибута.
