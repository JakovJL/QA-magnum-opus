## Что такое API

**API** (Application Programming Interface) — это набор правил, по которым два приложения общаются между собой.

**Клиент** — тот, кто отправляет запрос (браузер, мобильное приложение, Postman).
**Сервер** — тот, кто обрабатывает запрос и возвращает ответ.

**Пример:** когда ты нажимаешь «Оплатить» в приложении, приложение отправляет запрос к API банка. Банк обрабатывает его и возвращает ответ.

> [!danger] Главное
> API — это контракт между клиентом и сервером: клиент отправляет ==запрос== — сервер возвращает ==ответ==.

## Виды API

### REST

**REST** (Representational State Transfer) — архитектурный стиль для построения API поверх HTTP, где всё является ресурсом, доступным по URL.

- **Формат:** JSON (иногда XML)
- **Протокол:** HTTP / HTTPS

**Ключевые принципы (архитектурные ограничения):**
- **Stateless** — каждый запрос содержит всю необходимую информацию; сервер не хранит состояние сессии между запросами. *Тестирование:* каждый запрос должен быть самодостаточным — не полагайся на предыдущий.
- **Uniform Interface** — все ресурсы по одним правилам: стандартные HTTP-методы, коды статусов, единая структура URL. *Тестирование:* предсказуемое поведение.
- **Resource-Based** — всё является ресурсом, идентифицированным URL; действия определяются HTTP-методами, а не названиями URL.
- Другие ограничения (менее критичны для QA): **Client-Server**, **Cacheable**, **Layered System**.

> [!caution] Частая ошибка
> ❌ `POST /createUser` — действие в URL
> ✅ `POST /users` — ресурс в URL, действие из метода

**Преимущества:**
- Простой и лёгкий — легко изучить и использовать
- Не зависит от языка — работает с любым стеком технологий
- Кешируемость — ответы можно кешировать для ускорения
- Огромная поддержка инструментов и сообщества (Postman, браузеры)

**Недостатки:**
- Over-fetching / under-fetching — часто получаешь больше или меньше данных, чем нужно
- Может потребоваться несколько запросов для загрузки связанных данных
- Нет строгого контракта — структура ответа может измениться без предупреждения

**Пример:** `GET /users/42` → `{ "id": 42, "name": "John" }`

**Когда использовать:** веб- и мобильные приложения, публичные API.

### SOAP

**SOAP** (Simple Object Access Protocol) — строгий протокол для обмена структурированными данными в XML на основе формального контракта (WSDL).

- **Формат:** только XML
- **Протокол:** HTTP, SMTP и другие

**Преимущества:**
- Строгий контракт (WSDL) — обе стороны знают точный формат
- Встроенная обработка ошибок и стандарты
- Высокая безопасность (WS-Security) — подходит для банков и госсектора
- Надёжность для транзакций (повторы, ACID)

**Недостатки:**
- Многословность — XML-сообщения большие и тяжёлые
- Медленнее REST и gRPC
- Сложность — трудно писать и тестировать вручную

**Пример:** запрос оборачивается в SOAP-конверт:
```xml
<soap:Envelope>
  <soap:Body>
    <getUser><id>42</id></getUser>
  </soap:Body>
</soap:Envelope>
```

### WSDL — контракт SOAP

**WSDL** (Web Services Description Language) — это XML-документ, который является контрактом SOAP-сервиса. Он описывает доступные операции (методы), схемы запроса и ответа, транспортный протокол и URL endpoint. Тестировщик использует WSDL, чтобы знать, какие запросы валидны и каким должен быть ответ — инструменты вроде SoapUI могут импортировать его и автоматически создавать готовые запросы.

### SOAP Fault

Когда SOAP-запрос завершается неудачей, ошибка возвращается внутри тела ответа в элементе `<soap:Fault>` (не только через HTTP-статус). Fault содержит:
- **`faultcode`** — код ошибки (напр. `soap:Client` для неверного запроса, `soap:Server` для ошибки сервера)
- **`faultstring`** — человекочитаемое сообщение об ошибке
- **`faultactor`** — кто вызвал ошибку (опционально)
- **`detail`** — специфичные для приложения детали ошибки (опционально)

```xml
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <soap:Fault>
      <faultcode>soap:Server</faultcode>
      <faultstring>User not found</faultstring>
    </soap:Fault>
  </soap:Body>
</soap:Envelope>
```

**Когда использовать:** корпоративные системы, банки, государственные сервисы, legacy-интеграции.

### gRPC

**gRPC** (Google Remote Procedure Call) — современный фреймворк для быстрого взаимодействия между сервисами с использованием бинарных данных поверх HTTP/2.

- **Формат:** Protocol Buffers (бинарный)
- **Протокол:** HTTP/2

**Преимущества:**
- Очень быстрый — маленькие бинарные сообщения
- Строгая типизация — контракт описан в файле `.proto`
- Поддержка стриминга (клиентского, серверного или двустороннего)
- Автоматическая генерация кода для многих языков

**Недостатки:**
- Не читается человеком — бинарный формат трудно отлаживать глазами
- Ограниченная поддержка в браузерах — в основном для backend-сервисов
- Сложнее тестировать вручную — нужны специальные инструменты и файлы `.proto`

**Пример `.proto` файла:**
```protobuf
syntax = "proto3";

message UserRequest {
  int32 id = 1;
}

message UserResponse {
  string name = 1;
  int32 age = 2;
}

service UserService {
  rpc GetUser(UserRequest) returns (UserResponse);
}
```

**Когда использовать:** микросервисы, внутреннее высокопроизводительное взаимодействие сервисов.

#### Типы RPC (4 вида)

**Unary** — клиент отправляет 1 запрос, сервер отвечает 1 раз.
```
Клиент: ----[запрос]---->
Сервер: <---[ответ]------
```
*Пример:* `rpc GetUser(UserRequest) returns (UserResponse)` — получить пользователя.

**Server Streaming** — клиент отправляет 1 запрос, сервер отвечает потоком сообщений.
```
Клиент: ----[запрос]---->
Сервер: <---[часть1]-----
         <---[часть2]-----
         <---[часть3]-----
```
*Пример:* `rpc SubscribeNews(NewsRequest) returns (stream NewsResponse)` — подписка на новости.

**Client Streaming** — клиент отправляет поток запросов, сервер отвечает 1 раз.
```
Клиент: ----[часть1]---->
         ----[часть2]---->
         ----[часть3]---->
Сервер: <---[ответ]-------
```
*Пример:* `rpc UploadLogs(stream LogEntry) returns (UploadStatus)` — загрузка логов по частям.

**Bidirectional Streaming** — оба отправляют потоки в произвольном порядке.
```
Клиент: ----[msg1]------>
         <---[reply1]----
         ----[msg2]------>
         ----[msg3]------>
         <---[reply2]----
```
*Пример:* `rpc Chat(stream ChatMessage) returns (stream ChatReply)` — чат.

#### Коды ошибок gRPC (важно для тестировщика)

| Код | Число | Описание | Аналог HTTP |
|---|---|---|---|
| OK | 0 | Успех | 200 |
| CANCELLED | 1 | Отменено клиентом (таймаут) | — |
| INVALID_ARGUMENT | 3 | Неверные аргументы | 400 |
| DEADLINE_EXCEEDED | 4 | Таймаут — сервер не уложился | 504 |
| NOT_FOUND | 5 | Ресурс не найден | 404 |
| PERMISSION_DENIED | 7 | Нет прав | 403 |
| UNIMPLEMENTED | 12 | Метод не реализован на сервере | 501 |
| INTERNAL | 13 | Внутренняя ошибка сервера | 500 |
| UNAVAILABLE | 14 | Сервис недоступен | 503 |

#### Инструменты для тестирования gRPC

| Инструмент | Описание |
|---|---|
| **grpcurl** | curl для gRPC — вызов методов из терминала |
| **grpcui** | Веб-интерфейс на основе grpcurl (как Postman) |
| **Postman** | Поддерживает gRPC с 2022 года (импорт .proto) |
| **BloomRPC** | Десктопный GUI-клиент |
| **Evans** | Интерактивная REPL из терминала |

#### Что тестировать в gRPC

- **Unary:** стандартные проверки — валидные/невалидные данные, auth, граничные значения
- **Streaming:** корректно ли закрывается стрим, что при разрыве соединения, порядок сообщений
- **Contract:** ответ соответствует `.proto` схеме
- **Таймауты:** deadline — что будет, если сервер не ответил
- **Коды ошибок:** правильный код для каждого сценария (INVALID_ARGUMENT, NOT_FOUND и т.д.)
- **Нагрузка:** как сервер держит множество стримов одновременно

**Пример негативного сценария:** после обновления микросервиса B клиент A начал получать `UNIMPLEMENTED`. Причина: в B удалили или переименовали RPC-метод, а A использует старый `.proto` файл. Предотвратить: версионирование `.proto` контрактов и CI-проверка совместимости.

> [!tip] REST vs SOAP vs gRPC
> | | REST | SOAP | gRPC |
> |---|---|---|---|
> | Формат | JSON | XML | Бинарный (Protobuf) |
> | Скорость | Средняя | Медленная | Быстрая |
> | Сложность | Низкая | Высокая | Средняя |
> | Контракт | Опционально (OpenAPI) | Строгий (WSDL) | Строгий (.proto) |
> | Стриминг | Нет | Нет | 4 вида |
> | Применение | Веб / мобайл | Корпоративные системы | Микросервисы |

## HTTP-методы

HTTP-методы определяют, какое действие клиент хочет выполнить над ресурсом.

### GET

**Цель:** получить данные с сервера. Ничего не изменяет.

**Пример:** `GET /users/42` — получить пользователя с id 42.

### POST

**Цель:** создать новый ресурс на сервере.

**Пример:** `POST /users` — создать нового пользователя. Тело содержит данные пользователя.

### PUT

**Цель:** полностью заменить существующий ресурс.

**Пример:** `PUT /users/42` — заменить все данные пользователя 42.

### PATCH

**Цель:** частично обновить существующий ресурс.

**Пример:** `PATCH /users/42` — обновить только email пользователя 42.

### DELETE

**Цель:** удалить ресурс.

**Пример:** `DELETE /users/42` — удалить пользователя с id 42.

### Прочие методы

- **HEAD** — как GET, но возвращает только заголовки, без тела. Используется, чтобы проверить существование ресурса или узнать его размер/дату без скачивания.
- **OPTIONS** — сообщает, какие методы разрешены для ресурса. Используется браузером в CORS preflight.
- **TRACE** — возвращает запрос обратно (эхо) для отладки. Часто отключён из соображений безопасности.
- **CONNECT** — создаёт туннель (напр. HTTPS через прокси).

> [!tip] Методы — быстрое сравнение
> | Метод | Действие | Safe | Idempotent | Есть тело |
> |---|---|---|---|---|
> | GET | Чтение | ✅ | ✅ | Нет |
> | POST | Создание | ❌ | ❌ | Да |
> | PUT | Полное обновление | ❌ | ✅ | Да |
> | PATCH | Частичное обновление | ❌ | ❌* | Да |
> | DELETE | Удаление | ❌ | ✅ | Нет |
> | HEAD | Только заголовки | ✅ | ✅ | Нет |
> | OPTIONS | Список разрешённых методов | ✅ | ✅ | Нет |
> | TRACE | Эхо запроса (отладка) | ✅ | ✅ | Нет |
> | CONNECT | Создание туннеля | ❌ | ❌ | Нет |
>
> \*PATCH может быть идемпотентным в зависимости от реализации.
> **Safe** = не изменяет данные. **Idempotent** = повтор запроса даёт тот же результат.

## Коды статусов

Коды статусов HTTP сообщают клиенту, что произошло с запросом.

### 2xx — Успех

| Код | Название | Значение |
|---|---|---|
| 200 | OK | Запрос выполнен успешно |
| 201 | Created | Ресурс успешно создан |
| 202 | Accepted | Запрос принят, ещё обрабатывается |
| 204 | No Content | Успех, тело ответа отсутствует |

### 3xx — Перенаправление

| Код | Название | Значение |
|---|---|---|
| 301 | Moved Permanently | Ресурс перемещён навсегда |
| 302 | Found | Временное перенаправление |
| 304 | Not Modified | Кешированная версия актуальна (используется с ETag) |
| 307 | Temporary Redirect | Временное перенаправление, метод сохраняется |
| 308 | Permanent Redirect | Постоянное перенаправление, метод сохраняется |

### 4xx — Ошибки клиента

| Код | Название | Значение |
|---|---|---|
| 400 | Bad Request | Неверный формат или данные запроса |
| 401 | Unauthorized | Требуется Authentication |
| 403 | Forbidden | Аутентифицирован, но нет доступа |
| 404 | Not Found | Ресурс не существует |
| 405 | Method Not Allowed | Метод не поддерживается для этого ресурса |
| 408 | Request Timeout | Клиент слишком долго отправлял запрос |
| 409 | Conflict | Конфликт с текущим состоянием (напр. дубликат) |
| 415 | Unsupported Media Type | Формат тела не поддерживается (неверный Content-Type) |
| 422 | Unprocessable Entity | Формат верный, но данные невалидны |
| 429 | Too Many Requests | Превышен лимит запросов (rate limit) |

### 5xx — Ошибки сервера

| Код | Название | Значение |
|---|---|---|
| 500 | Internal Server Error | Что-то сломалось на сервере |
| 501 | Not Implemented | Метод не поддерживается сервером |
| 502 | Bad Gateway | Сервер получил неверный ответ от вышестоящего |
| 503 | Service Unavailable | Сервер недоступен или перегружен |
| 504 | Gateway Timeout | Вышестоящий сервер не ответил вовремя |

> [!caution] 401 vs 403
> **401** — «Кто ты?» — не авторизован или неверный токен.
> **403** — «Я знаю, кто ты, но у тебя нет доступа.»

## Структура запроса

Каждый HTTP-запрос состоит из следующих частей:

### URL

Адрес ресурса.

`https://api.example.com/users/42?format=json`

- `https://api.example.com` — базовый URL
- `/users/42` — endpoint + ==path parameter== (id ресурса)
- `?format=json` — ==query parameter==

> [!info] URL vs URI vs URN
> **URI** — общий идентификатор ресурса. **URL** — это URI, который также указывает, *где* ресурс и *как* до него добраться (напр. `https://site.com/users/42`). **URN** называет ресурс без его местоположения (напр. `urn:isbn:0451450523`). Каждый URL является URI, но не каждый URI является URL.

### Headers (Заголовки)

Пары ключ-значение с метаданными о запросе.

**Часто используемые заголовки:**
- `Content-Type: application/json` — формат тела запроса
- `Authorization: Bearer <token>` — Authentication
- `Accept: application/json` — ожидаемый формат ответа

> [!tip] Полезные заголовки, которые стоит знать
> - **`Accept`** — content negotiation: клиент сообщает серверу, какой формат ответа он хочет (напр. `Accept: application/json`). Сервер возвращает наиболее подходящий формат.
> - **`Cache-Control`** и **`ETag`** — HTTP-кеширование. Сервер сохраняет ответ, чтобы тот же запрос не шёл к серверу снова. `Cache-Control` указывает, как долго кешировать; `ETag` — метка версии — если данные не изменились, сервер отвечает **304 Not Modified**.
> - **`Access-Control-Allow-Origin`** — CORS (Cross-Origin Resource Sharing). Правило браузера, которое контролирует, может ли страница с одного домена вызывать API на другом домене. Сервер разрешает это этим заголовком.
> - **`Strict-Transport-Security`** — HSTS. Указывает браузеру всегда использовать HTTPS и никогда обычный HTTP, защищая от downgrade-атак.

### Body (Тело)

Данные, отправляемые вместе с запросом. Используется в POST, PUT, PATCH.

```json
{
  "name": "John",
  "email": "john@example.com"
}
```

Запросы GET и DELETE обычно не имеют тела.

## Authentication methods

Authentication подтверждает личность клиента перед доступом к ресурсу.

### Basic Auth

Логин и пароль, закодированные в Base64, передаются в заголовке `Authorization`.

```
Authorization: Basic dXNlcjpwYXNz
```

**Недостатки:** пароль виден, если соединение не зашифровано. Использовать только с HTTPS.

---

### API Key

Уникальный ключ, идентифицирующий клиента. Передаётся в заголовке или query-параметре.

```
Authorization: ApiKey abc123xyz
```

**Когда использовать:** простые интеграции, публичные API с контролем доступа.

---

### Bearer Token (JWT)

Токен, выданный после входа. Передаётся в заголовке каждого запроса.

```
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
```

**Когда использовать:** большинство современных REST API. Токен содержит информацию о пользователе и время истечения.

---

### OAuth 2.0

Протокол, позволяющий стороннему приложению получить доступ к ресурсам от имени пользователя без передачи пароля.

**Пример:** «Войти через Google» — ты даёшь приложению разрешение читать твой профиль, а не передаёшь пароль.

---

> [!tip] Сравнение методов Authentication
> | Тип | Безопасность | Сложность | Применение |
> |---|---|---|---|
> | Basic Auth | Низкая | Низкая | Устаревшие системы |
> | API Key | Средняя | Низкая | Публичные API |
> | Bearer Token | Высокая | Средняя | Современные REST API |
> | OAuth 2.0 | Высокая | Высокая | Доступ от имени пользователя |

## Основы Postman

**Postman** — инструмент для отправки HTTP-запросов и тестирования API без написания кода.

### Отправка запроса

1. Выбери HTTP-метод (GET, POST и т.д.)
2. Введи URL
3. Добавь заголовки, если нужно (например, `Authorization`)
4. Добавь тело для POST / PUT / PATCH запросов
5. Нажми **Send**
6. Проверь ответ: код статуса, тело, заголовки, время ответа

### Collections (Коллекции)

**Collection** — группа сохранённых запросов, организованных по функции или endpoint.

**Применение:** сохраняй все запросы проекта в одну коллекцию. Делись с командой.

### Environments (Окружения)

**Environment** — набор переменных, используемых во всех запросах.

**Пример:** переменная `{{base_url}}` = `https://api.dev.example.com`. Переключись на продакшн, изменив одно значение — все запросы обновятся автоматически.

### Authentication в Postman

Postman поддерживает основные методы Authentication во вкладке **Authorization**:
- Basic Auth — введи логин и пароль
- Bearer Token — вставь токен
- OAuth 2.0 — Postman обрабатывает весь процесс

## Что тестировать в API

### Позитивные тесты

Проверяем, что API работает корректно с валидными данными.

- Правильный метод + валидные данные → ожидаемый код статуса (200, 201)
- Тело ответа соответствует ожидаемой структуре и типам данных
- Время ответа в допустимых пределах

### Негативные тесты

Проверяем, что API корректно обрабатывает ошибки.

- Отсутствует обязательное поле → 400 Bad Request
- Неверный формат данных → 400 или 422
- Нет authentication token → 401 Unauthorized
- Валидный токен, неверная роль → 403 Forbidden
- Несуществующий ресурс → 404 Not Found

### Граничные тесты

Проверяем граничные значения.

- Максимально допустимая длина строки
- Минимальные и максимальные числовые значения
- Пустая строка vs null

### Auth-тесты

- Запрос без токена → 401
- Запрос с истёкшим токеном → 401
- Запрос с токеном другой роли → 403

## Модель OSI

**OSI** (Open Systems Interconnection) — это модель, которая описывает 7 уровней взаимодействия устройств в сети. Каждый уровень отвечает за свою часть передачи данных.

```
7. Прикладной  (Application)  — HTTP, FTP, DNS, SMTP
6. Представления (Presentation) — шифрование, сериализация, кодировка
5. Сеансовый   (Session)      — управление сессией, открытие/закрытие соединения
4. Транспортный (Transport)   — TCP, UDP — доставка данных
3. Сетевой      (Network)     — IP — маршрутизация пакетов
2. Канальный    (Data Link)   — Ethernet — передача между устройствами
1. Физический   (Physical)    — кабели, сигналы, Wi-Fi
```

### Для тестировщика — что важно

| Уровень | Что проверять |
|---|---|
| **7. Прикладной** | HTTP-запросы, статус коды, тело ответа, заголовки |
| **6. Представления** | Кодировка (UTF-8, Base64), шифрование (HTTPS), JSON/XML парсинг |
| **5. Сеансовый** | Таймауты сессии, keep-alive, повторное подключение |
| **4. Транспортный** | TCP port открыт? TCP handshake успешен? |
| **3. Сетевой** | Доступен ли сервер (ping)? Правильный ли IP? |

> [!info] OSI vs TCP/IP
> OSI — это теоретическая модель (7 уровней). На практике интернет работает по модели TCP/IP (4 уровня). OSI удобна для понимания, TCP/IP — для реальной работы.

## TCP/IP

**TCP/IP** — стек протоколов, на котором работает весь интернет. 4 уровня вместо 7 у OSI.

| Уровень | Протоколы | Задача |
|---|---|---|
| **4. Application** | HTTP, HTTPS, DNS, FTP, SMTP, gRPC | Данные приложения |
| **3. Transport** | **TCP**, UDP | Доставка между приложениями |
| **2. Internet** | **IP**, ICMP (ping) | Маршрутизация — как пакет дойдёт |
| **1. Network Access** | Ethernet, Wi-Fi | Железо — провода, радиосигнал |

### TCP — надёжная доставка

TCP гарантирует, что данные дойдут в правильном порядке и без потерь.

**3-Way Handshake (рукопожатие) — как открывается соединение:**
```
Клиент:  SYN — "Привет, ты там?"
Сервер:  SYN-ACK — "Да, я тут"
Клиент:  ACK — "Ок, погнали"
         === Соединение открыто ===
         HTTP GET /users →
         ← HTTP 200 {данные}
         === Соединение закрывается ===
Клиент:  FIN — "Всё, пока"
Сервер:  ACK — "Пока"
```

### UDP — быстрая, но ненадёжная

UDP не проверяет доставку — отправляет и забывает. Используется в DNS, стриминге, играх, VoIP. Для API-тестирования неактуален.

### Порты

**IP-адрес** = дом, **порт** = дверь.

| Порт | Протокол | Сервис |
|---|---|---|
| 80 | HTTP | Веб-сервер |
| 443 | HTTPS | Веб-сервер с шифрованием |
| 22 | SSH | Удалённый доступ |
| 53 | DNS | Доменные имена |
| 3306 | MySQL | База данных |
| 5432 | PostgreSQL | База данных |
| 8080 | HTTP | Альтернативный / dev |

### Версии HTTP

- **HTTP/1.1** — текстовый протокол. В основном один запрос за раз на соединение (head-of-line blocking). По умолчанию с 1997 года.
- **HTTP/2** — бинарные фреймы вместо текста. **Мультиплексирование**: много запросов и ответов делят одно TCP-соединение одновременно. Сжатие заголовков (HPACK). Быстрее HTTP/1.1. Используется в gRPC.
- **HTTP/3** — работает поверх **QUIC** (построен на UDP) вместо TCP. Убирает блокировку на уровне соединения и подключается быстрее (меньше циклов обмена).

> [!info] Почему это важно для QA
> gRPC требует HTTP/2 из-за мультиплексирования и бинарных фреймов. Медленный первый запрос может быть TCP + TLS handshake, а не самим API.

### Persistent Connection (Keep-Alive)

По умолчанию HTTP/1.1 переиспользует одно TCP-соединение для нескольких запросов и ответов вместо открытия нового каждый раз. Это называется **persistent connection** (keep-alive). Это убирает задержку от повторных TCP-рукопожатий, поэтому страницы с большим количеством ресурсов грузятся быстрее. В HTTP/2 эта идея превращается в мультиплексирование.

### Для тестировщика

- **Порт не открыт** → connection refused. Проверяй, на тот ли порт стучишься
- **Таймаут** → сервер не отвечает на уровне TCP (проверь ping/firewall)
- **Keep-alive** — HTTP/1.1 по умолчанию держит TCP-соединение для нескольких запросов (быстрее)
- **Латентность** — каждое TCP-рукопожатие добавляет задержку. gRPC (HTTP/2) уменьшает её мультиплексированием

```bash
# Проверить, открыт ли порт
curl -v https://api.example.com:443

# Проверить доступность сервера (ICMP)
ping api.example.com
```

## DNS

**DNS** (Domain Name System) — это «телефонная книга» интернета. Он переводит доменные имена (`api.example.com`) в IP-адреса (`192.168.1.10`), по которым сервер реально находится.

### Как работает DNS (упрощённо)

```
1. Браузер:   "Каким IP у api.example.com?"
2. ↓
3. DNS-резолвер (обычно твоего провайдера)
4. ↓
5. .com → кто отвечает за .com (корневой DNS)
6. example.com → кто отвечает за example.com
7. api.example.com → A-запись: 192.168.1.10
8. ↑
9. Браузер: "Привет, 192.168.1.10, дай /users"
```

**На практике** DNS кешируется на каждом уровне, поэтому повторные запросы быстрее.

### Типы DNS-записей

| Тип | Что хранит | Пример |
|---|---|---|
| **A** | IPv4 адрес | `example.com → 93.184.216.34` |
| **AAAA** | IPv6 адрес | `example.com → 2606:2800:220::1` |
| **CNAME** | Алиас (псевдоним) | `www.example.com → example.com` |
| **MX** | Почтовый сервер | `@ → mail.example.com` |
| **TXT** | Текстовые данные | SPF, DKIM для проверки почты |

### Для тестировщика

- **DNS не резолвится** → `curl: Could not resolve host`. Проверь домен или пропиши `--resolve` временно
- **Задержка DNS** — первый запрос к API может быть медленнее из-за DNS-запроса. Для тестов производительности учитывай это
- **Кеширование DNS** — если изменился IP сервера, старый может кешироваться и тесты падают. Сбрасывай DNS-кеш: `ipconfig /flushdns` (Windows), `sudo systemd-resolve --flush-caches` (Linux)
- **TXT записи** — используются для верификации домена (проверка, что домен твой)
- **Проверить DNS руками:**

```bash
nslookup api.example.com
dig api.example.com
```

## Сериализация и десериализация

**Сериализация** — превращение объекта из памяти в формат для передачи по сети.
**Десериализация** — обратный процесс: из полученного формата обратно в объект.

### Как это работает

```java
// В памяти — Java объект
User user = new User("John", 30);

// Сериализация (объект → JSON для REST)
String json = "{\"name\":\"John\",\"age\":30}";
// → отправляем в HTTP-запросе

// На сервере — десериализация (JSON → объект)
User user = objectMapper.readValue(json, User.class);
// → теперь можно: user.getName(), user.getAge()
```

### Сериализация в разных протоколах

| Протокол | Формат | Размер | Читаемость | Скорость |
|---|---|---|---|---|
| **REST (JSON)** | Текстовый | Средний | ✅ Да | Средняя |
| **SOAP (XML)** | Текстовый | Большой | ✅ Да (тяжело) | Медленная |
| **gRPC (Protobuf)** | Бинарный | Маленький | ❌ Нет | Высокая |

**JSON** — `{"name":"John","age":30}` — человек читает, но весит больше
**XML** — `<name>John</name><age>30</age>` — ещё больше из-за тегов
**Protobuf** — `\x0a\x04John\x10\x1e` — нечитаем, но очень компактный

### JSON Syntax Rules

У JSON строгий синтаксис.

**Основные правила:**
- объекты используют `{}`
- массивы используют `[]`
- строки пишутся в двойных кавычках
- числа пишутся без кавычек
- допустимые типы: `string`, `number`, `boolean`, `null`, `object`, `array`
- комментарии запрещены
- `NaN`, `Infinity` и `undefined` не являются валидными JSON-значениями
- trailing commas запрещены

**Корректный пример:**

```json
{
  "name": "John",
  "age": 30,
  "active": true,
  "skills": ["API", "SQL"],
  "manager": null
}
```

### XML Syntax Rules

У XML тоже есть строгие правила структуры.

**Основные правила:**
- каждый открытый тег должен иметь закрывающий тег
- теги чувствительны к регистру
- значения атрибутов должны быть в кавычках
- документ должен иметь один корневой элемент
- спецсимволы нужно экранировать: `&lt;`, `&gt;`, `&amp;`, `&quot;`, `&apos;`

**Корректный пример:**

```xml
<user id="42">
  <name>John</name>
  <active>true</active>
</user>
```

### XSD

**XSD** (XML Schema Definition) — формальная схема для валидации структуры XML.

**Цель:** определить, какие элементы, атрибуты, типы данных и обязательные поля допустимы в XML-документе.

**Почему это важно:**
- валидирует структуру XML
- проверяет типы данных
- отмечает обязательные и необязательные поля
- помогает находить невалидные SOAP requests и responses

**Связь с WSDL:** в SOAP-сервисах WSDL описывает операции, а XSD описывает типы данных, используемые внутри этих операций.

**Минимальный пример:**

```xml
<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema">
  <xs:element name="user">
    <xs:complexType>
      <xs:sequence>
        <xs:element name="name" type="xs:string"/>
        <xs:element name="age" type="xs:int"/>
      </xs:sequence>
    </xs:complexType>
  </xs:element>
</xs:schema>
```

### Для тестировщика

1. **Проверка структуры** — после десериализации поля должны совпадать с ожидаемой схемой
2. **Граничные значения** — как сериализуется null? пустая строка? спецсимволы?
3. **Версии .proto** — если клиент и сервер используют разные версии `.proto`, десериализация сломается
4. **Кодировка** — кириллица, `"`, `<`, `>` в JSON/XML — правильно ли экранируются
5. **Производительность** — сериализация/десериализация большого XML (SOAP) занимает заметное время

## GraphQL

**GraphQL** — язык запросов к API, созданный Facebook (Meta). Вместо REST-подхода (разные endpoint'ы под разные данные), GraphQL даёт **один endpoint** и позволяет клиенту самому решать, какие данные получить.

### REST vs GraphQL

```
REST:
  GET /users → {id, name, email, age, address, phone, ...}
  → Всегда получаешь полный объект, даже если нужен только id и name

GraphQL:
  POST /graphql — запрос:
  { users { id name } }

  Ответ — только то, что запросил:
  { "data": { "users": [{ "id": 1, "name": "John" }] } }
```

### Основные понятия

| Термин | Описание | Аналог в REST |
|---|---|---|
| **Query** | Чтение данных (GET) | `GET /users` |
| **Mutation** | Изменение данных (создание, обновление, удаление) | `POST /users`, `PUT /users/1` |
| **Subscription** | Подписка на изменения в реальном времени (WebSocket) | Server-Sent Events |
| **Schema** | Описание всех доступных типов и запросов | OpenAPI / Swagger |
| **Resolver** | Функция, которая достаёт данные для поля | Controller / Service |

### Преимущества

- Клиент сам решает, какие поля получить — **нет over-fetching** (лишних данных)
- Можно получить связанные данные в одном запросе — **нет under-fetching** (нескольких запросов)
- Строгая типизация — всё описано в Schema (как WSDL для SOAP, но легче)

### Недостатки

- Сложность на бэкенде — нужно писать resolvers для каждого поля
- Кеширование — REST кеширует URL (`GET /users`), GraphQL все запросы на один URL
- Тяжёлые запросы — клиент может запросить вложенную структуру на 10 уровней (N+1 проблема)
- Сложнее тестировать — нужно понимать schema и GraphQL-синтаксис

### Пример запроса и ответа

```graphql
# Запрос
query {
  user(id: 42) {
    name
    email
    posts {
      title
      comments {
        text
      }
    }
  }
}
```

```json
// Ответ
{
  "data": {
    "user": {
      "name": "John",
      "email": "john@example.com",
      "posts": [
        {
          "title": "Hello World",
          "comments": [
            { "text": "Nice post!" }
          ]
        }
      ]
    }
  }
}
```

Мутация (изменение данных):
```graphql
mutation {
  createUser(name: "John", email: "john@test.com") {
    id
    name
  }
}
```

### Инструменты для тестирования GraphQL

| Инструмент | Описание |
|---|---|
| **GraphQL Playground / GraphiQL** | Встроенная IDE для запросов (обычно идёт с сервером) |
| **Postman** | Поддерживает GraphQL (переключи тип на GraphQL) |
| **Altair GraphQL Client** | Специализированный клиент |
| **Apollo Studio** | Платформа от создателей GraphQL |

### Что тестировать в GraphQL

- **Query:** запросили только нужные поля → вернулись только они
- **Mutation:** создали/обновили/удалили → проверить результат
- **Unauthorized query:** запрос без токена → ошибка
- **Глубоко вложенный запрос:** что если клиент запросил 10 уровней вложенности? Сервер не упадёт?
- **Nullable vs Required:** обязательное поле не пришло → правильная ошибка schema
- **Subscription:** открылся WebSocket, данные приходят в реальном времени

## Вопросы для собеседования

### Вопросы начального уровня

**1. Что такое API-тестирование?**
API-тестирование — вид тестирования ПО, который проверяет API напрямую — его функциональность, надёжность и безопасность — на уровне бизнес-логики, без графического интерфейса. Тестировщик отправляет запросы к endpoint'ам API и проверяет ответы: код статуса, тело, заголовки и время ответа.

**2. Почему API-тестирование важно?**
- API — основа современных приложений: веб, мобайл и другие сервисы зависят от них.
- Баги находятся раньше и дешевле — API-тесты можно запускать до готовности UI (shift-left).
- Быстрее и стабильнее UI-тестов — нет браузера, меньше нестабильности (flakiness).
- Лучше покрытие — бизнес-логику, граничные случаи и безопасность можно тестировать напрямую.
- Один API обслуживает много клиентов — один дефект может сломать веб, мобайл и партнёров сразу.

**3. Что такое автоматизация API-тестов и как она работает?**
Автоматизация API-тестов — это запуск API-тестов автоматически через скрипты или инструменты вместо ручной отправки каждого запроса. Как это работает:
1. Пишутся тест-скрипты, отправляющие запросы к endpoint'ам (напр. Postman/Newman, REST Assured, Karate или Python + requests/pytest).
2. Каждый тест отправляет запрос (метод, заголовки, тело) и проверяет ответ (код статуса, поля тела, схему, время ответа).
3. Тесты собираются в набор (suite) и запускаются автоматически — локально или в CI/CD-пайплайне (напр. Jenkins, GitHub Actions) на каждой сборке.

**4. Что такое XSD?**
XSD — это XML Schema Definition, формальная схема для валидации структуры XML, типов данных и обязательных полей.
4. Отчёт показывает, какие тесты прошли, а какие упали.

Преимущества: быстрая регрессия, воспроизводимость, отсутствие ручного труда и раннее обнаружение поломок.

**4. Какие бывают виды API-тестирования?**
- **Functional** — возвращает ли API правильные результаты для заданных входных данных?
- **Validation** — соответствует ли требованиям, с корректным поведением и данными?
- **Load / Performance** — как ведёт себя под большим числом запросов?
- **Security** — Authentication, Authorization, шифрование, атаки инъекций.
- **Reliability** — стабильное соединение и согласованные ответы.
- **Negative / Error** — корректная обработка невалидного ввода и кодов ошибок.
- **Integration** — совместная работа нескольких API.

**5. Что именно мы проверяем при API-тестировании?**
Для каждого запроса мы проверяем ответ и его поведение:
- **Status code** — корректный код для сценария (200, 201, 400, 401, 404, 500…).
- **Response body** — правильные данные, структура и типы; соответствие ожидаемой JSON-схеме.
- **Headers** — корректные `Content-Type`, кеширование и заголовки безопасности.
- **Response time** — запрос отвечает в допустимых пределах.
- **Обработка ошибок** — невалидный ввод возвращает правильный код и понятное сообщение.
- **Корректность данных** — значения совпадают с базой / бизнес-логикой (напр. созданный ресурс реально существует).
- **Authentication и Authorization** — доступ к ресурсу только у разрешённых пользователей (401 / 403).
- **Граничные и негативные случаи** — пустые, слишком длинные, null или неверного типа значения обрабатываются корректно.
- **Идемпотентность и побочные эффекты** — повторные вызовы ведут себя ожидаемо и не меняют данные нежелательно.

**6. Что такое API?**
API — это набор правил, по которым два приложения общаются между собой. Клиент отправляет запрос — сервер возвращает ответ.

**7. В чём разница между REST, SOAP и gRPC?**
REST использует JSON поверх HTTP — простой, лёгкий и широко используемый для веб/мобайл, но может получать лишние или недостающие данные (over/under-fetching). SOAP использует XML со строгим контрактом (WSDL) и сильной безопасностью — медленнее и тяжелее, применяется в банках и корпоративных системах. gRPC использует бинарный формат (Protocol Buffers) поверх HTTP/2 — очень быстрый и строго типизированный, используется в микросервисах, но не читается человеком.

**8. Какие HTTP-методы ты знаешь? Что делает каждый?**
GET — чтение, POST — создание, PUT — полное обновление, PATCH — частичное обновление, DELETE — удаление.

**9. Что означают коды статусов 200, 201, 400, 401, 403, 404, 500?**
200 — OK, 201 — Created, 400 — Bad Request, 401 — Unauthorized, 403 — Forbidden, 404 — Not Found, 500 — Internal Server Error.

**10. В чём разница между 401 и 403?**
401 — пользователь не аутентифицирован (нет токена или он неверный). 403 — пользователь аутентифицирован, но не имеет доступа.

**11. Из чего состоит REST-запрос?**
URL (базовый URL + endpoint + path/query параметры), HTTP-метод, заголовки и тело (для POST / PUT / PATCH).

**12. Какие методы Authentication ты знаешь?**
Basic Auth, API Key, Bearer Token (JWT), OAuth 2.0.

**13. В чём разница между Bearer Token и API Key?**
Bearer Token выдаётся после входа и содержит информацию о пользователе с временем истечения. API Key — статический ключ, идентифицирующий клиентское приложение.

**14. Что ты тестируешь в API-тестировании?**
Позитивные кейсы (валидные данные → корректный ответ), негативные кейсы (невалидные данные, отсутствующие поля → корректные коды ошибок), граничные значения и проверки Authentication/Authorization.

**15. Для чего используется Postman?**
Postman — инструмент для отправки HTTP-запросов и проверки ответов API без написания кода. Используется для исследовательского тестирования, сохранения коллекций запросов и управления окружениями.

**16. В чём разница между Authentication и Authorization?**
Authentication отвечает на вопрос «кто ты?» — проверяет личность (логин, токен). Authorization отвечает «что тебе разрешено?» — проверяет права. Статус 401 относится к Authentication, 403 — к Authorization.

**17. Что такое endpoint?**
Endpoint — конкретный URL, по которому доступен ресурс или действие API. В `GET /users/42` endpoint — это `/users/42`. Каждый endpoint обычно соответствует одному ресурсу и принимает определённые HTTP-методы.

**18. В чём разница между path-параметрами и query-параметрами?**
Path-параметры — часть пути URL и указывают на конкретный ресурс: `/users/42` (42 — path-параметр). Query-параметры идут после `?` и используются для фильтрации, сортировки или пагинации: `/users?role=admin&page=2`.

**19. В чём разница между PUT и POST?**
POST создаёт новый ресурс и не идемпотентен — два вызова создают два ресурса. PUT создаёт или полностью заменяет ресурс по известному URL и идемпотентен — два вызова дают одинаковый результат.

**20. Какие инструменты используются для API-тестирования?**
Postman и Insomnia (ручное/исследовательское), Newman (запуск коллекций Postman в CI), REST Assured и Karate (Java), Python + requests/pytest, SoapUI (SOAP), а также Swagger/OpenAPI для документации и проверки контракта.

### Вопросы среднего уровня

**1. Что такое HTTP и почему он stateless?**
HTTP (HyperText Transfer Protocol) — протокол для отправки запросов и ответов между клиентом и сервером. Он stateless — сервер ничего не помнит между запросами, поэтому каждый запрос должен нести всю нужную информацию (напр. токен).

**2. В чём разница между HTTP и HTTPS?**
HTTP передаёт данные открытым текстом. HTTPS — это HTTP поверх TLS/SSL: данные шифруются, их нельзя прочитать или изменить в пути, а личность сервера подтверждается сертификатом. HTTP использует порт 80, HTTPS — 443.

**3. Что такое TLS/SSL?**
TLS (Transport Layer Security; SSL — его старое название) — слой шифрования, который превращает HTTP в HTTPS. Он шифрует соединение и использует сертификаты, чтобы доказать подлинность сервера.

**4. Какие порты используют HTTP и HTTPS?**
HTTP по умолчанию использует порт 80; HTTPS — порт 443.

**5. В чём разница между HTTP/1.1, HTTP/2 и HTTP/3?**
- **HTTP/1.1** — текстовый, в основном один запрос за раз на соединение (head-of-line blocking).
- **HTTP/2** — бинарный, мультиплексирование (много запросов в одном соединении), сжатие заголовков — быстрее.
- **HTTP/3** — работает поверх QUIC (UDP) вместо TCP, что убирает блокировку на уровне соединения и ускоряет подключение.

**6. В чём разница между TCP и UDP?**
TCP надёжен и упорядочен — гарантирует доставку (используется HTTP/1.1 и HTTP/2). UDP быстрее, но не гарантирует доставку и порядок. HTTP/3 использует QUIC поверх UDP, чтобы получить скорость плюс собственную надёжность.

**7. Что такое persistent connection (keep-alive)?**
Keep-alive позволяет нескольким запросам и ответам переиспользовать одно TCP-соединение вместо открытия нового каждый раз, что уменьшает задержку. По умолчанию включён в HTTP/1.1.

**8. Что такое HTTP-кеширование?**
Кеширование сохраняет ответы, чтобы переиспользовать их без повторного обращения к серверу. Управляется заголовками `Cache-Control` (как долго кешировать) и `ETag` (метка версии — сервер может ответить 304 Not Modified, если ничего не изменилось).

**9. Как хранится состояние поверх stateless HTTP?**
Поскольку HTTP забывает между запросами, состояние хранят через cookies, серверные сессии или токены (JWT). Клиент отправляет их в каждом запросе, чтобы сервер знал, кто ты.

**10. Что такое HTTP-заголовки? Назови частые.**
Заголовки — пары ключ-значение с метаданными о запросе или ответе. Частые: `Content-Type`, `Authorization`, `Accept`, `User-Agent`, `Cache-Control`, `Set-Cookie`.

**11. В чём разница между URL, URI и URN?**
URI — общий идентификатор ресурса. URL — это URI, который ещё указывает, где ресурс и как до него добраться (напр. `https://site.com/users/42`). URN называет ресурс без его местоположения (напр. `urn:isbn:0451450523`). Каждый URL является URI.

**12. Что такое content negotiation?**
Content negotiation позволяет клиенту запросить конкретный формат ответа заголовком `Accept` (напр. `Accept: application/json`). Сервер возвращает наиболее подходящий формат.

**13. Что такое Protocol Buffers (protobuf)?**
Protocol Buffers — компактный бинарный формат данных от Google, используемый в gRPC. Структура данных описывается в файле `.proto`; сообщения меньше и быстрее парсятся, чем JSON, но не читаются человеком.

**14. В чём разница между PUT и PATCH?**
PUT заменяет весь ресурс — если пропустить поле, оно будет удалено или сброшено. PATCH обновляет только указанные поля.

**15. Почему gRPC быстрее REST?**
gRPC передаёт данные в компактном бинарном формате (Protocol Buffers) вместо текстового JSON и использует HTTP/2, который позволяет нескольким запросам делить одно соединение (мультиплексирование). Меньшие сообщения плюс более быстрый протокол дают выше скорость.

**16. Какие HTTP-методы идемпотентны?**
Идемпотентность означает, что многократный вызов запроса даёт тот же результат, что и однократный. GET, PUT и DELETE идемпотентны. POST — нет: каждый вызов обычно создаёт новый ресурс. PATCH может быть или не быть — зависит от реализации.

### Вопросы повышенного уровня

**1. Что такое CORS?**
CORS (Cross-Origin Resource Sharing) — правило браузера, которое контролирует, может ли веб-страница с одного origin (домена) обращаться к API на другом origin. Сервер разрешает это заголовками вроде `Access-Control-Allow-Origin`.

**2. Что такое HSTS?**
HSTS (HTTP Strict Transport Security) — заголовок, который указывает браузеру всегда использовать HTTPS для сайта и никогда обычный HTTP — защита от downgrade-атак.

**3. Что такое WSDL и зачем он тестировщику?**
WSDL (Web Services Description Language) — это XML-документ, который описывает контракт SOAP-сервиса: какие методы (operations) доступны, какие параметры принимает каждый метод, какой формат ответа, по какому протоколу обращаться и URL endpoint.

**Зачем WSDL тестировщику:**
- **Документация** — вся информация о сервисе в одном файле, не нужно искать по разным источникам
- **Генерация запросов** — SoapUI может импортировать WSDL и автоматически создать готовые запросы с правильной структурой
- **Проверка контракта** — если WSDL изменился (добавили/удалили метод, поменяли параметр), тесты нужно обновить — это сигнал для регрессии
- **Поиск скрытых методов** — в WSDL могут быть операции, не описанные в документации, но доступные для вызова
- **Валидация ответа** — по WSDL можно проверить, соответствует ли ответ заявленной схеме

**Пример структуры WSDL:**
```xml
<wsdl:definitions>
  <wsdl:types>      <!-- типы данных -->
  <wsdl:message>    <!-- сообщения (запрос/ответ) -->
  <wsdl:portType>   <!-- операции (методы) -->
  <wsdl:binding>    <!-- протокол и формат -->
  <wsdl:service>    <!-- URL сервиса -->
</wsdl:definitions>
```

**4. Что проверять в SOAP-ответе?**
При тестировании SOAP-ответа нужно проверить:

1. **SOAP Envelope** — ответ обёрнут в `<soap:Envelope>` с корректным namespace (`xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/"`)
2. **SOAP Body** — внутри есть `<soap:Body>` с данными
3. **SOAP Fault** — если произошла ошибка, приходит `<soap:Fault>`:
   - `faultcode` — код ошибки (напр. `SOAP-ENV:Server`, `SOAP-ENV:Client`)
   - `faultstring` — текстовое описание ошибки
   - `detail` — детали ошибки (опционально)
4. **Данные ответа** — структура и типы полей соответствуют WSDL
5. **HTTP-статус**: 200 OK — успех, 500 Internal Server Error — SOAP Fault
6. **Content-Type** — должен быть `text/xml` или `application/soap+xml`
7. **Пространства имён (namespaces)** — все XML namespace'ы корректны

```xml
<!-- Пример успешного ответа -->
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <getUserResponse>
      <name>John</name>
      <age>30</age>
    </getUserResponse>
  </soap:Body>
</soap:Envelope>

<!-- Пример ответа с ошибкой (SOAP Fault) -->
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <soap:Fault>
      <faultcode>soap:Server</faultcode>
      <faultstring>User not found</faultstring>
    </soap:Fault>
  </soap:Body>
</soap:Envelope>
```

**5. Может ли GET иметь тело запроса?**
Технически да, но это не рекомендуется REST-конвенциями. GET предназначен только для чтения данных. Большинство серверов игнорируют тело GET-запроса.

**6. REST — это протокол?**
Нет. REST — это архитектурный стиль, набор ограничений. HTTP — это протокол.

**7. Код статуса 200 всегда означает, что тест прошёл?**
Нет. Сервер может вернуть 200 с сообщением об ошибке в теле ответа. Всегда проверяй и код статуса, и тело ответа.

**8. Безопасно ли передавать токен или пароль в query-параметрах?**
Нет. Query-параметры — часть URL, который часто логируется серверами и прокси, сохраняется в истории браузера и может кешироваться. Чувствительные данные должны идти в заголовках (напр. `Authorization`) или в теле запроса, всегда поверх HTTPS.

**9. Что произойдёт, если отправить DELETE для несуществующего ресурса?**
Зависит от API. Многие возвращают 404 Not Found. Некоторые возвращают 204 No Content, чтобы сохранить идемпотентность DELETE — конечное состояние (ресурса нет) одинаково в любом случае. Всегда проверяй ожидаемое поведение в документации API.

**10. Может ли POST-запрос быть идемпотентным?**
По умолчанию нет — два вызова POST обычно создают два ресурса. Но его можно сделать идемпотентным через «idempotency key»: клиент отправляет уникальный ключ, а сервер игнорирует повторные запросы с тем же ключом. Это распространено в платёжных API.

### Вопросы с кодом

**1. Логин `POST /login` возвращает HTTP 200, но тело `{"error": "invalid credentials"}` — это pass или fail?**
**Fail.** Статус 200 означает лишь, что сервер обработал запрос и ответил на уровне HTTP. Бизнес-результат — ошибка (`"invalid credentials"`), поэтому *функциональный* итог — провал. Для неверного пароля ожидаемое поведение обычно **401 Unauthorized** с понятным сообщением. Всегда проверяй и код статуса, *и* тело ответа — 200 с телом-ошибкой — частый API smell.

**2. Напиши минимальный валидный конверт SOAP-запроса для операции `GetUser`, которая принимает `id`.**
```xml
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <GetUser>
      <id>42</id>
    </GetUser>
  </soap:Body>
</soap:Envelope>
```
Ключевые моменты: корень — `<soap:Envelope>` с SOAP namespace; полезная нагрузка внутри `<soap:Body>`; имя операции (`GetUser`) оборачивает параметр (`id`). Ожидаемый ответ успеха — `<GetUserResponse>` внутри тела, а ошибка вернётся как `<soap:Fault>` (обычно с HTTP 500).

**3. Дан запрос `GET https://api.example.com/users/42`. Определи метод, path-параметр и безопасно ли добавить `?token=abc` для Authentication.**
- **Метод:** `GET` (безопасная, идемпотентная операция чтения).
- **Path-параметр:** `42` — идентифицирует конкретный ресурс `users` (`/users/{id}`).
- **`?token=abc`:** не рекомендуется. Query-строка — часть URL, поэтому она логируется серверами и прокси, сохраняется в истории браузера и может кешироваться. Лучше поместить токен в заголовок — `Authorization: Bearer abc` — и всегда использовать HTTPS.
