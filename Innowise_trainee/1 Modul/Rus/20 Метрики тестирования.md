## Что такое метрики тестирования

**Метрики тестирования** — это числа, которые помогают команде понимать прогресс тестирования, качество продукта, покрытие тестами и риски релиза.

**Цель:** сделать тестирование видимым и помогать команде принимать решения на основе фактов, а не только мнений.

**Пример:** «80% запланированных тест-кейсов выполнено» — это метрика тестирования. Она показывает прогресс, но не доказывает, что продукт готов к релизу.

> [!danger] Главное правило
> Метрика — это сигнал, а не окончательный ответ. Всегда анализируй метрики вместе с контекстом, рисками и знанием продукта.

---

## Зачем нужны метрики

Метрики помогают отвечать на важные вопросы:

- Успеваем ли мы с тестированием?
- В каких областях много дефектов?
- Какие требования покрыты тестами?
- Даёт ли автоматизация быстрый и стабильный feedback?
- Достаточно ли безопасно выпускать продукт?

**Преимущества:**
- Делают прогресс видимым для QA, разработчиков, менеджеров и product owners
- Помогают рано находить рискованные области
- Поддерживают решения о релизе
- Помогают улучшать процесс тестирования со временем

**Недостатки:**
- Могут дать ложную уверенность, если использовать их отдельно
- Могут искажаться, если использовать их для обвинения людей
- Могут скрыть реальные проблемы качества, если выбрана неправильная метрика

> [!tip] Хорошие метрики
> Хорошая метрика помогает принять решение. Перед сбором метрики спроси: «Что мы сделаем иначе, если это число изменится?»

## Метрики прогресса тестирования

Метрики прогресса показывают, какая часть запланированного тестирования уже выполнена.

| Метрика | Формула / значение | Что показывает |
|---|---|---|
| **Test execution progress** | выполненные тест-кейсы / запланированные тест-кейсы x 100% | Какая часть запланированного набора выполнена |
| **Pass rate** | passed test cases / executed test cases x 100% | Сколько выполненных тестов прошло успешно |
| **Fail rate** | failed test cases / executed test cases x 100% | Сколько выполненных тестов упало |
| **Blocked rate** | blocked test cases / planned test cases x 100% | Какая часть тестирования заблокирована |
| **Not run count** | тест-кейсы, которые ещё не выполнены | Оставшийся объём тестирования |

**Пример:** если выполнено 80 тестов из 100 запланированных, test execution progress = 80%.

**Когда использовать:** ежедневный статус тестирования, тестирование в спринте, релизное тестирование и отчёты о прогрессе.

> [!caution] Частая ошибка
> Высокий pass rate не всегда означает высокое качество. Тесты могут покрывать только простые сценарии или пропускать рискованные области.

## Метрики дефектов

Метрики дефектов показывают, где находятся дефекты, насколько они серьёзные и как быстро они исправляются.

| Метрика | Формула / значение | Что показывает |
|---|---|---|
| **Defect count** | количество найденных дефектов | Общий объём дефектов |
| **Defect density** | дефекты на модуль, функциональность, story или размер кода | Области с большим числом дефектов |
| **Defect severity distribution** | дефекты, сгруппированные по severity | Технический impact дефектов |
| **Defect priority distribution** | дефекты, сгруппированные по priority | Business urgency дефектов |
| **Defect fix rate** | исправленные дефекты / заведённые дефекты x 100% | Как быстро команда исправляет дефекты |
| **Defect reopen rate** | переоткрытые дефекты / закрытые дефекты x 100% | Качество исправлений и retesting |
| **Defect leakage** | дефекты, найденные после релиза | Баги, которые ушли за пределы тестирования |
| **Defect rejection rate** | отклонённые дефекты / заведённые дефекты x 100% | Качество defect reports или ясность требований |

**Пример:** если в модуле оплаты 15 дефектов, а в модуле профиля 2 дефекта, модуль оплаты, вероятно, более рискованный.

**Когда использовать:** risk analysis, defect triage, оценка готовности релиза и улучшение процесса.

→ см. [[07 Баг-репорт и жизненный цикл дефекта]]

## Метрики покрытия

Метрики покрытия показывают, какая часть продукта, требований или кода покрыта тестами.

| Метрика | Формула / значение | Что показывает |
|---|---|---|
| **Requirements coverage** | требования, покрытые хотя бы одним тестом / все требования x 100% | Проверяются ли требования |
| **Test case coverage** | созданные тест-кейсы / необходимые тест-кейсы | Насколько полный набор тестов спроектирован |
| **Risk coverage** | high-risk areas, покрытые тестами | Проверяются ли самые опасные области |
| **Code coverage** | код, выполненный автотестами | Какой код был затронут тестами |
| **Regression coverage** | важные regression areas, покрытые тестами | Насколько сильный regression suite |

**Пример:** 100% requirements coverage означает, что у каждого требования есть хотя бы один связанный тест-кейс. Это не означает, что проверены все возможные сценарии.

**Когда использовать:** test planning, RTM, выбор regression suite, планирование автоматизации и решения о релизе.

→ см. [[03 Тестовые артефакты]], [[06 Техники тест-дизайна]], [[10 Регрессионное тестирование]]

> [!warning] Ограничение покрытия
> 100% coverage не гарантирует отсутствие багов. Это означает только, что выбранная цель покрытия достигнута.

## Метрики автоматизации

Метрики автоматизации показывают, полезны ли автотесты, стабильны ли они и стоит ли их поддерживать.

| Метрика                           | Формула / значение                                               | Что показывает                                    |
| --------------------------------- | ---------------------------------------------------------------- | ------------------------------------------------- |
| **Automation coverage**           | automated test cases / total important test cases x 100%         | Какая часть важного тестирования автоматизирована |
| **Automated regression coverage** | automated regression checks / important regression checks x 100% | Какая часть регрессии запускается автоматически   |
| **Flaky test count**              | тесты, которые проходят и падают без изменений продукта          | Стабильность test suite                           |
| **False positive rate**           | failures не из-за product defects / все failures x 100%          | Шум в test suite                                  |
| **Execution time**                | время запуска test suite                                         | Скорость feedback                                 |
| **Feedback time in CI**           | время от commit до видимого результата                           | Как быстро команда узнаёт о проблемах             |
| **Maintenance effort**            | время на исправление тестов и тестовых данных                    | Стоимость поддержки автоматизации                 |

**Пример:** если у автоматизации pass rate 95%, но есть 30 flaky tests, suite ненадёжен. Команда может начать игнорировать failures.

**Когда использовать:** улучшение AQA-процесса, проверка здоровья CI, планирование regression automation и поддержка test framework.

→ см. [[16 Основы автоматизации тестирования]] и [[20 Архитектура фреймворка автотестов]]

## Метрики качества и релиза

Метрики качества и релиза помогают решить, готов ли продукт к выпуску.

| Метрика | Формула / значение | Что показывает |
|---|---|---|
| **Open critical defects** | незакрытые дефекты с высокой severity или priority | Риск блокировки релиза |
| **Escaped defects** | дефекты, найденные пользователями после релиза | Качество pre-release testing |
| **Customer-reported defects** | дефекты от пользователей или support | Реальный impact на пользователей |
| **Known issues count** | принятые незакрытые дефекты | Остаточный риск релиза |
| **Performance metrics** | response time, throughput, error rate, resource usage | Готовность по non-functional требованиям |
| **Security findings** | открытые security issues по severity | Security-риск релиза |

**Пример:** релиз может иметь pass rate 98%, но всё равно быть заблокирован одним критическим дефектом в оплате.

**Когда использовать:** release readiness meetings, go/no-go decisions и анализ после релиза.

→ см. [[17 Тестирование производительности]] и [[15 Основы тестирования безопасности]]

## Как правильно использовать метрики

Используй метрики как инструмент для решений, а не как таблицу очков.

Хорошие практики:

- Сначала определить цель, потом выбрать метрику.
- Использовать несколько метрик вместе.
- Совмещать числа с risk analysis и judgement тестировщика.
- Смотреть тренды со временем, а не только один snapshot.
- Объяснять, что метрика означает и чего она не означает.
- Использовать метрики для улучшения процесса, а не для обвинения людей.

**Пример:** `pass rate`, `requirements coverage`, `open critical defects` и `defect leakage` вместе дают более точную картину релиза, чем один `pass rate`.

> [!tip] Метрики в контексте
> Если pass rate растёт, но escaped defects тоже растут, тесты могут быть слабыми или сфокусированными не на тех областях.

---

## Частые ошибки

- **Использовать одну метрику отдельно.** Например, судить о качестве только по pass rate.
- **Путать прогресс и качество.** Выполнить много тестов не значит получить хороший продукт.
- **Измерять то, что легко, а не то, что важно.** Например, считать тест-кейсы без проверки risk coverage.
- **Использовать метрики для обвинения людей.** Тогда люди начинают скрывать проблемы вместо их решения.
- **Игнорировать тренды.** Один плохой день может быть шумом; повторяющийся паттерн — реальный сигнал.
- **Считать 100% coverage доказательством отсутствия багов.** Coverage показывает, что было затронуто, а не что всё работает правильно.

## Вопросы для собеседования

### Вопросы начального уровня

**1. Что такое метрики тестирования?**
Метрики тестирования — это числа, которые помогают команде понимать прогресс тестирования, качество продукта, покрытие тестами и риски релиза.

**2. Зачем нужны метрики тестирования?**
Они нужны, чтобы делать тестирование видимым, отслеживать прогресс, находить рискованные области, поддерживать решения о релизе и улучшать процесс тестирования.

**3. Какие метрики прогресса тестирования ты знаешь?**
Test execution progress, pass rate, fail rate, blocked rate и not run count.

**4. Что такое pass rate?**
Pass rate — это passed test cases, делённые на executed test cases и умноженные на 100%. Метрика показывает, сколько выполненных тестов прошло успешно.

**5. Что такое defect density?**
Defect density показывает, сколько дефектов найдено в модуле, функциональности, story или области кода. Она помогает находить рискованные части продукта.

---

### Вопросы среднего уровня

**1. Почему pass rate недостаточно для оценки качества продукта?**
Pass rate показывает только, сколько выполненных тестов прошло. Он не показывает, покрывают ли тесты рискованные области, проверены ли важные требования и есть ли открытые критические дефекты.

**2. В чём разница между requirements coverage и code coverage?**
Requirements coverage показывает, покрыты ли требования тест-кейсами. Code coverage показывает, какой код был выполнен автотестами. Обе метрики полезны, но ни одна не гарантирует отсутствие багов.

**3. Какие метрики автоматизации полезны?**
Полезны automation coverage, automated regression coverage, flaky test count, false positive rate, execution time, feedback time in CI и maintenance effort.

**4. Как defect metrics помогают в risk analysis?**
Они показывают, где появляется много дефектов, какие дефекты критичны, как часто исправления переоткрываются и какие дефекты уходят в production. Это помогает фокусировать тестирование на high-risk areas.

**5. Как команда должна правильно использовать метрики?**
Команда должна сначала определить цель, использовать несколько метрик вместе, анализировать их в контексте, смотреть тренды и применять их для улучшения процесса, а не для обвинения людей.

---

### Вопросы повышенного уровня

**1. Что такое defect leakage и почему это важно?**
Defect leakage — это количество дефектов, найденных после релиза. Метрика важна, потому что показывает, какие баги прошли через pre-release testing, и помогает улучшить будущую test strategy.

**2. Может ли 100% requirements coverage или 100% code coverage гарантировать качество?**
Нет. 100% requirements coverage означает, что у каждого требования есть хотя бы один тест. 100% code coverage означает, что код был выполнен тестами. Это не доказывает, что все сценарии, edge cases и риски проверены правильно.

**3. Что может означать высокий defect rejection rate?**
Он может означать, что defect reports неясные, есть дубликаты, требования неоднозначные или у команды нет общего понимания expected behavior.

**4. Какие метрики ты бы использовал для решения о готовности релиза?**
Я бы использовал test execution progress, pass/fail rate, requirements coverage, open critical defects, known issues, тренды defect leakage и важные non-functional metrics, например performance и security findings.

**5. Что значит, если pass rate высокий, но escaped defects тоже много?**
Это может означать, что тесты слабые, сфокусированы на low-risk areas, пропускают реальные пользовательские сценарии или не проверяют важные assertions. Команде нужно пересмотреть coverage, риски и test design.

### Вопросы с кодом

**1. По данным ниже посчитай test execution progress и pass rate.**

```text
Запланированные тест-кейсы: 120
Выполненные тест-кейсы: 90
Passed test cases: 72
Failed test cases: 18
```

**Ответ:**
- Test execution progress = 90 / 120 x 100% = 75%
- Pass rate = 72 / 90 x 100% = 80%

**2. У релиза такие метрики. Готов ли он?**

```text
Pass rate: 98%
Requirements coverage: 95%
Open critical defects: 1
Defect: payment fails for Visa cards
```

**Ответ:** Скорее всего, не готов. Pass rate высокий, но один открытый критический дефект в оплате может блокировать релиз. Метрики нужно читать вместе с business risk.

**3. У automation suite такие метрики. В чём главная проблема?**

```text
Pass rate: 96%
Flaky test count: 25
Feedback time in CI: 55 minutes
False positive rate: high
```

**Ответ:** Главная проблема — низкое доверие к автоматизации. Большое число flaky tests и false positives создаёт шум, а медленный CI feedback задерживает расследование. Команде нужно исправить или изолировать flaky tests и сократить feedback time.
