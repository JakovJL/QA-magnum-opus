## Основные понятия

**База данных** — организованная коллекция структурированных данных, хранящихся в электронном виде. Данные организованы так, чтобы их можно было эффективно извлекать, запрашивать и обновлять.

**DBMS** (Database Management System, система управления базами данных) — программное обеспечение, управляющее базой данных. Отвечает за хранение, извлечение, безопасность, параллельный доступ и резервное копирование.

**Примеры DBMS:** MySQL, PostgreSQL, SQLite, Oracle, Microsoft SQL Server, MongoDB, Redis.

**Зачем QA-инженеру нужны базы данных:**
- Проверить, что данные сохраняются корректно после действия пользователя
- Проверить целостность данных — нет дубликатов, нет «осиротевших» записей
- Подтвердить бизнес-логику через прямые запросы к данным
- Воспроизвести и исследовать баги, связанные с состоянием данных
- Понять сообщения об ошибках, в которых упоминаются таблицы или ограничения

## Типы баз данных

### Реляционные базы данных (SQL)

Данные организованы в **таблицах** с фиксированными столбцами и строками. Таблицы связаны между собой через отношения (внешние ключи). Используют SQL (Structured Query Language) для всех операций.

**Примеры:** MySQL, PostgreSQL, SQLite, Oracle, Microsoft SQL Server

**Подходит для:** Структурированных данных с чётко определёнными отношениями — аккаунты пользователей, заказы, транзакции, инвентарь.

---

### NoSQL базы данных

Гибкая схема — структура может отличаться для каждой записи. Несколько подтипов для разных задач.

| Подтип | Структура | Примеры | Подходит для |
|---|---|---|---|
| Документный | JSON-подобные документы | MongoDB, CouchDB | Профили пользователей, каталоги, контент |
| Ключ-Значение | Ключ → Значение | Redis, DynamoDB | Кэш, сессии, простые запросы |
| Колоночный | Ориентированные на столбцы | Cassandra, HBase | Временные ряды, аналитика, большой масштаб |
| Графовый | Узлы и рёбра | Neo4j, Amazon Neptune | Социальные графы, рекомендательные системы |

> [!tip] Реляционные vs NoSQL
> | | Реляционные (SQL) | NoSQL |
> |---|---|---|
> | **Схема** | Фиксированная — определяется заранее | Гибкая — может отличаться для каждой записи |
> | **Отношения** | Внешние ключи, JOIN | Вложенные данные или уровень приложения |
> | **Масштабирование** | Вертикальное (более мощный сервер) | Горизонтальное (больше серверов) |
> | **Язык запросов** | SQL (стандартизован) | Различается для каждой системы |
> | **Согласованность** | Строгая (ACID) | Часто согласованность в конечном счёте |
> | **Подходит для** | Сложные отношения, транзакции | Большой масштаб, эволюционирующая схема |

## Реляционные базы данных

### Ключевые понятия

**Таблица** — коллекция связанных данных, организованных в строки и столбцы. Аналог листа таблицы.

**Строка (Запись)** — отдельная запись в таблице. Представляет один экземпляр сущности.

**Столбец (Поле / Атрибут)** — свойство сущности. Каждая строка имеет одинаковые столбцы.

**Primary Key (PK, Первичный ключ)** — столбец (или комбинация столбцов), однозначно идентифицирующий каждую строку. Не может быть NULL, должен быть уникальным.

**Foreign Key (FK, Внешний ключ)** — столбец в одной таблице, ссылающийся на Primary Key другой таблицы. Создаёт отношение между таблицами.

**Индекс** — структура данных, ускоряющая извлечение данных. Без индекса запрос сканирует всю таблицу. Индексы ускоряют чтение за счёт замедления записи и дополнительного места.

**NULL** — отсутствие значения. Отличается от нуля или пустой строки. `NULL != NULL` в логике SQL.

**Пример — простая схема интернет-магазина:**

| Таблица users | Таблица orders |
|---|---|
| **id** (PK) | **id** (PK) |
| name | user_id (FK → users.id) |
| email | total_price |
| created_at | status |

`orders.user_id` ссылается на `users.id` — это и есть отношение.

### Основы SQL для QA

**SELECT** — извлечь данные из таблицы.

```sql
SELECT * FROM users;                          -- все столбцы, все строки
SELECT name, email FROM users;                -- конкретные столбцы
SELECT * FROM users WHERE email = 'a@b.com'; -- с фильтрацией
```

**WHERE** — фильтровать строки по условию.

```sql
SELECT * FROM orders WHERE status = 'pending';
SELECT * FROM orders WHERE total_price > 100;
SELECT * FROM users WHERE created_at >= '2024-01-01';
```

**JOIN** — объединить данные из двух таблиц через отношение.

```sql
-- Получить каждый заказ с именем пользователя
SELECT orders.id, users.name, orders.total_price
FROM orders
JOIN users ON orders.user_id = users.id;
```

**Типы JOIN:** Обычный `JOIN` (также называемый `INNER JOIN`) возвращает только те строки, которые имеют совпадение в *обеих* таблицах. `LEFT JOIN` сохраняет каждую строку из левой таблицы, даже если нет совпадения в правой (недостающие столбцы возвращаются как `NULL`). Для QA это важно: `INNER JOIN` может незаметно отбросить строки без совпадения, что скроет отсутствующие или осиротевшие данные. JOIN также может породить **дублирующиеся строки**, если ключ соединения не уникален, поэтому всегда проверяйте уникальность ключа перед тем, как доверять объединённому результату.

**ORDER BY / LIMIT** — сортировка и ограничение результатов.

```sql
SELECT * FROM orders ORDER BY created_at DESC LIMIT 10;
```

**COUNT / GROUP BY** — агрегация данных.

```sql
SELECT status, COUNT(*) FROM orders GROUP BY status;
```

**INSERT / UPDATE / DELETE** — изменение данных (использовать осторожно в тестовых средах).

```sql
INSERT INTO users (name, email) VALUES ('Алиса', 'alice@example.com');
UPDATE orders SET status = 'shipped' WHERE id = 42;
DELETE FROM users WHERE id = 99;
```

> [!caution] Запись в базу данных при тестировании
> Всегда проверяйте, что вы работаете с нужным окружением, перед запуском INSERT/UPDATE/DELETE. Случайный запуск на продакшн-базе может привести к потере данных. Используйте транзакции (`BEGIN` / `ROLLBACK`) при тестировании разрушительных запросов.

## MongoDB

### Документная модель

MongoDB — **документная база данных** — данные хранятся в виде JSON-подобных документов (внутренний формат BSON — Binary JSON).

| Понятие в реляционных БД | Аналог в MongoDB |
|---|---|
| База данных | База данных |
| Таблица | Коллекция (Collection) |
| Строка | Документ (Document) |
| Столбец | Поле (Field) |
| Primary Key | Поле `_id` (генерируется автоматически) |
| JOIN | `$lookup` в агрегации / вложенные документы |

**Документ** — гибкий JSON-объект. Поля могут различаться между документами, поддерживаются вложенные объекты и массивы.

```json
{
  "_id": "64a3f1c2e3b4d5e6f7890123",
  "name": "Алиса",
  "email": "alice@example.com",
  "orders": [
    { "item": "Ноутбук", "price": 1200 },
    { "item": "Мышь", "price": 25 }
  ]
}
```

**Преимущества перед реляционными БД:**
- Не нужна миграция схемы при добавлении полей
- Вложенные данные избавляют от сложности JOIN
- Горизонтальное масштабирование по дизайну

**Недостатки:**
- Нет JOIN по умолчанию — отношения реализуются в коде приложения или через `$lookup`
- ACID-транзакции между коллекциями ограничены (добавлены в MongoDB 4.0)
- Дублирование данных — распространённая практика

---

### Базовые запросы

```javascript
// Найти все документы в коллекции
db.users.find()

// Найти с фильтром
db.users.find({ email: "alice@example.com" })

// Найти с несколькими условиями
db.orders.find({ status: "pending", total: { $gt: 100 } })

// Найти один документ
db.users.findOne({ name: "Алиса" })

// Вставить документ
db.users.insertOne({ name: "Боб", email: "bob@example.com" })

// Обновить документ
db.users.updateOne({ name: "Боб" }, { $set: { email: "new@example.com" } })

// Удалить документ
db.users.deleteOne({ name: "Боб" })

// Подсчитать документы
db.orders.countDocuments({ status: "shipped" })
```

**Основные операторы запросов:**

| Оператор | Значение | Пример |
|---|---|---|
| `$eq` | Равно | `{ age: { $eq: 30 } }` |
| `$gt` / `$gte` | Больше / больше или равно | `{ price: { $gt: 100 } }` |
| `$lt` / `$lte` | Меньше / меньше или равно | `{ price: { $lt: 50 } }` |
| `$ne` | Не равно | `{ status: { $ne: "cancelled" } }` |
| `$in` | Значение входит в список | `{ status: { $in: ["pending", "processing"] } }` |
| `$exists` | Поле существует | `{ phone: { $exists: true } }` |

## Вопросы для собеседования

### Вопросы начального уровня

**1. В чём разница между базой данных и DBMS?**
База данных — это организованная коллекция данных. DBMS (Database Management System) — это ПО, которое ею управляет: хранение, извлечение, безопасность, параллельный доступ, резервное копирование. MySQL и MongoDB — это DBMS; данные, которые они хранят, — это база данных.

**2. Что такое первичный ключ и внешний ключ?**
Первичный ключ (Primary Key) однозначно идентифицирует каждую строку в таблице и не может быть NULL. Внешний ключ (Foreign Key) — это столбец, ссылающийся на первичный ключ другой таблицы, создавая отношение между двумя таблицами.

**3. Что делает JOIN?**
JOIN объединяет строки из двух или более таблиц по связанному столбцу. Например, соединив `orders` и `users` по `user_id`, можно показать каждый заказ вместе с именем пользователя, который его оформил.

**4. Что означает NULL в SQL?**
NULL означает отсутствие значения — это не ноль и не пустая строка. В логике SQL `NULL` не равен `NULL`, поэтому для сравнений с NULL нужны `IS NULL` / `IS NOT NULL`.

**5. Назовите несколько распространённых инструментов для работы с БД, которые может использовать QA.**
DBeaver (GUI для многих БД), MySQL Workbench, pgAdmin (PostgreSQL), TablePlus и MongoDB Compass для MongoDB. Они позволяют просматривать данные, выполнять запросы и инспектировать схему.

### Вопросы среднего уровня

**1. Зачем QA-инженеру знать SQL?**
Чтобы проверять, что данные сохраняются корректно после действия, проверять целостность данных (нет дубликатов и осиротевших записей), подтверждать бизнес-логику прямыми запросами и воспроизводить или исследовать баги, связанные с данными.

**2. В чём разница между WHERE и JOIN?**
WHERE фильтрует строки по условию внутри набора результатов. JOIN объединяет столбцы из нескольких таблиц по отношению. Их часто используют вместе: JOIN соединяет таблицы, WHERE фильтрует объединённый результат.

**3. В чём разница между SQL и NoSQL базами данных?**
SQL (реляционные) базы хранят данные в таблицах с фиксированной схемой и используют SQL и JOIN. NoSQL базы имеют гибкую схему и бывают типов: документные, ключ-значение, колоночные, графовые. SQL подходит для структурированных данных с отношениями; NoSQL — для большого масштаба или меняющихся данных.

**4. Чем MongoDB отличается от реляционной базы данных?**
MongoDB — документная база: она хранит JSON-подобные документы в коллекциях вместо строк в таблицах. Её схема гибкая, поля могут различаться между документами, а связанные данные часто вкладывают, а не соединяют через JOIN.

**5. Что такое индекс и в чём компромисс?**
Индекс — это структура данных, ускоряющая извлечение данных, чтобы запрос не сканировал всю таблицу. Компромисс — более медленная запись и дополнительное место, потому что индекс нужно обновлять и хранить.

### Вопросы повышенного уровня

**1. Когда NoSQL будет худшим выбором, чем SQL?**
Когда у данных много сильных отношений и нужны транзакции между ними — например, банкинг или заказы со строгой согласованностью. Реляционные базы обеспечивают это внешними ключами и ACID-транзакциями, тогда как большинство NoSQL-хранилищ предпочитают гибкость строгой согласованности.

**2. Как JOIN может скрыть баг?**
JOIN может породить дублирующиеся строки, если ключ соединения не уникален, или потерять строки, если использовать INNER JOIN там, где нужен был LEFT JOIN. QA должен понимать тип соединения и уникальность ключа, чтобы доверять результату.

**3. В MongoDB одно и то же поле — строка в одном документе и число в другом. Это допустимо и почему рискованно?**
Да, гибкая схема MongoDB это допускает. Это рискованно, потому что запросы и приложение могут ожидать один тип, поэтому смешанные типы вызывают тонкие баги, неудачные сравнения или падения. Чтобы этого избежать, нужна валидация схемы или проверки на уровне приложения.

### Вопросы с кодом

**1. Сценарий — пользователь сообщает, что его заказ пропал из UI. Подтвердите, существуют ли данные на самом деле в базе данных.**

```sql
SELECT * FROM orders WHERE user_id = :user_id;
```

Как бы вы интерпретировали результат, чтобы локализовать дефект?

**Ответ:** Если строка существует, баг в слое UI или API; если нет — данные никогда не сохранялись. Прямой запрос к базе локализует, где на самом деле находится дефект.

**2. Даны таблицы `users` и `orders`, напишите запрос, чтобы вывести каждый заказ вместе с именем пользователя.**

```sql
SELECT orders.id, users.name, orders.total_price
FROM orders
JOIN users ON orders.user_id = users.id;
```

**Ответ:** Этот запрос соединяет две таблицы по внешнему ключу `orders.user_id → users.id`, поэтому каждая строка заказа возвращается вместе с именем соответствующего пользователя.

**3. Вы случайно выполнили UPDATE без WHERE на тестовой базе. Что произошло и что нужно сделать в следующий раз, чтобы это предотвратить?**

```sql
UPDATE orders SET status = 'shipped';   -- обновлена каждая строка!
```

**Ответ:** UPDATE применился ко всем строкам. В следующий раз сначала пишите WHERE, проверяйте целевые строки через SELECT перед обновлением и оборачивайте разрушительные запросы в транзакцию (`BEGIN` / `ROLLBACK`), чтобы их можно было откатить.
