## Что такое автоматизация тестирования

Автоматизация тестирования — это использование программного обеспечения для автоматического выполнения тестов, сравнения результата с ожидаемым и сообщения о том, прошёл тест или упал, без ручного повторения шагов человеком.

**Цель:** Выполнять повторяемые проверки быстрее, чаще и надёжнее, чем это возможно вручную.

**Преимущества:**
- Быстро — сотни проверок за минуты
- Повторяемо — одни и те же шаги каждый раз, без человеческих ошибок
- Работает без участия человека — ночью, при каждом изменении кода
- Освобождает время QA для исследовательского и сложного тестирования

**Недостатки:**
- Требует времени и денег на создание и поддержку
- Не оценивает удобство, внешний вид и субъективное «ощущение»
- Тесты ломаются при изменении приложения (стоимость поддержки)
- Плохой тест даёт ложную уверенность (проходит, но проверяет не то)

> [!info] Автоматизация не заменяет ручное тестирование
> Автоматизация хороша для повторяемой **проверки** известных вещей. Люди хороши в **исследовании**, оценке качества и поиске неожиданного. Здоровый проект использует и то, и другое.

## Что автоматизировать, а что нет

Не каждый тест стоит автоматизировать. Выбирайте по соотношению **ценность / стоимость**.

**Хорошие кандидаты для автоматизации:**
- Повторяющиеся тесты в каждом релизе (регрессия)
- Smoke-тесты — быстрая проверка «жива ли сборка?»
- Тесты с множеством комбинаций данных (data-driven)
- Стабильные функции, которые редко меняются
- Тесты, которые трудно или долго делать вручную (расчёты, большие наборы данных)

**Плохие кандидаты для автоматизации:**
- Тесты, выполняемые один раз или очень редко
- Функции, которые часто меняются (высокая стоимость поддержки)
- Удобство использования, внешний вид, исследовательское тестирование
- Тесты, где ожидаемый результат неясен или субъективен

> [!tip] Быстрое правило
> Автоматизируйте то, что **стабильно, повторяемо и ценно**. Оставьте человеку то, что **новое, меняющееся или субъективное**.

## Пирамида автоматизации

Пирамида автоматизации показывает, сколько тестов каждого типа должно быть. Больше тестов внизу (дёшево, быстро), меньше наверху (дорого, медленно).

| Уровень | Скорость | Стоимость | Сколько |
|---|---|---|---|
| **UI / E2E** | Медленно | Высокая | Мало |
| **API / Интеграционные** | Средне | Средняя | Несколько |
| **Unit** | Быстро | Низкая | Много |

**Почему такая форма:**
- Unit-тесты быстры и точно указывают на сломанную функцию — здесь их больше всего
- API-тесты проверяют бизнес-логику без медленного браузера — надёжный средний слой
- UI-тесты медленные и хрупкие — оставляйте только важнейшие пользовательские сценарии

> [!warning] Антипаттерн «рожок мороженого»
> Если большинство тестов на уровне UI, а unit-тестов мало, набор становится медленным, нестабильным и дорогим в поддержке. Эта перевёрнутая пирамида называется «ice-cream cone», её следует избегать.

→ см. [[04 Пирамида тестирования]] для полного разбора уровней тестов.

## Как начать автоматизацию

Практический порядок настройки автоматизации на проекте:

1. **Определить цель.** Какую проблему решаем — медленную регрессию? Нестабильные релизы? Будьте конкретны.
2. **Выбрать охват.** Начните с малого: несколько стабильных и ценных тест-кейсов (часто smoke-тесты).
3. **Выбрать инструменты.** Под уровень и стек:

| Уровень | Распространённые инструменты |
|---|---|
| Unit | JUnit, TestNG (Java); pytest (Python) |
| API | Postman/Newman, REST Assured, Playwright API |
| UI (веб) | Selenium, Playwright, Cypress |
| UI (мобайл) | Appium, Espresso, XCUITest |

4. **Настроить фреймворк.** Структура проекта, test runner, отчётность, конфигурация.
5. **Написать первые тесты.** Простые, независимые, надёжные.
6. **Запустить в CI.** Автоматизация имеет ценность, только когда запускается автоматически при каждом изменении.
7. **Поддерживать.** Чинить нестабильные тесты, обновлять локаторы, удалять устаревшие тесты.

> [!info] Практика на Java
> Модуль 2 шаг за шагом строит реальный фреймворк автоматизации — см. [[AQA Java rus]] для JUnit 5 и Selenium WebDriver.

## Инструменты и фреймворки автоматизации

Automation tools выбираются по уровню тестирования и стеку проекта.

| Область | Типичные инструменты |
|---|---|
| Unit tests | JUnit, TestNG |
| UI web tests | Selenium, Playwright, Cypress |
| API tests | REST Assured, Postman/Newman, Karate |
| Mobile tests | Appium, Espresso, XCUITest |
| Reports | Allure, JUnit XML, CI dashboards |

**Automation framework** — это структура проекта вокруг этих tools: test runner, configuration, page objects или API clients, test data, logs и reports.

→ см. [[Архитектура фреймворка автотестов]] про структуру framework в Module 2.

## Отчётность

Прогон тестов полезен, только если люди быстро понимают результат. Хороший отчёт отвечает: сколько прошло/упало, какие тесты упали и почему.

**Что показывает хороший отчёт:**
- Всего тестов, прошло, упало, пропущено
- Название каждого упавшего теста и причину падения
- Скриншоты или логи для упавших UI-тестов
- Время выполнения и динамику во времени

**Распространённые инструменты отчётности:**
- **Allure** — наглядные HTML-отчёты; очень популярен в Java/AQA
- **Встроенные отчёты раннеров** — XML-вывод JUnit/TestNG
- **Дашборды CI** — Jenkins, GitHub Actions показывают pass/fail по каждому прогону

> [!tip] Упавший тест должен быть диагностируемым
> Если тест упал и по отчёту нельзя понять *почему* — отчёт неполон. Прикладывайте логи, скриншоты и понятные сообщения assert.

## CI/CD и автоматизация

**CI (Continuous Integration)** — разработчики часто вливают код, и автоматическая сборка + прогон тестов проверяет каждое изменение.
**CD (Continuous Delivery/Deployment)** — успешные сборки автоматически готовятся к релизу (или выкатываются).

**Почему автоматизация живёт в CI/CD:**
- Тесты запускаются при **каждом** коммите или pull request, ловя баги рано
- Упавший тест может **заблокировать** вливание сломанного изменения
- Никому не нужно помнить о запуске тестов — это делает пайплайн

**Типичный CI-пайплайн:**
1. Разработчик пушит код
2. Пайплайн собирает приложение
3. Запускаются unit-тесты (быстро — падаем рано)
4. Запускаются API- и UI-тесты
5. Публикуется отчёт
6. Если всё прошло, сборка деплоится

**Распространённые инструменты CI/CD:** GitHub Actions, GitLab CI, Jenkins, TeamCity.

> [!caution] Нестабильные тесты отравляют CI
> Тест, который случайно то проходит, то падает (без реального бага), разрушает доверие к пайплайну. Люди начинают игнорировать красные сборки. Чините или изолируйте flaky-тесты немедленно.

## Типичные ошибки

- **Автоматизировать всё.** Некоторые тесты не стоят поддержки — будьте избирательны.
- **Тесты зависят друг от друга.** Каждый тест должен сам готовить данные и работать независимо в любом порядке.
- **Жёсткие паузы (`sleep`).** Используйте умные ожидания условий, а не фиксированные задержки — они медленные и нестабильные.
- **Слабые проверки.** Тест, проверяющий лишь «страница загрузилась», пропускает реальные баги. Проверяйте фактический ожидаемый результат.
- **Нет плана поддержки.** Автоматизация — это код; ей нужен уход. Заброшенные наборы тестов гниют и игнорируются.
- **Игнорирование flaky-тестов.** Нестабильные тесты надо чинить, а не перезапускать до зелёного.

## Вопросы для собеседования

### Топ 10

**1. Что такое автоматизация тестирования?**
Автоматизация тестирования использует ПО для автоматического выполнения тестов, сравнения результата с ожидаемым и отчёта о pass/fail — без ручного повторения шагов человеком. Её цель — быстрые, повторяемые, надёжные проверки.

**2. Заменяет ли автоматизация ручное тестирование?**
Нет. Автоматизация хороша для повторяемой проверки известных вещей; люди хороши в исследовании, оценке удобства и поиске неожиданного. Здоровый проект использует оба подхода.

**3. Что стоит автоматизировать, а что нет?**
Автоматизировать стабильные, повторяемые, ценные тесты — регрессию, smoke, data-driven кейсы. Не автоматизировать одноразовые тесты, часто меняющиеся функции и usability/исследовательское тестирование, где результат субъективен.

**4. Что такое пирамида автоматизации?**
Модель, показывающая, сколько тестов каждого типа иметь: много быстрых дешёвых unit-тестов внизу, несколько API/интеграционных в середине и мало медленных дорогих UI/E2E-тестов наверху.

**5. Что такое анти-паттерн «рожок мороженого»?**
Перевёрнутая пирамида — большинство тестов на уровне UI и мало unit-тестов. Она делает набор медленным, нестабильным и дорогим в поддержке, и её следует избегать.

**6. Почему flaky-тесты опасны?**
Тест, который случайно то проходит, то падает без реального бага, разрушает доверие к набору. Люди начинают игнорировать красные сборки, и реальные падения пропускаются. Flaky-тесты нужно немедленно чинить или изолировать.

**7. Какова роль CI/CD в автоматизации?**
CI/CD запускает автотесты при каждом коммите или pull request, ловя баги рано и блокируя вливание сломанных изменений. Автоматизация имеет реальную ценность, только когда запускается автоматически, а не вручную.

**8. Какие есть хорошие практики написания автотестов?**
Держать тесты независимыми, использовать понятные имена, тестировать поведение, а не реализацию, использовать простые тестовые данные, предпочитать умные ожидания жёстким паузам и использовать сильные проверки фактического ожидаемого результата.

**9. Что делает отчёт о тестах хорошим?**
Он показывает итоги (прошло/упало/пропущено), название и причину каждого падения, логи или скриншоты для упавших UI-тестов. Если по отчёту нельзя понять, почему тест упал, отчёт неполон.

**10. Как начать автоматизацию на новом проекте?**
Определить цель, выбрать небольшой ценный охват (часто smoke-тесты), выбрать инструменты под уровень и стек, настроить фреймворк и отчётность, написать простые надёжные тесты, запустить их в CI и поддерживать.

### Каверзные вопросы

**1. Тест проходит локально, но падает в CI. В чём может быть причина?**
Различия в окружении — версия браузера, размер экрана, тайминг, тестовые данные или параллельный запуск. Также может быть скрытая зависимость от локального состояния. Решение — сделать тесты независимыми и не зависящими от окружения, и воспроизвести условия CI.

**2. Руководство хочет автоматизировать 100% тест-кейсов. Это хорошая идея?**
Нет. Некоторые тесты не стоят стоимости поддержки — одноразовые кейсы, быстро меняющиеся функции и субъективные usability-проверки. Автоматизация должна нацеливаться на стабильные, повторяемые, ценные тесты; погоня за 100% тратит силы и создаёт хрупкий набор.

**3. Ваш набор UI-тестов медленный и часто ломается. Как улучшить?**
Сместить проверки вниз по пирамиде — покрыть бизнес-логику быстрыми unit- и API-тестами, а на уровне UI оставить только важнейшие пользовательские сценарии. Заменить жёсткие паузы умными ожиданиями и стабилизировать локаторы. Это решает проблему «рожка мороженого».

**4. Тест использует `Thread.sleep(5000)` и работает. Зачем менять?**
Фиксированные паузы делают тесты медленными (всегда ждут полное время) и нестабильными (элемент может всё ещё быть не готов или готов гораздо раньше). Используйте явное ожидание реального условия — быстрее и надёжнее.

**5. Все тесты зелёные, но баги всё равно доходят до продакшна. Что может быть не так?**
У тестов могут быть слабые проверки (только «страница загрузилась»), они покрывают не те области или опираются на те же ошибочные допущения, что и код. Зелёный не значит корректный — пересмотрите, что тесты реально проверяют, и добавьте покрытие пропущенных сценариев.
